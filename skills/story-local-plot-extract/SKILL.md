# story-local-plot-extract：本地 TXT 小说情节提取

你是本地 TXT 小说情节抽取器。

**核心：不是抽零散桥段，不是抽单一元素，而是把一本书拆成可复用、可写作、可追踪关系变化的多层剧情资料库。**

## 适用范围

适用于：
- 长篇网文
- 中篇小说
- 章节清晰的短篇合集式文本
- 已在本地存在完整正文的 `txt` 小说文件

不适用于：
- 只有设定没有正文的材料
- 纯世界观文档
- 明显未分章且文本过短、无法形成连续剧情链的片段
- 非 `txt` 格式小说文件

## 核心约束

### 1. 用模型，不走专用抽取脚本

本 skill 的主体工作由 Codex 模型完成。

- **禁止**把任务改写成“运行已有抽取脚本得结果”
- **禁止**依赖事件卡脚本先抽卡再聚合，除非用户明确要求“脚本版”
- **允许**使用基础 shell 做这些辅助动作：
  - 读取原文
  - 按章节定位
  - 统计字数/章数
  - 创建输出目录与文件
  - 断点续写已有 markdown/jsonl

### 2. 先标题，后正文

抽取顺序必须是：

1. **先看章节标题**
   - 建立粗粒度剧情走向
   - 识别阶段切换、事件段落、关系转折
2. **再回到正文**
   - 校正标题误导
   - 补齐起因、推进、转折、结果
   - 确认哪些章节真属于同一条连续剧情

**标题只提供线索，正文才是裁决依据。**

### 3. 默认产出六层，不允许只停在中层

本 skill 的标准产物不是一层，而是六层：

1. `剧情骨架层`
2. `连续剧情层`
3. `细粒度施工层`
4. `跨章情节事件层`
5. `人物关系网层`
6. `角色状态线层`

如果只做出前两三层，没有继续拆到“章怎么写”“角色怎么变”“关系怎么变”，**任务不算完成**。

### 4. 要通用，不要只抽单一题材元素

一本书里可能同时存在：
- 权谋
- 感情
- 家族冲突
- 升级成长
- 追查旧案
- 身份暴露
- 战争/求生/交易/误会/反转

**不能因为小说主类型是古言/玄幻/都市，就只保留一种元素。**

抽取时要保留这本书里真正存在的多种剧情类型，最后形成**通用可迁移的剧情库**。

## 输入要求

用户必须提供：

- 本地 `txt` 小说文件路径

不是本地 `txt` 文件时，不进入抽取。

## 输出目录

固定输出到：

`本地TXT情节库/{书名}/`

开始抽取前，先复制：

`references/book-template/`

到目标目录：

`本地TXT情节库/{书名}/`

再逐步回填各文件内容。

最小命令示例：

```bash
mkdir -p "本地TXT情节库/{书名}"
cp -r "references/book-template/." "本地TXT情节库/{书名}/"
```

标准产物：

- `剧情骨架库.md`
- `连续剧情库.md`
- `连续剧情库.jsonl`
- `细粒度剧情施工库.md`
- `细粒度剧情施工库.jsonl`

## 抽取流程

### Phase 1：确认材料

先确认：

- 书名
- 输入路径/文本来源
- 是否已完结
- 是否已有旧版剧情库

如果已经有同名目录：
- 读取 `进度.md`
- 读取已有各层产物
- 从断点续做，不整本重来

进入抽取前先检查：

- 路径是否指向本地 `txt` 文件
- 文件是否可直接读取正文
- 是否已将 `references/book-template/` 复制到 `本地TXT情节库/{书名}/`

### Phase 2：建立标题级剧情骨架

先通读目录或章节标题，做一份**粗骨架**。

目标不是立即下最终结论，而是先回答：

- 这本书大致分成几个阶段
- 哪些章节标题明显指向同一段连续事件
- 哪些标题是转折点、兑现点、爆点、收束点

这一阶段输出到 `抽取说明.md`，并同步沉淀 `剧情骨架库.md`。

### Phase 3：正文校正与连续剧情识别

按章节顺序读正文，用模型直接判断连续性。

每遇到一段剧情时，先判断它是否满足以下条件：

1. 是否有明确触发事件
2. 后续章节是否围绕同一目标/冲突推进
3. 关键人物是否基本连续
4. 冲突是否在因果上连续，而非仅仅同场出场
5. 是否出现阶段性结果、反转或悬念落点

满足以上条件，才立为一条“连续剧情”。

#### 连续剧情的划分原则

可以拆成不同剧情条的情形：
- 目标变了
- 核心冲突变了
- 主要对手变了
- 场域彻底切换且前一段已完成收束
- 前一段只是背景铺垫，后一段已进入新问题

应该保留为同一剧情条的情形：
- 只是换了地点，但核心矛盾未变
- 只是推进手段变化，但目标一致
- 只是从试探进入正面对撞，本质仍是一条线
- 结果未兑现，只是中途加压或误导

### Phase 4：生成连续剧情条

每条连续剧情必须按统一结构输出。

最低要求字段：

- `plot_id`
- `plot_name`
- `plot_type`
- `start_chapter`
- `end_chapter`
- `chapter_span`
- `main_characters`
- `core_conflict`
- `trigger`
- `development`
- `turning_points`
- `result_or_hook`
- `transferable_value`
- `tags`
- `transferable_genres`

#### 这一层的定位

这一层是：
- 中剧情层
- 可复用情节种子层

这一层不是最终写作层。

### Phase 5：下钻为细粒度剧情施工条

在连续剧情条完成后，必须继续下钻，产出 `细粒度剧情施工库.md` 与 `细粒度剧情施工库.jsonl`。

每条连续剧情，至少继续拆成：

1. `切入场`
2. `第一次冲突`
3. `中段加压`
4. `转折/反转`
5. `结果/章尾钩子`

#### 细粒度施工条硬标准

每条施工条都必须满足：

- **能直接拿去写 1-3 章**
- **能看出场景顺序**
- **能看出冲突升级**
- **能看出章尾钩子**
- **不是泛泛而谈的剧情概述**

### Phase 6：抽跨章情节事件

这一层参考 `chaishu-vue3` 的思路，专门抽“跨章连续发生的事件链”。

注意：它不是大剧情骨架，也不是单章小动作，而是：

- 一个事件从哪章开始
- 中途经历了哪些关键推进
- 在哪些章节发生转折
- 以什么章节落点结束或转挂到下一条线

适合抽的事件类型：
- 追杀链
- 交易链
- 查案链
- 试探链
- 联姻链
- 夺权链
- 修炼突破链
- 误会发酵链
- 仇怨升级链

最低建议字段：

- `event_id`
- `event_name`
- `event_type`
- `start_chapter`
- `end_chapter`
- `involved_characters`
- `event_goal`
- `chapter_beats`
- `turning_chapters`
- `event_outcome`
- `next_link`

### Phase 7：抽人物关系网

这一层不是简单列“谁和谁认识”，而是要写明关系性质和变化。

每对关键角色至少要判断：

- 初始关系
- 当前关系
- 关系变化原因
- 关键变化章节
- 关系中的权力高低
- 关系中的利益/感情/敌意/利用成分

最低建议字段：

- `relation_id`
- `character_a`
- `character_b`
- `relation_type`
- `initial_relation`
- `current_relation`
- `change_path`
- `key_chapters`
- `dominance`
- `emotion_bias`
- `future_risk`

### Phase 8：抽角色状态线

这一层专门回答“这个人一路变成了什么样”。

每个核心角色都要至少拆出：

- 身份变化
- 阵营变化
- 能力变化
- 目标变化
- 心态变化
- 与主角关系变化
- 关键节点章节

最低建议字段：

- `character_id`
- `character_name`
- `role_position`
- `initial_state`
- `state_changes`
- `current_state`
- `motivation_shift`
- `power_shift`
- `relation_shift`
- `critical_chapters`

### Phase 9：质量门控

提交结果前，逐条自检。

#### 必过检查

1. **连续性**
   - 这一条是不是完整的因果链
2. **非局部化**
   - 它是不是只是一个局部动作/一句话/一个姿态
3. **非单元素化**
   - 有没有把书误抽成只剩一种元素
4. **可复用性**
   - 这条剧情能否迁移到别的题材中改写使用
5. **命名合格**
   - 标题是否像剧情模板名，而不是正文摘句
6. **去重**
   - 相邻两条是否实为同一条剧情被机械拆开
7. **覆盖**
   - 重要主线、强支线、关键反转线不能漏
8. **可写性**
   - 有没有下钻到“直接能写章节”的粒度
9. **关系可追踪**
   - 人物关系是否能看到变化，而非静态标签
10. **状态可追踪**
   - 角色状态是否有时间线，而非静态介绍

#### 发现问题时的修正原则

- 宁可少一条重复条，不要多一条错拆条
- 宁可人物字段留空，也不要填明显伪人名
- 宁可摘要压缩，也不要堆原文长句
- 宁可多拆几层，也不要只停在粗骨架

## 输出格式

- `剧情骨架库.md`：整书阶段结构
- `连续剧情库.md/jsonl`：中剧情条
- `细粒度剧情施工库.md/jsonl`：可直接开写
- `跨章情节事件库.md/jsonl`：跨章事件链
- `人物关系网.md/jsonl`：关系结构与变化
- `角色状态线.md/jsonl`：人物成长与状态演变
- `_meta.json`：程序侧统一元信息

## 元信息要求

提交前必须补出 `_meta.json`。

最低字段：

- `book_name`
- `skill_name`
- `skill_version`
- `source_file`
- `output_dir`
- 各层覆盖范围
- 各层条数
- 对应的 `jsonl` / `md` 文件名
- 下一步断点

## 长文本处理原则

如果整本书过长，不能一次稳定吃完：

1. 先按章节批次处理
2. 每批先抽骨架与连续剧情候选
3. 再补事件链、关系网、状态线
4. 再把关键剧情下钻成施工条
5. 批次结束后回看边界章节
6. 把跨批次实际上属于同一条线的剧情、事件、关系变化合并
7. 每批结束更新 `进度.md`

最终要回收成六层完整库，而不是只留下粗骨架。
