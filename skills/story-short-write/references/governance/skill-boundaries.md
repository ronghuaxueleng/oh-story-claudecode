# 短篇写作相关 skill 边界

这份文件只解决一个问题：

`story-short-write`、`story-short-analyze`、`story-deslop` 各自到底负责什么，不负责什么。

---

## 一、story-short-analyze

负责：

1. 拆原文
2. 做样本分级
3. 判断高敏层级
4. 拆高敏桥段
5. 产出可供后续写作使用的 `book.profile.json`、`profile_source.md`、`同桥段过检规则.md`

不负责：

1. 直接开正文
2. 对用户现成稿做逐段回修
3. 代替写作 skill 做送检后定点返工

---

## 二、story-short-write

负责：

1. 从 `profile` 和拆书资产出发起盘、搭骨架、写正文
2. 判断当前问题更偏 `讲法型 / 桥段链型 / 混合型`
3. 在高风险写作与回修场景里，决定：
   - 是否先回到细纲拆链
   - 是否先选母稿
   - 是否先走高敏同桥手册
4. 跑内部审计
5. 产任务单
6. 决定这一轮先修哪几个块、哪几个场

不负责：

1. 把所有 Gate A-F 去味细则全文重复一遍
2. 把去味 skill 变成自己的附庸入口
3. 直接把某一段当成纯润色任务处理

调用口径：

- 如果核心问题是 `写什么、怎么排、桥怎么换、尾怎么收`
  - 走 `story-short-write`
- 如果核心问题是 `这一稿已经写完，只想去味`
  - 走 `story-deslop`

---

## 三、story-deslop

负责：

1. 对已成稿文本做 AI 味检测
2. 处理词壳、句壳、书面腔、重复描写、过度工整感
3. 对高风险段执行第二闸门：
   - `受限重写防错协议`
   - `失败即重写判定`

不负责：

1. 代替写作 skill 重搭整篇骨架
2. 决定主桥是否应该换链
3. 代替拆书 skill 判断样本分级
4. 代替写作 skill 长流程管理整篇多轮回修

升级口径：

- 如果去味时发现问题本质是：
  - `桥段链过敏`
  - `尾声入口错误`
  - `证据触发顺序错误`
  - `同桥标准承载方式过强`
  这类结构病

  就不要继续在 `story-deslop` 里硬磨。

  应升级回：

  - `story-short-write`

---

## 四、最短判断句

以后遇到短篇任务，先问一句：

`这次要解决的是“怎么写”，还是“怎么去味”，还是“这篇原文到底能学什么”。`

答案分别对应：

1. `怎么写` → `story-short-write`
2. `怎么去味` → `story-deslop`
3. `原文能学什么` → `story-short-analyze`
