# 短篇拆书主执行提示词

> 2026-07-17 起，正式推荐的主入口不再是“一次性吞完整份执行提示词”，而是：
>
> 1. 先读 `SKILL.md`
> 2. 再读 `references/pipeline/staged-execution-index.md`
> 3. 按当前阶段只加载对应 `stage-*.md`
>
> 本文件保留为完整执行备忘录，不承担任何兜底生成职责，也不建议无差别整份加载进主上下文。

这份文件不是方法论说明，而是主会话执行模板。

用途：

- 让 `story-short-analyze` 在正式拆书时固定按一套话术和顺序执行
- 防止模型漏拆、跳批、先写结论后补文件
- 把“快速厚拆落盘 + 批末快检 + 最终验收”推进到主执行层

---

## 1. 使用方式

进入正式拆书后，主会话应直接套用下面骨架执行，不要自由发挥流程顺序。

硬口径：

- 不允许只写分析回复，不落盘文件
- 不允许用户没点名，就默认少产某些文件
- 不允许先说“核心文件完成”，再把剩余文件留到后续
- 不允许跳过批末快检
- 不允许为了批量生成压低任何既有厚度门槛
- 不允许长时间在内部规划整套文件后集中落盘；读完原文后先写 `事实与推断台账.md`
- 不允许不跑最终验收脚本就宣告完成
- 不允许任何脚本补写、扩写或修复 Markdown 正式产物
- 不允许使用 bak、上一本文档、旧 profile 或其他拆书目录作为当前书模板
- 默认进入 `厚拆模式`：目标不是做“刚好能过验收的压缩摘要”，而是优先保留最值钱的中段承重桥、人物口气差、动作权限差、物件回流、旧伤触发器、章法失效测试
- 厚拆不等于平均灌长所有文件；它的意思是高价值层默认宁可多保留，不要先压掉
- 只有原文确实没有对应资产时，才允许写薄；不能因为想省篇幅，主动把关键中段桥和人物不同脸压没
- 不允许用“先把所有文件都铺出来，后面再回头补厚”的方式规避主报告深拆；`主报告批` 必须先过厚拆闸门，后续批次才允许开始
- 如果 token、时间或篇幅出现压力，优先保住 `拆文报告.md / 情节节点.md / 写作手法.md / profile_source.md` 的厚拆层，不允许为了平均铺满 16 张表和细节库，把主报告写成压缩版
- 默认进入 `快速厚拆模式`：完整读取原文和样本一次，资产阶段使用缓存锚点、候选池与精确原文切片，不重复吞全文
- 批量输出失败时只二分责任批次；二分仍失败才回退双文件模式

## 2. 主会话固定执行骨架

把下面内容当作执行提示词使用：

> 你现在执行 `story-short-analyze` 正式拆书。目标不是只写分析结论，而是自动产出 skill 定义过的整套拆书资料包，并在结束前完成全量验收。
>
> 默认按 `厚拆模式` 执行：不要满足于“文件齐了、脚本过了”，而要优先保留后续仿写最值钱的层，尤其是中段承压桥、人物口气差、动作权限差、物件铰链、旧伤触发器、章法失效测试。
>
> 先读取并遵守：
>
> - `references/pipeline/analyze-execution-core.md`
> - `references/pipeline/output-contract.md`
> - `references/pipeline/output-templates.md`
> - `references/pipeline/quality-checklist.md`
> - `references/pipeline/auto-full-output-task.md`
> - `references/examples/INDEX.md`
> - `references/pipeline/session-manual-execution-protocol.md`
>
> 读取完以后，先判断当前原文最容易拆坏的地方，再只选 1-2 本 few-shot 样本对照，不要把 3 本样本全文一起吞进来：
>
> - 中段桥被压掉 / 题面预期翻转缺失 / 细节库缩水：优先看《幼薇》
> - 公权场撞见后中段只剩吵架 / 私域越界不实 / 证据链清算被压平：优先看《扫黄扫到了我老公》
> - 偏心剥夺链被写成背景 / 换亲和掉马只剩标签 / 回门清算只剩控诉：优先看《归月学生》
>
> 每本所选样本必须实际读取 `README.md + 相关原文段 + 正反例对照.md`，并先落 `_sample_comparison.md`。只声明样本名不算使用样本。
>
> 再固定按快速厚拆顺序执行：
>
> 1. `_sample_comparison.md`
> 2. `事实与推断台账.md`
> 3. `_analysis_brief.md`
> 4. 第一波并发：`拆文报告.md` / `情节节点.md + 写作手法.md` / `本书动态信号字典.json + 原文资产候选池.md`
> 5. 回看样本反例区并更新 `_sample_comparison.md`，补 `_meta.json`
> 6. 运行 foundation 预检
> 7. 第二波并发：16 表两条 lane / 原文细节库 / 常规资产 / 高敏资产
> 8. 统一核销与 BID 贯通
> 9. `profile_source.md`
> 10. 生成 `book.profile.json`
> 11. 全量验收
>
> 每个批次都必须：
>
> - 批次内连续落盘，不重复加载原文全文、样本全文或整套规则
> - 第二波并发直接继承第一波会话，公共重读清单为空；只追加当前 lane 的 `delta_reads`，不重新打开主报告、节点、手法或共享台账
> - 批末只检查文件齐全、截断、厚度下限、标题、证据、空字段和专项要求，不提前跑全量 validator
> - 失败时只二分当前批次；二分仍失败才降级为双文件
> - 通过后立即进入下一批，不回头统一“补厚”
>
> 全量 `finalize` 只在所有文件齐备后运行一次。若失败，按错误清单整体评估 skill，不在同一次冷启动里反复补丁式修复。

## 3. 第 0 步：原文读取与事实边界闸门

正式进入第 1 批前，先确认：

1. 已拿到原文全文，不是摘要
2. 已读取 `_source_manifest.json` 和 `_source_reading_plan.md`
3. 已按读取计划逐块读到 manifest 记录的最后一行，不是只看一次受限输出
4. 已核对最后一节、最后事件、最后关系状态和尾部锚点
5. 已确定输出目录 `拆文库/{书名}/`
6. 已完成拆前 4 判断，并准备写入 `拆文报告.md`
7. 已明确本次任务不是“精简版拆书”
8. 已根据 `references/examples/INDEX.md` 选好 1-2 本样本，并明确本次是拿它们防什么失败，不是拿来照抄壳
9. 已实际读取每本样本的 README、相关原文段和正反例对照，并落 `_sample_comparison.md`

如果任意一项不满足，不能进入正式拆书。

进入第 1 批时固定执行：

- 先落 `事实与推断台账.md`，再写主报告；台账至少覆盖 `主体边界 / 时间边界 / 证据来源`
- 台账格式固定为 `F序号 | L起-L止 | 锚点：... | 类别：... | 主体：... | 动作：... | 结果：... | 叙述时点：... | 故事时点：... | 时间依据：... | 口径：... | 禁止越界：...`
- 每写完一条带 `那次 / 当时 / 此前 / 后来 / 多年前 / 第二天` 的事实，立即回到原文定位所指事件；正文出现位置和事件发生位置必须分开
- 任何高主动性判断必须句末回指 `【原文明确 Fxx】` 或 `【人工推断 Fxx】`
- 引用存在不等于事实支持结论；引用前必须核对 Fxx 的主体、动作、结果与当前断言一致。事实只证明“等待/利用既有条件”时，禁止写成“策划/发布/推动条件形成”
- `拆文报告.md` 固定补 `主角能动性三层判断`：分别写 `原文明确动作 / 叙事意图判断 / 未知边界`。事实谨慎只限制未知实现，不得把原文支持的借势、选择和策略性一并抹掉
- `拆文报告.md` 先写 `### 原文覆盖确认`
- `情节节点.md` 每个节点写成 `N序号 | L起-L止 | 锚点：原文短语 | 类型：... | 情绪：... | 涉及：... | 状态变化：... | 因果：... | 故事时序：...`
- 承重桥节点把 BID 直接写在同一条节点行，推荐 `类型：BID-01 中段承重桥`；只在说明区或下游资产写 BID 不算节点已标注
- 锚点必须能在对应行范围内找到，不能写概括词代替原文锚点
- 节点必须覆盖全部自动 Chunk；存在章节标记时，每章至少 1 个有效节点
- 最后有效节点必须位于原文最后 10%

## 4. 六批执行模板

## 第 1 组：事实底座与第一波并发

### 固定串行顺序

1. 1A：`事实与推断台账.md`
2. 1B：`拆文报告.md`
3. 1C：`情节节点.md`
4. 1D：`写作手法.md + _meta.json`

禁止把 1A-1D 合并成一次生成动作。每份文件使用自己的原文证据预算，不为后续文件预先压缩。

每个主报告文件的固定闭环：

1. 模型写当前文件。
2. 模型重读本文件引用的原文范围，不依赖脚本摘要。
3. 模型检查事实主体、双时间轴、人话表达、性别/羞辱或标题边界。
4. 脚本只做机械检查；模型读取检查结果后裁决并回写。
5. 当前文件语义正确后才能进入下一项。脚本返回 0 不能替代第 2-4 步。

### 必做动作

1. 先按 `output-templates.md` 生成三份主报告
2. `写作手法.md` 必须严格按固定骨架落盘
3. `_meta.json` 必须补齐结构计数和文件计数基础字段

### 本批自检

- `拆文报告.md` 是否包含：
  - `原文覆盖确认`
  - `样本分级与可学层判断`
  - `高敏桥段识别`
  - `题面拆解`
  - `故事梗概`
  - `结构划分`
- `拆文报告.md` 是否额外补齐：
  - `#### 1. 脚本硬筛`
  - `#### 2. 规则拆层判断`
  - `#### 4. 可学层 / 禁学层`
  - `#### 5. 后续调用方式`
  - `### 叙事时间线`
  - `#### 信息释放顺序`
  - `#### 故事实际时间线`
  - `#### 回叙 / 插叙对照`
  - `### 故事核`
- `拆文报告.md` 的 `题面拆解 / 结构划分 / 人物分析 / 情感曲线 / 反转分析` 是否至少各有 1 组 `原文现象证据 -> 为什么成立 -> 新稿最容易写假的点`
- `拆文报告.md` 是否已经把“为什么原文能过”的第一层解释写在主报告里，而不是把解释全部后置到 `profile_source.md / 桥段施工卡.md / 细节库`
- `拆文报告.md` 的 `结构划分` 是否显式写了 `字数范围 / 占比 / 功能 / 对应节`，而不是只写 4-6 条概括式分段
- `情节节点.md` 是否覆盖全文主要节点
- `情节节点.md` 是否每个节点都有 `L起-L止 + 锚点`
- `情节节点.md` 是否每个节点都有 `类型 / 情绪 / 涉及 / 状态变化 / 因果 / 故事时序`
- 节点锚点是否覆盖 `_source_manifest.json.chunks` 的全部分块
- 有章节标记时，是否每章至少存在 1 个有效节点
- 最后有效节点是否进入原文最后 10%，而不是靠“终局前夜”字样冒充尾部覆盖
- `情节节点.md` 是否拆到可施工颗粒度：
  - 节点数可用分档阈值与 `ceil(原文字数 / 400)` 的较大值作漏拆复核参考，不作硬最低值
  - 8000 字以上基础阈值 20，5000-7999 字基础阈值 16，5000 字以下基础阈值 12
  - 10840 字文本可用 28 节点作复核参考；原文事件密度较低时允许更少，但必须由模型说明覆盖依据
- 原文存在中段承重桥时，`情节节点.md` 是否显式保留并分配 BID；原文没有时是否明确声明，而不是强造
- `写作手法.md` 是否按固定 9 节骨架完整落盘
- `写作手法.md` 是否不是每节只写 1 句总括，而是至少拆到：`人物嘴型/口气差`、`章法失效测试`、`物件铰链或意象物件`、`迁移风险`
- `写作手法.md` 是否写清“原文怎么做才成立”，而不是只写“用了倒叙/留白/铺垫”
- `写作手法.md` 的 `对话手法` 是否显式落到至少 3 类角色嘴型或口气差，而不是只写“对话有反差/推进快”
- `写作手法.md` 是否各有至少 1 条 `活词 / 句法模板 / 段落节拍 / 反面仿写句`
- `拆文报告.md` 是否不只停在基础结构层；如果目标是后续写作，是否已补 `情感曲线 / 爆点分析 / 反转分析 / 人物分析 / 开头分析 / 结尾分析 / 五维评分`
- 如果资产层明显比主报告更像“解释层”，是否回到本批重写主报告，而不是继续往后补资产
- `_meta.json` 是否存在 `structure_counts`

只要当前文件有一项缺失，停在当前项修正；通过后立即前进，不等四份文件一起返工。

## 第 2 组：16 张可直接仿写表快速批

### 前置硬闸：原文资产候选池

先读取 `references/pipeline/dynamic-signal-dictionary.md`，按全部 Chunk 建立：

- `写作资产/本书动态信号字典.json`

完成“首次全文发现”后，再读取 `references/pipeline/source-asset-coverage-ledger.md`，按全部 Chunk 第二遍读取原文并落盘：

- `写作资产/原文资产候选池.md`

要求：

- 每条候选含 `C编号 / L起-L止 / 锚点 / 类别 / 资产名 / 去向 / 状态 / 理由`
- 候选覆盖全部 Chunk 和 12 类核心资产
- 表格完成后再回扫全部 Chunk，把首次遗漏项追加到候选池和目标表
- 表格完成后先更新动态字典的“表后回扫”记录，再用新增词重扫全部 Chunk
- 表后回扫完成后必须再执行两轮独立漏项审计；两轮都覆盖全部 Chunk、零新增且 `state_sha1` 与当前字典/候选池一致时，validator 才允许 `stabilized: true`
- 候选池必须写至少 5 项 `反向漏项审计`，并对每项给出 `收录 / 不收录 + 理由`；这是硬闸，但仍不得凑伪候选
- 5000-7999 字候选池至少 28 条，8000 字及以上至少 40 条；不足时继续按 Chunk 和 16 类回扫，不得直接进入第二波
- 通用提示词命中不等于必须收录；若 `视频 / 低下头 / 攥紧拳头 / 联系不上` 在该处不承担独立写作功能，必须在反向漏项审计逐项说明后排除，不得为过校验制造低价值表格行
- 完成 `物件替换对 / 微动作角色覆盖 / 对白侵占与假道歉 / 安静等待与未归 / 未来公开事件钩子` 五项专项回扫
- 不允许达到每表最低行数后停止

### 第一波首写格式硬闸

- 派发第一波 lane 时，必须把对应 `_parallel_plan.json.foundation_lanes[].first_write_contract` 完整放入 worker prompt
- 主报告固定标题、节点单行字段与 BID、写作手法固定章节、动态字典 JSON、候选池字段必须在第一次落盘时正确
- worker 落盘前先检查固定标题逐字命中且各一次、Markdown 标题唯一、无占位标题、无空标签字段
- 契约与个人惯用模板冲突时，以 `first_write_contract` 为准；禁止依赖 foundation validator 做常规格式返修

### 第二波首写格式硬闸

- 派发第二波 lane 时，必须把对应 `_parallel_plan.json.asset_lanes[].first_write_contract` 完整放入 worker prompt
- worker 在第一次写文件前，先按契约固定表头、最低行数、卡片字段和唯一标题；禁止先写旧模板再靠 validator 返修
- 契约与个人惯用模板冲突时，以 `first_write_contract` 为准
- 表格三段施工标题、细节卡五字段、高敏桥段字段都必须在首写正文中逐项完成，禁止批末重复追加同名标题

### 固定分组

1. 2A 结构/物件/动作 8 表：导语拆解、顺序事件、物件、动作、误判、钩子、微动作、安静压迫场
2. 2B 对白/人物/关系/秩序 8 表：对白功能、对话衔接、人物偏手、失控说话、烂关系漏出、外部秩序、公开炸场、后果链

每个大批完成后统一检查对应 8 张表是否消费候选、是否有核心/次级层级、是否写出专属施工段。若输出截断或厚度不足，先拆成两个 4 表子批；仍失败才降级为双文件。

### 每张表必须补的 3 段

1. `可直接借的承重结构`
2. `迁移顺序提醒`

### 写法硬约束

- 每张桥段施工卡是否有 `一句人话抓手`，且不是“权限/秩序/现实后果/承重结构”一类抽象术语拼接
- `story_guardrails.character_face_split` 是否完整含有非空的 `different_face_evidence / reaction_order_split / action_authority_split`
- `story_guardrails.consequence_structure` 是否完整含有非空的 `pre_evidence_reality_consequences / consequence_rebound_modes / tail_entry_owner / tail_entry_exclusion_reason`

## 第 6 批：全量验收批

### 洁净与贯通硬闸

- 全量正式 Markdown 不得有重复标题、空字段、`桥1 / 桥段卡1 / 待补 / 占位` 标题或半成品模板壳
- 初始化脚本不再预填正式资产；不要自行先造空骨架再向后追加
- 候选池和事实台账共同确认的核心承重桥，必须贯通主报告、情节节点、对应仿写表、高敏桥资产和 profile_source
- finalize 通过后，`_progress.md` 必须全量勾选，`_meta.json.last_stage_in_progress` 必须为空

### 固定命令

```bash
python3 "$CODEX_HOME/skills/story-short-analyze/scripts/run_short_analyze_finalize.py" "拆文库/{书名}"
```

### 本批自检

- 收口脚本是否通过
- `book.profile.json` 是否已自动生成
- 是否仍有缺件、缺标题、缺字段
- `_meta.json` 与真实文件数是否一致

不过线就不能说“拆书完成”。

- `当前状态：blocked-on-assets`
- `缺失项：...`
- `下一步：回到第 X 批补齐`

## 6. 与验收脚本的关系

这份模板负责约束主会话执行顺序；
`validate_short_analyze_outputs.py` 负责收口脚本内部的最终验收；
两者缺一不可。

主会话不能用“我觉得差不多齐了”代替脚本验收。
