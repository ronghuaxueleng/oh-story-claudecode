# 短篇拆书主执行提示词

这份文件不是方法论说明，而是主会话执行模板。

用途：

- 让 `story-short-analyze` 在正式拆书时固定按一套话术和顺序执行
- 防止模型漏拆、跳批、先写结论后补文件
- 把“全量自动落盘 + 分批自检 + 最终验收”推进到主执行层

---

## 1. 使用方式

进入正式拆书后，主会话应直接套用下面骨架执行，不要自由发挥流程顺序。

硬口径：

- 不允许只写分析回复，不落盘文件
- 不允许用户没点名，就默认少产某些文件
- 不允许先说“核心文件完成”，再把剩余文件留到后续
- 不允许跳过批次自检
- 不允许不跑最终验收脚本就宣告完成

---

## 2. 主会话固定执行骨架

把下面内容当作执行提示词使用：

> 你现在执行 `story-short-analyze` 正式拆书。目标不是只写分析结论，而是自动产出 skill 定义过的整套拆书资料包，并在结束前完成全量验收。
>
> 先读取并遵守：
>
> - `references/pipeline/analyze-execution-core.md`
> - `references/pipeline/output-contract.md`
> - `references/pipeline/output-templates.md`
> - `references/pipeline/quality-checklist.md`
> - `references/pipeline/auto-full-output-task.md`
>
> 再固定按 6 批顺序执行：
>
> 1. 主报告批
> 2. 16 张可直接仿写表批
> 3. 原文细节库批
> 4. 写作资产批
> 5. `profile_source + book.profile` 批
> 6. 全量验收批
>
> 每一批都必须：
>
> - 先写完本批全部文件
> - 再做本批自检
> - 自检通过后才能进下一批
>
> 如果任意一批缺文件、缺骨架标题、缺关键段落，立即回到该批补齐，不准继续往下走。

---

## 3. 第 0 步：正式拆书前检查

正式进入第 1 批前，先确认：

1. 已拿到原文全文，不是摘要
2. 已确定输出目录 `拆文库/{书名}/`
3. 已完成拆前 4 判断，并准备写入 `拆文报告.md`
4. 已明确本次任务不是“精简版拆书”

如果任意一项不满足，不能进入正式拆书。

---

## 4. 六批执行模板

## 第 1 批：主报告批

### 必产文件

- `拆文报告.md`
- `情节节点.md`
- `写作手法.md`
- `_meta.json`

### 必做动作

1. 先按 `output-templates.md` 生成三份主报告
2. `写作手法.md` 必须严格按固定骨架落盘
3. `_meta.json` 必须补齐结构计数和文件计数基础字段

### 本批自检

- `拆文报告.md` 是否包含：
  - `样本分级与可学层判断`
  - `高敏桥段识别`
  - `题面拆解`
  - `故事梗概`
  - `结构划分`
- `情节节点.md` 是否覆盖全文主要节点
- `写作手法.md` 是否按固定 9 节骨架完整落盘
- `_meta.json` 是否存在 `structure_counts`

只要有一项缺失，本批不通过。

## 第 2 批：16 张可直接仿写表批

### 必产文件

- `可直接仿写_导语拆解表.md`
- `可直接仿写_顺序事件表.md`
- `可直接仿写_物件表.md`
- `可直接仿写_动作表.md`
- `可直接仿写_对白功能表.md`
- `可直接仿写_对话衔接表.md`
- `可直接仿写_误判表.md`
- `可直接仿写_钩子表.md`
- `可直接仿写_微动作表.md`
- `可直接仿写_安静压迫场表.md`
- `可直接仿写_人物偏手表.md`
- `可直接仿写_失控说话表.md`
- `可直接仿写_烂关系漏出表.md`
- `可直接仿写_外部秩序表.md`
- `可直接仿写_公开炸场表.md`
- `可直接仿写_后果链表.md`

### 每张表必须补的 3 段

1. `可直接借的承重结构`
2. `迁移顺序提醒`
3. `为什么这个顺序不能乱`

### 写法硬约束

- 先把表格行写够，再写 3 段施工层，不能先写空说明后补表
- 3 段施工层都必须点名表格里的具体条目，至少 2 个
- `迁移顺序提醒` 必须写成“先 A，再 B，最后 C”；A/B/C 都要能在表格里找到
- `为什么这个顺序不能乱` 必须写“如果把 B 抢到 A 前面，会假在哪里”
- 16 张表不允许共用同一句提醒句；每张表都要像只对这张表有效
- 写完后再问一句：`把表名遮住，这 3 段还能不能只属于这一张表？`
- 如果答案是“放到别的表也差不多能用”，立即重写，不准带着万能句进入下一张表

### 生成时自问

- 这 3 段里，我有没有直接点到表里写出来的资产名？
- 如果把表名遮住，这 3 段还能看出它只属于这一张表吗？
- 如果不能，说明还在写通用空话，必须重写
- 这张表更像“事件顺序类 / 物件动作类 / 对白口气类 / 关系秩序类”中的哪一种？
- 我写出来的 3 段，是否符合这一类表该有的施工重点？
- 我有没有明确写出“哪一步先来，哪一步后翻，抢跑后具体假在哪里”？

### 本批自检

- 16 张表是否齐全
- 每张表是否非空
- 每张表是否都含固定 3 段
- 每张表的 `迁移顺序提醒` 是否至少点名 2 个本表条目
- 每张表的 `为什么这个顺序不能乱` 是否明确写了“哪一步抢跑会假”
- 是否出现跨文件复用的统一提醒句；只要 2 张以上雷同，整批退回

缺任意一张或缺任意一段，都回到本批补齐。

## 第 3 批：原文细节库批

### 必产目录与文件

- `原文细节库/场景细节库.md`
- `原文细节库/关系细节库.md`
- `原文细节库/情绪细节库.md`
- `原文细节库/对白细节库.md`
- `原文细节库/翻车细节库.md`
- `原文细节库/旧伤细节库.md`
- `原文细节库/动作细节库.md`
- `原文细节库/场面细节库.md`

### 本批自检

每一份细节库至少回答：

1. 具体发生了什么
2. 这个细节为什么有用
3. 它压的是谁、压在哪
4. 后续能迁到什么新桥段
5. 它对应的角色 / 情绪 / 反转是什么

### 写法硬约束

- 每个 `##` 小节都要先确定“这一小节对应的是原文哪一件具体事”，再写五条
- 第一条必须写具体事件，不许写“原文里出现了某类场面”
- 第四条必须写 1-2 个具体可迁桥段，不许写“同题材桥段”
- 不同 `##` 小节的五条答案不能整段复用，只换标题
- 如果五条里有三条几乎放到别的小节也通用，说明还没拆到细
- 第 2 条必须说明“为什么这件证据比一句总结更有效”，不能只写“让情绪落地”
- 第 3 条必须写清压的是 `脸面 / 名分 / 私域 / 旧伤 / 身体 / 秩序` 中哪一种，不能只写“压主角”
- 第 4 条除了桥段名，还要补一句“迁移时真正不能丢的承重件是什么”
- 第 5 条必须落回这一小节自己的角色站位，不能拿整本三角关系总评来糊

### 生成时自问

- 读这一小节的人，能不能直接看出是谁在什么地方做了什么？
- 这五条答案如果把标题删掉，还能不能分辨它和别的小节不同？
- 如果不能，说明还是模板壳，必须重写
- 这一小节真正的证据件是场景、物件、动作、对白，还是关系掉位？我有没有写出来？
- 这条细节真正压的是体面、名分、私域、旧伤、身体还是秩序？我有没有写出来？
- 这一条如果只剩“抽象评价”，删掉具体事件后还成立吗？如果成立，说明还没落到证据层

另外强制检查：

- 不允许出现“原文里出现了这一类场面”“同题材桥段”这类泛化壳句
- 同一份细节库里，不允许多个 `##` 小节五条答案只换标题不换内容
- 每个 `##` 小节至少要有 1 个具体事件锚点，能看出是谁在什么地方做了什么
- 每个 `##` 小节都应该像一张“证据卡”，不是一个题材词解释

### 必产文件

- `写作资产/母结构_故事走法.md`
- `写作资产/主冲突_副升级器.md`
- `写作资产/异物清单.md`
- `写作资产/第二层冲突清单.md`
- `写作资产/角色口气模板.md`
- `写作资产/关系重组方式.md`
- `写作资产/公开场_关键硬牌_后果.md`
- `写作资产/平台适配提醒.md`
- `写作资产/情绪母线.md`
- `写作资产/新状态清单.md`
- `写作资产/虐点对照细节.md`
- `写作资产/样本分级与可学层.md`
- `写作资产/高敏桥段识别.md`

### 高敏桥强制补项

- `重大证据前隔开的现实后果`

- `人物不同脸证据 / 谁先解释谁先压场 / 动作权限差`

### 本批自检

- 是否全部文件齐全
- `作者DNA指纹.md` 是否不是空标签
- `仿写约束_禁写清单.md` 是否每条都回答“为什么假”
- `同桥段过检规则.md` 是否写清起手件、承重件、假点、顺序原因

## 第 5 批：`profile_source + book.profile` 批

### 必产文件

- `写作资产/profile_source.md`
- `book.profile.json`

### 固定顺序

1. 先由模型补完 `profile_source.md`
2. 再生成 `book.profile.json`
3. 再检查 JSON 关键字段是否齐

### 本批自检

- `profile_source.md` 是否覆盖流派、主梗、副梗、组合公式、梗位分工
- `book.profile.json` 是否与 `profile_source.md` 对齐

## 第 6 批：全量验收批

### 固定命令

```bash
python3 "$CODEX_HOME/skills/story-short-analyze/scripts/run_short_analyze_finalize.py" "拆文库/{书名}"
```

### 本批自检

- 收口脚本是否通过
- `book.profile.json` 是否已自动生成
- 是否仍有缺件、缺标题、缺字段
- `_meta.json` 与真实文件数是否一致

不过线就不能说“拆书完成”。

---

## 5. 固定收口话术

只有在第 6 批通过后，才允许输出类似结论：

- `拆文资料包已全量备齐`
- `ready-for-write`
- `可进入 story-short-write`

如果脚本没过，必须明确说：

- `当前状态：blocked-on-assets`
- `缺失项：...`
- `下一步：回到第 X 批补齐`

---

## 6. 与验收脚本的关系

这份模板负责约束主会话执行顺序；
`validate_short_analyze_outputs.py` 负责收口脚本内部的最终验收；
两者缺一不可。

主会话不能用“我觉得差不多齐了”代替脚本验收。
