# Session 手工执行协议

这份协议吸收高质量 Codex session 的有效做法，同时明确拒绝其中会让结果失真的做法。

## 必须保留的做法

- 模型直接阅读原文并逐文件做语义判断，不用脚本批量铺正文。
- 原文与 1-2 本样本只完整读取一次，并把可复用锚点落入 `_sample_comparison.md`、事实台账、节点、动态字典和候选池。
- 事实台账与 `_analysis_brief.md` 串行保边界；主报告、节点/手法、字典/候选池按第一波并发，16 张表、细节库、写作资产按第二波并发。
- 每个大批完成后做文件齐全、截断、表行/细节卡/有效字符快检，不重复读取整本原文或样本。
- validator 报错后，只回到对应微批检查真实语义，不为命中机械规则硬塞原句。
- 16 张表、8 类细节库和写作资产分别按文件职责写，不共用统一壳。
- 收口只负责生成结构化 profile 和报告错误，不负责替模型补正文。

## 明确禁止的做法

- 原文未读到 EOF 就先写主报告。
- 读取上一本文档、旧 profile、bak 目录或测试目录作为当前书模板。
- 没读内置样本的原文与正反例对照，只在回复里声称“已选样本”。
- 为通过 validator 反复运行 finalize，再按报错机械替换表格原句。
- 让脚本补标题、补“为什么假”、扩写情绪母线或生成任何 Markdown 内容。
- 信息不足时使用通用婚姻、家庭、职场或古言事件兜底。
- 每产 1-2 个文件就重新吞一遍原文、样本和整套阶段文档。
- 为规避输出截断而预先把所有任务固定拆成双文件微批。

## 固定闭环

1. 读完当前原文全部 Chunk。
2. 读取 `references/examples/INDEX.md`，选 1-2 本样本。
3. 每本样本实际读取 `README.md`、相关原文段和 `正反例对照.md`。
4. 写 `_sample_comparison.md`，记录正例锚点、反例锚点和本书对应风险。
5. 事实台账后写 `_analysis_brief.md`，冻结角色称谓、时间边界和 BID。
6. 第一波并发完成主报告、节点/手法、字典/候选池。
7. 回看样本反例区，更新 `_sample_comparison.md`，随后运行 foundation 预检。
8. 预检通过后启动第二波 5 条资产 lane；失败只返修责任 lane。
9. 全部正式文件完成后运行一次 finalize；失败则返回责任批次定点重写。

## 放行原则

- 没有 `_sample_comparison.md`，不进入主报告。
- 样本记录没有具体文件和正反例锚点，视为没有使用样本。
- finalize 前后任一 Markdown 的 SHA1 发生变化，视为收口脚本越权。
- 无法从当前原文或内置样本得到依据时，直接阻断并说明缺什么，不允许生成兜底内容。
