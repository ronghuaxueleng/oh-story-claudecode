# story-review：多视角对抗式审查

你是审查协调器。并行 spawn 4 个子代理，各自从不同角度找问题，然后综合裁决。

**执行铁律：审查是找问题，不是验证正确性。**

---

## 规则分层

本 skill 内的规则分三层，优先级和用途明确区分：

### 1. 门禁层

- 定义：命中即否决并入、必须落工单、必须给失败码
- 适用：默认 hard checks、章尾硬规则、失败码、并入裁决、工单落盘、章节组重组建议
- 执行要求：不得省略，不得模糊，不得降级成“建议”

### 2. 质量层

- 定义：用于判断正文质量是否达到发书和留存要求，部分依赖人工综合判定
- 适用：人物显影、题面兑现、留存强度、高潮兑现、信息场比例、主角声线、高价值角色魅力
- 执行要求：必须写清证据、层级和修法，不得泛泛而谈

### 3. 说明书层

- 定义：审查视角、方法论、子代理分工、参考资料路径、提示模板
- 适用：不同视角的审查问题清单、方法、参考路径
- 执行要求：用于帮助找问题，不单独作为裁决来源

判定原则：

- 门禁层 > 质量层 > 说明书层
- 只有高频翻车、可客观识别、能绑定裁决后果的问题，才应升级为门禁层
- 审查不能把审美建议冒充门禁，也不能把门禁说成“仅供参考”

## 默认 Hard Checks

无论用户有没有追加补充词，`story-review full` 默认都必须检查以下硬项，不得省略：

1. 正文纯净度：工具残片、补丁标记、广告感、非故事语料
2. 一致性：人名、时间线、情报日次、角色状态、任务承接
3. 题面兑现：书名/文案/金手指/复仇/身份/流放卖点是否持续主驱动
4. 场面兑现：当章是否真有硬场面、外显战果、现实盘面变化
5. 章尾硬规则：必须单独抽看最后 `5-15` 行判型
6. 人物显影：外形锚点、动作锚点、说话方式、压场方式、YY价值
7. AI味：解释腔、作者代判句、抽象狠词顶动作、配角当翻译器
8. 追踪反校验：角色卡、情报台账、时间线、角色状态、伏笔是否同步
9. 连续章节留存：是否“查得很深、赢得很少”“副驱动反客为主”
10. 并入裁决：是否可并入主正文、是否不可继续顺写
11. 篇幅预算：是否超出本章目标上限、是否存在首章过量/单章失控

用户附加检查项的作用是“加码”，不是替代默认 hard checks。

这些默认 hard checks 中，以下项目属于门禁层，命中后默认可否决并入：

- 正文纯净度
- 一致性重大冲突
- 章尾硬规则
- 追踪假完成/双版本并存
- 题面滑轨
- 情报兑现超时

以下项目属于质量层，必须审，但通常需要结合证据综合裁决：

- 人物显影
- 留存强度
- 高潮兑现力度
- 高价值角色魅力
- 主角独特声线

## 统一失败码

`story-review` 和 `story-long-write` 共用同一套失败码：

- `F0`：过线
- `F1`：句子层失败
- `F2`：章节层失败
- `F3`：机制层失败
- `F4`：污染失败
- `F5`：流程/追踪失败

主线程和子代理报告里，最终都必须能映射到上述失败码；禁止只给形容词不给级别。

## 回修模式裁决

- `story-review` 不只负责报病，还必须裁决本轮应走哪种回修模式：
  - `补丁式抠句子`
  - `一次性收口重写`
- 仅当问题满足“单点、单层、单因”时，才允许裁成“补丁式抠句子”：
  - 只落在单个句子或单个短段
  - 失败码主要是 `F1`
  - 不牵动收益落点、段落骨架、人物站位、追踪口径
- 只要命中以下任一项，主线程必须明确裁成“一次性收口重写”并落盘：
  - 同一章连续 2 轮复审后仍保留相同主失败码
  - 同一问题同时跨“句子层 + 章节层”两层
  - 问题同时涉及“确认链/段落骨架/收益落点/人物压场”中的 2 项及以上
  - 已经出现“局部越改越碎、复审口径反复收窄但不过线”
- 一旦裁成“一次性收口重写”：
  - 工单、固定验收清单、复盘单必须写明“回修模式：一次性收口重写”
  - 后续复审若仍不过线，优先建议继续整段重写，不得再把同一问题降回补丁式小修

## Review Mode 选择

- `/story-review` 或 `/story-review full`，以及不带斜杠的同名触发 → spawn 全部 4 个子代理；如果当前已经在子代理内，自动降级为 `solo`
- `/story-review lean`，以及不带斜杠的同名触发 → 只 spawn `story-architect` + `consistency-checker`；如果当前已经在子代理内，自动降级为 `solo`
- `/story-review solo`，以及不带斜杠的同名触发 → 不 spawn 子代理，自身做基础检查
- 未指定 → 默认 full，并告知用户

## Phase 1：收集待审查内容

1. 确定审查范围：
   - 用户指定了章节/文件 → 只审查指定内容
   - 用户未指定 → 审查最近修改的内容（git diff）或当前章节
2. 读取待审查的正文内容
2.1. 必须直接读取当前待审查正文文件，不得只依赖旧摘录、旧审查缓存或上轮复制片段：
   - 特别是章尾判型、字数、最后 `5-15` 行、最新回修段落，必须以当前文件实读为准
   - 若子代理或主线程引用的正文末尾与当前文件不一致，主线程必须以当前文件实读结果覆盖旧结论，并在综合裁决中显式纠偏
3. 读取相关的设定文件和大纲
3.5. 优先读取 `story-review/references/review-gate-protocol.md` 作为本轮门禁主协议：
   - 解释句密度 / 黑名单
   - 人物显影最低配额
   - 高潮双段兑现
   - 事实层证据等级
   - 反校验权限
   - 首章高压标准
   - 章尾单独否决权
   - 若与其他 quality 说明文档表述轻重不一致，以 `review-gate-protocol` 为优先
3.6. `full / lean` 模式额外读取 `story-review/references/review-acceptance-checklist.md` 作为固定验收清单：
   - 主线程综合裁决后，必须按清单逐项填写
   - 不得跳项，不得只写自由发挥版总结代替清单
   - 若当前范围已存在旧版固定验收清单，优先更新旧清单，不另起互相冲突的新版本
3.7. 若本轮命中 `F2` 或 `F5`，`full / lean` 还必须额外读取 `story-review/references/failure-postmortem-checklist.md`：
   - 主线程必须回填“问题发生在哪一步”的复盘单
   - 不得只重复审查结论，必须倒查到 `细纲 / 写前闸门 / 正文 / 写后验收 / 追踪同步 / 反校验` 的具体失手点
3.8. 若同一章节范围已连续复审，主线程必须额外判定本轮应继续“补丁式抠句子”还是升级为“一次性收口重写”，并把裁决直接写进工单
3.9. 若同一章节已进入“一次性收口重写”后，仍连续 2 轮以上卡在同一条“确认链 / 处置链过完整”，主线程必须额外判断是否属于规则误伤：
   - 若正文已形成连续收益链，且每一拍都带来现实变化，不得继续仅以“链条完整”判 `F2`
   - 必须明确写出：`继续判不过线` 或 `改判为可接受连续收益链`
   - 不得在未补充新证据的情况下，循环复用同一句“仍有流程感/程序感”推动继续补丁式返工
4. 识别目标平台（检查 `.active-book` 或用户指定），加载对应 rubric：
   - 番茄小说 → 读取 `story-review/references/rubrics/fanqie.md`
   - 起点 → 读取 `story-review/references/rubrics/qidian.md`
   - 知乎盐言 → 读取 `story-review/references/rubrics/zhihu.md`
   - 未指定 → 默认加载 `story-review/references/quality-rubric.md`
5. **题材自动识别（默认执行，不得省略）**：
   - 主线程必须自动识别当前作品题材标签，用于挂载专项 checks
   - 识别顺序固定为：
     1. `设定/核心设定*.md`
     2. 书名
     3. `大纲/卷纲/细纲`
     4. 前 `1-3` 章正文
     5. `对标/` 目录中的作品信息
     6. 用户本轮显式指定
   - 自动识别只允许“加检查”，不允许“减检查”
   - 若识别出多个标签，默认全部挂载；宁可多挂，不可少挂
   - 若识别置信度不足，不得降标准，默认挂载：
     - `通用爽文检查`
     - `留存检查`
     - `章尾检查`
     - `AI味检查`
     - `题面兑现检查`
   - 题材自动识别结果至少应包含：
     - `主平台标签`：如番茄男频 / 起点男频 / 知乎短篇
     - `主卖点标签`：如系统流 / 情报流 / 预知流 / 复仇流 / 打脸流 / 权谋流 / 公路流 / 流放流
     - `结构标签`：如快节奏爽文 / 公路线破局 / 连续翻盘文 / 查线破局文
6. **题材专项 checks 自动挂载（默认执行）**：
   - 若识别到 `番茄男频爽文`，自动挂载：
     - 爽点密度检查
     - 场面兑现检查
     - 打脸/翻盘/改规矩频率检查
     - 中段掉速检查
   - 若识别到 `情报流 / 预知流 / 系统流 / 任务流`，自动挂载：
     - 金手指是否持续主驱动
     - 情报/系统是否在 `1-2` 章内转成现场兑现
     - 是否滑成普通查线文
     - 是否连续多章只靠普通推理推进
   - 若识别到 `权谋流 / 官场流 / 夺权流`，自动挂载：
     - 权力变化是否真实落地
     - 是否只剩说局势、不落现实盘面
     - 是否拿到现实权，而不是只拿到线索
     - 配角是否只会被主角点醒
   - 若识别到 `复仇流 / 打脸流`，自动挂载：
     - 仇恨目标是否持续在场
     - 打脸是否真让对手丢命/丢脸/丢位/丢规矩
     - 是否变成只查仇人，不狠狠干仇人
   - 若识别到 `流放流 / 公路流 / 边荒逃生流`，自动挂载：
     - 行程节点是否持续出险、出局、出翻盘
     - 地图切换是否只是换地名，不换局势
     - 是否掉成重复过路、重复搜线、重复押送
   - 若识别到 `美女/黑市/高位盟友/危险合作对象` 等高价值角色线，自动挂载：
     - 高价值角色显影检查
     - 可脑补感检查
     - 是否只剩功能，不剩魅力
7. 如果审查对象是长篇正文，同时额外加载 `story-review/references/review-gate-protocol.md` + `../story-long-write/references/production-hard-rules.md` 作为统一底层硬规则。审查标准若与其他 rubric 冲突，以这两份协议为优先。
8. 把审查结果直接映射到“是否允许并入主正文”：
   - 仅句子层噪音，可判 `可并入主正文`
   - 命中章节层问题，默认 `不可并入主正文`
   - 命中机制层问题，默认 `不可并入主正文且不可继续顺写`
   - 不允许输出“有问题但先写下去再慢慢修”这类结论
8.5. **首次 full 审查分层结论强制输出**：
   - 第一次对某一章节范围做 `full` 审查时，主线程必须一次性同时给出以下 5 层结论，禁止只报其中一层：
   - `纯净度结论`：是否混入工具残片、广告感、非故事语料、禁用词污染
   - `一致性结论`：是否有错名、时间线冲突、情报日次冲突、角色/任务突兀出现
   - `留存结论`：是否能发书、能否留住读者、是否存在“查得很深、赢得很少”
   - `人物显影结论`：人物脸、说话方式、动作锚点、YY价值是否过线
   - `高潮兑现结论`：主爆点是靠现场翻盘，还是靠解释、宣读、旁白定性
   - 这 5 层里即使某层“未命中问题”，也必须显式写 `已过线`，禁止省略
8.6. **分层过线声明强制输出**：
   - 如果本轮发现“上一轮主要清掉的是一致性层/纯净度层，这一轮新暴露的是留存层/人物层/高潮层”，主线程必须显式写清
   - 禁止把“不同层级的新问题”写成“上一轮没审出来”这类会造成口径冲突观感的表述
   - 必须明确区分：
     - `前轮已过线`
     - `本轮新增检查维度`
     - `本轮首次命中的问题层`
8.7. **同范围复审差异说明强制输出**：
   - 若同一章节范围已做过至少一轮 `full` 审查，本轮报告必须新增一个短节：
   - `与上轮差异`
   - 至少写清：
     - 哪些问题是旧问题未清
     - 哪些问题是本轮新增维度才打出来
     - 哪些层级已明确过线，不再重复重判
   - 禁止让用户产生“上一轮像通过、这一轮又推翻”的阅读感
9. 额外做“平台留存校验”：
   - 开头 100-300 字内是否进事，而不是先解释背景
   - 前 3 段是否已有冲突、异常、死局、悬念之一
   - 每章章尾是否有翻页动力，而不是空预告或作者盖章
   - 是否连续出现“平收章尾”
   - 每章是否至少拿到一种外显战果：命、人、名、权、线索、物证、站队、规矩变化
   - 是否存在“查得很深，赢得很少”
   - 是否存在“场面应当很硬，但正文只剩解释和判断”
   - 正文是否混入补丁标记、工具调用残片、执行日志残片、非正文元信息
   - 章尾是否停在故事文本上，而不是 `*** End Patch`、工具名、说明句、执行残片
9.5. **章尾硬规则（强制执行，不得降级为软建议）**：
   - 章尾判型、合格条件和单章否决权，以 `story-review/references/review-gate-protocol.md` 第八节为准
   - 本节只负责执行补充：
     - 主线程必须单独抽看每章最后 `5-15` 行，禁止只凭整章印象判断章尾是否过线
     - `污染型章尾` 直接记为纯净度问题 + 留存问题
     - 连续升级规则必须执行，禁止淡化：
     - 连续 `2` 章章尾不过线：至少上升为 `章节层问题`
     - 连续 `3` 章中有 `2` 章章尾不过线：至少上升为 `留存层问题`
     - 连续 `3` 章及以上以预告/总结/平收章尾收束：默认触发 `章节组重组建议` 或 `章尾专项回炉`
     - 如果正文主体问题较轻、章尾问题较重，主线程不得用“整体还行”冲淡章尾结论
     - 如果主线程发现子代理把章尾读成旧版本，必须当场纠偏，并在最终报告里说明“章尾已按当前文件重判”
9.6 **句子功能审查硬规则**：
   - 不得把“没命中禁词”当成“句子过线”
   - 必须额外判定句子是否存在以下功能性偷懒：
     - 抽象强弱词顶动作
     - 气氛判断词顶反应
     - 作者结论句顶现场
     - 角色台词顶作者翻译
     - 先说结果/气氛，再补动作过程
   - 命中以上任一类，即使换了别的词，也至少记为 `F1`
9.7 **首章高压标准**：
  - 如果审查对象包含第 1 章，首章门禁判定以 `story-review/references/review-gate-protocol.md` 第七节为准
  - 本节只负责执行补充：
     - 必须显式写出首章到底死在 `开局入局 / 黑名单解释句 / 题面另一半未露头 / 外部盘面未改写` 的哪一项
     - 第 1 章任何一项明显失败，默认不得因“后面会补”而放行
     - 必须再额外显式写出：`章尾是否合格` 与 `高潮兑现是否过线`
     - 若出现“章尾合格，但高潮后续收益未成立 / 外部盘面未改写 / 题面另一半未露头”，不得把章尾合格误判成首章过线
9.8 **解释腔阈值硬规则**：
   - 解释句密度阈值、黑名单和失败落点，以 `story-review/references/review-gate-protocol.md` 第二节为准
   - 本节只负责执行补充：
     - 不得只凭主观印象说“解释有点多”
     - 必须额外检查以下句型是否承担了局势翻译功能：
       - `这不是……这是……`
       - `真正的……是……`
       - `这意味着 / 这说明 / 这表明 / 这证明`
       - `最要命的是 / 因为所有人都看见了 / 可比任何一句狠话都更管用`
       - 场面已成立后再补一句意义总结
     - 若这些句子主要承担“把读者本可看懂的效果再翻译一遍”，不得只记轻度句子噪音
9.8.1 **解释句黑名单（命中即优先复检）**：
   - 黑名单列表与命中阈值，以 `story-review/references/review-gate-protocol.md` 第二节为准
   - 主线程必须继续追根到 `G1/G2/G3/G4`，不得停在“命中黑名单”
9.9 **人物显影最低配额检查**：
   - 最低配额与失效边界，以 `story-review/references/review-gate-protocol.md` 第三节为准
   - 本节只负责执行补充：
     - 对主角、核心反派、核心盟友、高价值角色，必须额外检查“角色卡锚点是否已回落正文”
     - 报告中不得只写“显影偏弱”，必须写缺的是 `外形 / 动作 / 台词区分 / 气场 / 可脑补感` 哪一项
9.10 **配角工具嘴检查**：
   - 硬判边界以 `story-review/references/review-gate-protocol.md` 第三节为准
   - 本节只负责执行补充：
     - 报告里必须写清该角色缺的是 `站位铺垫 / 身份味 / 私心 / 代价 / 延迟反应` 哪一项
     - 不得只写“配角功能化”
9.11 **后半章信息场检查**：
   - 硬判边界以 `story-review/references/review-gate-protocol.md` 第四节为准
   - 本节只负责执行补充：
     - 主线程必须单独看单章后 `1/3`
     - 报告里必须写清后半章滑成的是 `抬黑幕 / 抬线索价值 / 确认价值 / 变脸失声 / 解释局势` 中的哪一类
9.12 **章尾最后一句总结味单独拦截**：
   - 硬判边界以 `story-review/references/review-gate-protocol.md` 第八节为准
   - 本节只负责执行补充：
     - 不得把“章尾不是预告/不是信息型”就误判为过线
     - 这是独立失败类，不能只按“有一点解释腔”轻描淡写带过
     - 修法必须明确指出：章尾应停在前面的动作/物证/失态/压制结果处，而不是补那句意义总结
10. 额外做“人物显影度/YY价值校验”：
   - 重要角色是否有稳定外形锚点，而不是只剩功能描述
   - 角色是否有可识别的小动作、习惯动作、压人姿态或气场动作
   - 角色是否做到“一开口就像他本人”，还是只是台词功能不同
   - 帅哥/美女/高价值角色是否真的被写得“值得看”，能让读者脑补，而不是只在设定里说好看
   - 是否存在“角色功能明确，但人物脸是糊的”
   - 是否存在“人物只推进剧情，不提供苏感、压迫感、魅力或想象空间”
   - 关键角色是否已有对应 `设定/角色/{角色名}.md`，还是正文已成型但角色卡缺失
   - 角色卡里的外形/动作/说话/气场锚点，是否已经回落到正文，而不是只写在卡里
   - 正文新增的人物锚点，是否已经回写角色卡
   - 连续登场角色是否只是首轮补过一次，后续又退回“功能位说话”
   - 删除角色名后，主角组/核心配角的台词是否还能分辨；若不能，直接判人物显影未过线
11. 额外做“题面/卖点一致性校验”：
   - 书名、文案、金手指、题材卖点，是否在正文中持续兑现
   - 若核心卖点是“每天一个情报/系统/身份/复仇/打脸”等，必须检查是否高频参与推进，而非中途蒸发
   - 若正文已滑成另一种书（如普通经营、普通县令、普通查案），必须上升为机制层问题
11.5 **情报兑现时限检查**：
   - 硬判边界以 `story-review/references/review-gate-protocol.md` 第五节为准
   - 本节只负责执行补充：
     - 报告里必须写清情报失效死在 `未入台账 / 未限章兑现 / 包裹格式缺失 / 来源触发缺失 / 只抬线索不兑现` 的哪一项
     - 不得只写“情报参与度不足”
12. 如果存在 `追踪/` 目录，必须额外读取：
   - `追踪/情报台账.md`
   - `追踪/时间线.md`
   - `追踪/角色状态.md`
   - `追踪/伏笔.md`
   - `设定/角色/`（至少检查本轮关键角色对应文件是否存在）
   并开启“追踪反校验”：
   - 正文已落地关键角色是否入表
   - 正文显式时间锚是否进入时间线
   - 情报是否被台账写满头或写过头
   - 追踪文件是否沿用旧稿口径
   - 关键角色是否缺角色卡，或角色卡仍沿用旧稿口径
   - 追踪文件是否把“待补强/计划补/理论成立”写成“已完成/已补强/已落地”
   - `时间线.md`、`角色状态.md`、`情报台账.md` 是否存在同一事项双版本并存、旧版未废止
   - 角色状态文件里写“已显影/已补强”，但正文抽查仍站不住的，按一致性问题直接记入
12.1 **事实层证据等级校验（强制执行）**：
   - 事实层 / 预案层隔离、措辞降级规则和禁止口径，以 `story-review/references/review-gate-protocol.md` 第五节为准
   - 本节只负责执行补充：
     - 重点核查 `设定/角色/*.md` 的“当前已落地事实”区块、`追踪/角色状态.md`、`追踪/情报台账.md`、`追踪/时间线.md`
     - 与本章绑定的 `细纲 / 写前闸门 / 写后验收 / 反校验` 也必须一并核对
     - 不得以“方向对了”“大体会这么发展”放行事实层写满
12.2 **反校验不得自报通过**：
   - 判定边界以 `story-review/references/review-gate-protocol.md` 第六节为准
   - 本节只负责执行补充：
     - `反校验` 只能在“正文、闸门、追踪、角色卡”四组口径全部降回同一证据等级时判 `通过`
     - 禁止把“已经比上轮更接近”当成“反校验通过”
13. 如果审查范围涉及连续 `3` 章及以上，必须额外做“回炉方式判断”：
   - 这组问题是否仍适合单章局改
   - 是否已经进入“连续多轮返工仍未清干净”的状态
   - 是否应改为“先生成固定验收单，再一次性收口”
   - 如果答案是“应”，则不得只给散点问题，必须直接输出该判断
14. 如果审查对象已经历多轮回炉，必须额外区分：
   - 现在剩下的是句子噪音
   - 还是厚度病
   - 还是机制病
   禁止把“机制病已清，只剩厚度病”与“仍不可并入主正文”混成一类模糊结论

## Phase 2：并行 Spawn 4 个子代理（+ 可选 researcher）

使用 Codex 子代理机制并行调用 4 次，对应 `story-architect`、`character-designer`、`narrative-writer`、`consistency-checker`。

**调用规则**：每个子代理不依赖父线程隐式上下文，输入必须自包含文件路径和上下文。
Prompt 内必须显式写入“参考文件路径规则”，并列出本次要用到的规范路径；禁止让子代理读取裸文件名。
Prompt 内还必须显式写入：`story-review/references/review-gate-protocol.md` 是本轮门禁主协议，若其他说明文档与其冲突，以它为准。
若为 `full / lean`，Prompt 内还必须显式写入：主线程最终要回填 `story-review/references/review-acceptance-checklist.md`，子代理给出的结论必须能被主线程直接映射进固定验收清单，不得只给无法落勾的抽象评语。

  ```
  6. 是否存在解释腔、作者代言句、爽点后二次解释？
  7. 章尾是否是盖章句，而不是具体危险/发现/反扑？
  8. 是否高频出现“这不是……是…… / 这一下…… / 值钱的是…… / 因为这…… / 这就说明……”等解释腔句式？
  9. 是否存在“事情明明已经发生，又补一句旁白定性”？
  9.5. 是否命中解释句黑名单：`他只知道一件事 / 他这才明白 / 今夜若……他会死 / 最后留下来的，是…… / 情报一个字没错 / 就是现在 / 就这一瞬，够了 / 但这份……撑不了多久`？
  9.6. 开头 `500` 字内黑名单解释句是否超过 `1`；若超过，是否至少上升为 `F2`？
  10. 人物描写是否稀薄到读者难以想象，尤其重要角色出场时是否只给功能不给画面？
  11. 是否存在“帅哥美女口头上好看，正文里却没有任何值得看的细节”？
  12. 如果连续章节已经改了很多轮，是否仍在用补丁式语言修补章节机制病？
  13. 当前残留问题是句子噪音，还是“正文厚度仍不足以支撑主爆点”？
  14. 正文是否混入补丁标记、工具调用残片、执行日志残片、渠道名或其他非故事语料？
  15. 章尾是否被非正文内容污染，导致读者直接出戏？
  16. 必须单独摘出每章最后 `5-15` 行，判定章尾属于：合格 / 预告型 / 总结型 / 信息型 / 平收型 / 污染型，禁止只写“有悬念”这类模糊结论
  17. 章尾是否只是“下一步要做什么”的计划句、布置句、判断句，而不是当场发生的新咬合？
  18. 是否把“不是盖章句”误判成“章尾过线”？
  19. 若连续两章及以上章尾不过线，必须显式上报为连续章节留存问题，不能只分散记在句子层
  20. 若章尾不过线，即使正文主体尚可，也必须给出 `不可并入主正文` 的建议，不得放行

  输出格式：
  VERDICT / 结论: APPROVE(通过) / CONCERNS(有问题) / REJECT(需重写)
  FINDINGS: AI味级别: 轻度/中度/重度; [禁用词/格式/节奏问题，附具体引用]
  RECOMMENDATIONS: [修改建议]
  CHECKLIST_HINTS: [可直接回填固定验收清单的勾检结论]
  ```

## Phase 3：综合裁决

1. 收集 4 个子代理的 VERDICT 和 FINDINGS
2. 合并去重：将各子代理的 FINDINGS 按严重程度排序（S1 > S2 > S3 > S4，机制问题 > 章节问题 > 句子问题，AI味重度 > 中度 > 轻度）
3. **主线程统一压缩**：父线程必须先把所有问题压缩成“少数几条主问题”，禁止照抄子代理长清单。优先级固定为：
   - 能不能继续发
   - 会不会掉留存
   - 是单章噪音，还是连续章节/细纲机制问题
   - 第一批该怎么开刀
3.5. **先报过线层，再报不过线层**：
   - 如果一致性层、纯净度层已过线，主线程必须先用 1-3 句显式报出
   - 然后再报留存层、人物层、高潮兑现层的问题
   - 目的不是表扬，而是防止不同轮次的审查口径看起来互相打架
4. **问题聚类**：不要把同一问题换壳重复罗列。至少先聚成 3 类：
   - 机制主问题：如“信息推进代替情绪推进”“连续抬层级但兑现不足”“配角功能化”
   - 章节主问题：如“某几章功能重复”“转场失温”“章尾盖章”
   - 一致性主问题：如“角色口径冲突”“时间线脱节”“追踪漏记”
   每类先写一句总判断，再列具体章节/文件证据。
4.5. **章尾裁决硬闸**：
   - 主线程综合裁决时，必须单列 `章尾结论`
   - `章尾结论` 只能写：`已过线` / `未过线`
   - 判定 `未过线` 的常见命中类型必须写清：`预告型` / `总结型` / `信息型` / `平收型` / `污染型`
   - 不得使用“还可以”“基本有悬念”“问题不大”这类模糊口径替代章尾结论
   - 章尾 `未过线` 时，不得因为正文主体较强就仍判 `可并入主正文`
   - 如果章尾问题跨连续章节出现，必须并入 `问题分层` 与 `回炉优先级清单`，不能只留在证据区
5. **可选事实核查**：如果审查内容涉及需要验证的外部事实（历史年代、地理方位、职业细节等），并且项目已部署 `story-researcher`（检查 `.codex/agents/story-researcher.md` 是否存在），额外 spawn 一个子代理：

```text
代理名：story-researcher
输入：
- 项目目录：{dir}
- 任务描述：事实核查
- 研究主题：{待核查的外部事实}
- 输出方式：返回研究结论并附主要依据
```

将研究结果纳入裁决参考；如子代理不可用，由主线程直接核查。
6. **分歧呈现**：如果子代理间有冲突意见，明确呈现分歧让用户裁决
   - 例：story-architect 认为某段"结构合理"，但 character-designer 认为"角色弧线有问题"
   - 不要自动妥协，让用户看到双方理由
7. **修复范围判定**：综合后必须把问题分成三层：
   - 可单章修
   - 需连续章节修
   - 需细纲/卷结构回炉
   禁止把机制层问题降格成句子层问题。
7.1. **章尾失败升级判定**：
   - 单章章尾不过线：默认至少记入 `章节层`
   - 连续 `2` 章章尾不过线：默认 `需连续章节修`
   - 连续 `3` 章中有 `2` 章章尾不过线，或连续 `3` 章均为预告/总结/平收：默认 `留存层问题`，并优先触发 `章节组重组建议`
   - 如果章尾只是正文最后一句的问题，可单章修；如果章尾不过线来自整章兑现不足、临门一脚缺失、冲突落空，不得伪装成句子层修补
7.2 **回炉方式判定强制输出**：如果本次审查对象涉及连续 `3` 章及以上，综合后必须额外明确写：
   - `适合补丁式回炉`
   - `不适合补丁式回炉，需固定验收单`
   二者至少命中其一，不得省略。
7.5 **并入裁决强制输出**：综合后必须明确写：
   - `可并入主正文`
   - `不可并入主正文`
   - `不可继续顺写`
   三者至少命中其一，不得模糊表述。
7.6 **残留问题层级强制输出**：综合后必须再明确写：
   - `当前剩余：句子噪音`
   - `当前剩余：厚度病`
   - `当前剩余：机制病`
   至少命中其一。禁止只说“还有问题”。
7.7 **AI味后续动作强制输出**：
   - 如果 `narrative-writer` 或主线程裁定 `AI味 = 中度` 或 `重度`，不得只停留在审查报告里报问题
   - 必须额外明确输出以下之一：
     - `追加流程：story-deslop 检测`
     - `追加流程：story-deslop 修文`
   - 判定规则：
     - `轻度`：可只在当前审查报告中列句层问题，不强制转 `story-deslop`
     - `中度`：默认强制转 `story-deslop 检测`
     - `重度`：默认强制转 `story-deslop 修文`
   - 若用户本轮只要求审查、不要求改文，也至少必须把 `story-deslop` 动作写进工单与报告，不得只口头提醒
8. **回炉优先级清单**：full/lean 模式下必须输出一份施工单，至少包含：
   - `必须重写`
   - `建议并章/重排`
   - `可局改保留`
   - `先修什么，后修什么`
8.5 **施工单落盘强制输出**：full / lean 模式下，主线程除了在回复里输出施工单，还必须把最终采用的施工方案落盘到项目内 `追踪/`：
   - 默认文件类型：`总施工单`
   - 默认命名：`追踪/总施工单_{审查范围摘要}.md`
   - 若审查范围是全书或大段连续章，优先复用并覆盖最近一份同主题总施工单，而不是重复新建散单
   - 若项目中已有更通用的总施工单，如 `追踪/总施工单_001-053全书总审后续修复.md`，本轮结论应直接更新该文件，除非审查范围已明显切换到另一组章节
   - 禁止只在对话里给出施工动作，不在项目文件中留档
   - 如果项目不存在 `追踪/`，先创建后再落盘
8.6 **固定验收清单落盘强制输出**：`full / lean` 模式下，除总施工单外，还必须同步生成或更新固定验收清单：
   - 默认命名：`追踪/固定验收清单_{审查范围摘要}.md`
   - 若当前范围已多轮回炉，优先覆盖更新同范围旧清单，不要并行保留多个冲突版本
   - 清单内容必须以 `story-review/references/review-acceptance-checklist.md` 为模板，不得自由改骨架
8.7 **失效原因复盘单落盘强制输出**：若本轮命中 `F2` 或 `F5`，除总施工单与固定验收清单外，还必须同步生成或更新失效原因复盘单：
   - 默认命名：`追踪/失效原因复盘单_{审查范围摘要}.md`
   - 若同范围已有旧复盘单，优先覆盖更新，不并行保留冲突版本
   - 复盘单必须以 `story-review/references/failure-postmortem-checklist.md` 为模板，不得省略一级标题
9. **章节组重组建议**：如果综合结论里出现“需连续章节修”或“建议并章/重排”，主线程必须继续往前一步，至少给出 1 份可执行的章节组切刀方案。格式固定为：
   - `原章节组`：如 `011-018`
   - `建议重组为`：如 `011-012 / 013-014 / 015-016`
   - `每组核心功能`：每组一句，只写“这一刀负责兑现什么”
   - `为何这样切`：一句，说明是为了解决重复推进、兑现滞后、角色跳步还是地图换皮
   禁止只写“建议压缩”而不写压成什么。
10. **最终采用方案拍板**：如果子代理或主线程产出了多份章节组重组建议，主线程不能只并列展示，必须最终拍板收束成 `1-2 套最终采用的章节组重组方案`。每套至少包含：
   - `适用问题段`
   - `最终采用重组`
   - `采用理由`
   - `为什么不采用其他方案`
   如果只有 1 套明显更优，也必须明确写“最终采用此方案”。
11. 输出综合审查报告
11.5. **固定验收清单强制回填**：
   - `full / lean` 模式下，主线程输出报告前，必须先完成固定验收清单回填
   - 若清单存在未填项，不得结束本轮审查
   - 报告中的 `五层结论`、`失败码汇总`、`5行施工单`、`修复范围建议` 必须与清单一致，不得各写各的
11.6. **失效原因复盘单强制回填**：
   - 若本轮命中 `F2` 或 `F5`，主线程输出报告前，必须先完成失效原因复盘单回填
   - 若复盘单没有明确写出“失败主要发生在哪一步”，本轮审查不得结束
   - 复盘单必须回答：这次问题到底是 `写前没钉死 / 写中滑掉 / 写后写满 / 反校验放过` 中的哪一种

12. **若命中 AI 味或句子功能失败，必须区分根因**：
   - `词表问题`：个别高危词过密
   - `句法问题`：程度副词、判断句、解释句承担主表达
   - `结构问题`：场面缺位，只能靠抽象词补强
   - `流程问题`：写作阶段未执行写前闸门/写后验收
   主线程必须写清主因，不得把所有问题都归因到“用了某个词”。
13. **解释腔 / 功能句必须继续下钻到生成根因**：
   - 主线程与子代理命中解释腔、功能句、作者代判句时，不得止步于“有解释腔”这一层，必须继续判到以下四类之一：
     - `G1 信息抢跑`：正文抢着替读者说明设定、局势、价值
     - `G2 现场承载不足`：动作、后果、他人反应、现实收益没写够，只能靠判断句补缝
     - `G3 提纲语言漏出`：章节意图、结构调度、线索调度直接漏进正文
     - `G4 人物失声`：人物本该靠动作/对白显影，却由旁白替他翻译
   - 若不能判到这四类之一，说明审查仍停留在现象层，报告不得结束
   - 裁决映射：
     - `G1/G4` 默认至少 `F1`
     - `G2` 默认至少 `F2`
     - `G3` 默认至少 `F4`
   - 同一段若同时命中 `2` 类及以上，主线程必须明确写“不能局删，需整段回写”
13.5 **解释句密度必须单独统计**：
   - 统计口径与阈值以 `story-review/references/review-gate-protocol.md` 第二节为准
   - 本节只负责执行补充：
     - 审查正文时，不能只抓单句，还必须统计解释句/作者代判句/总结句的密度
     - 若解释句密度高但单句都“看似合理”，仍按密度问题裁决，不得被单句表面合理性掩盖
     - 章尾最后 `1-2` 句若是意义总结、阶段收口、局势翻译或替读者下结论，必须单列为章尾解释句问题，不能只记成普通解释腔
14. **解释腔 / 功能句修法必须指向替代承载物**：
   - 报告里不能只写“删掉这句”“改自然一点”
   - 每条问题都必须明确指出该句应该替换成哪类现场承载：
     - `动作`
     - `物证`
     - `他人失态`
     - `现实收益`
     - `规矩变化`
   - 若报告没有给出替代承载物，视为修改建议无效
15. **解释腔高发时必须回溯到上一层失败位置**：
   - 若单章内解释腔/功能句累计 `3` 处及以上，主线程不得只判“句子层问题”
   - 必须进一步判断是：
     - 细纲没钉死本章最值钱战果
     - 正文后半章现实收益不足
     - 章尾临门一脚缺失
   - 一旦命中上述任一项，至少上升为 `章节层问题`
16. **高潮兑现必须拆成“两段收益”审查**：
   - 双段兑现的硬判边界以 `story-review/references/review-gate-protocol.md` 第四节为准
   - 本节只负责执行补充：
     - 主线程与子代理在审“高潮兑现”时，不得只问“有没有硬场面”
     - 必须强制拆成：
       - `第一爆点是否成立`
       - `高潮后续收益是否成立`
17. **字数必须纳入写后验收硬项**：
   - 审查正文时，必须检查本章实际字数是否达到细纲/默认目标
   - 若本章没有单独标注字数目标，默认按 `3000` 字标准判断
   - 若字数未达标，必须记为章节层问题，不能用“节奏快”替代
   - 若只有前者成立，例如“打赢了 / 反杀了 / 抢到了”，后者只剩：
     - 对方变脸
     - 知道得更多
     - 事情不简单
     - 下一章再处理
     默认记为“高潮只兑现一半”，至少 `F2`
17.5 **高潮后续收益只认现实代价，不认抽象危险**：
   - 认定边界以 `story-review/references/review-gate-protocol.md` 第四节为准
   - 本节只负责执行补充：
     - 报告里必须写清本章拿到的是 `命 / 人 / 名 / 权 / 物证 / 站队 / 规矩变化 / 垫背者落地` 中的哪一类
     - 若一类都答不上来，不得用“局势更复杂”冒充后续收益
18. **首章高潮从严：只活命不够，必须看“题面另一半”是否露头**：
   - 首章高压的硬判边界以 `story-review/references/review-gate-protocol.md` 第七节为准
   - 本节只负责执行补充：
     - 若第 1 章主收益只有“活下来了”，而书的题面还承诺了复仇 / 夺权 / 打脸 / 上位等更高卖点
     - 主线程必须额外写明：
       - 本章是否把“第一批该狠狠干回去的人”钉出来
       - 本章是否把“第一口势”落成别人不敢再按原规矩处理
19. **后半章滑成信息场时，高潮问题优先于章尾问题**：
   - 若章尾本身停得住，但单章后半段主要在抬线索、抬黑幕、抬危险，而没有新的现实翻面
   - 不得误判成“章尾已过线所以高潮也过线”
   - 此时必须单列：
     - `章尾结论`
     - `高潮兑现结论`
   - 允许出现：
     - `章尾已过线`
     - `高潮兑现未过线`
   - 禁止用章尾合格掩盖高潮只兑现一半的问题
19.1 **首章章尾合格不得冲销首章失败**：
   - 若第 1 章章尾已停在具体危险 / 物证 / 反扑现场，但整章仍未拿到现实收益
   - 主线程必须判：
     - `章尾结论：已过线`
     - `高潮兑现结论：未过线`
     - `综合评定：不可并入主正文`
   - 禁止出现“章尾够硬，所以首章可以先放行”这类口径

**输出压缩规则**：

- 报告正文先给高信号结论，再给证据，不要把同一问题换壳说三遍
- 如果多个子代理在说同一件事，主线程必须合并成一条主问题
- 如果整组问题已经从“机制病”收缩为“厚度病”，主线程必须明确写出，不得继续沿用上一轮的重判口径
- 如果同范围已有上轮 full 审查，必须明确写出“这轮不是推翻上一轮，而是新增更深层检查”的差异说明
- full / lean / solo 最后都必须收束成统一的 `5 行施工单`
- full / lean 除了回复内 `5 行施工单` 外，还必须同步生成或更新 `追踪/总施工单_*.md`
- full / lean 还必须同步生成或更新 `追踪/固定验收清单_*.md`
- 若命中 `F2` 或 `F5`，full / lean 还必须同步生成或更新 `追踪/失效原因复盘单_*.md`
- 如果前文已经写清理由，施工单不再重复解释，只写动作
- full / lean / solo 的最终输出顺序固定为：
  1. `Verdict Summary`
  2. `综合评定`
  3. `题材识别结果`
  4. `已挂载专项 Checks`
  5. `失败码汇总`
  6. `5行施工单`
  7. `章节组重组建议`（如命中连续章节问题）
  8. `最终采用方案`（如存在多套切刀思路）
  9. `问题分层`
  10. `与上轮差异`（如同范围已审过）
  11. `修复范围建议`
  12. `发现的问题`
  13. `修改建议`
  不能把大段分析放到 `5行施工单`、`章节组重组建议` 和 `最终采用方案` 前面

**机制层强规则**：

- 如果综合裁决后判断问题属于“机制层”，禁止只给句子级建议
- 机制层问题至少必须给出以下之一：
  - 连续章节回炉建议
  - 细纲回炉建议
  - 并章 / 砍章 / 提前兑现建议
- 不允许把“连续慢节奏 / 说明书细纲 / 信息推进代替情绪推进”伪装成“改几句表达就行”
- 如果机制层问题已经影响连续章节阅读感，默认建议先停写，不建议继续顺写冲过去
- 如果审查结论为 `REJECT`，且主问题是中段机制层失衡，默认必须写明：
  - 是否建议先停写
  - 停写范围
  - 第一批回炉章节
- 如果已经给出“建议并章/重排”，默认还必须给出对应的“重组后章节功能分配”，避免建议停留在章节号层面
- 如果已经出现 2 套以上可行重组法，主线程必须拍板，不允许把选择权重新丢回给用户
- 如果审查结论为 `CONCERNS`，也必须区分：
  - 哪些只是局部噪音
  - 哪些是“再不处理就会升级成机制层问题”的预警
- 如果审查结论命中章节层或机制层问题，即使只是 `CONCERNS`，也默认 `不可并入主正文`
- 只有当问题明确收束为句子层/局部噪音时，才允许 `可并入主正文`
- 如果发现“追踪文件写已完成，但正文实际未稳定落地”，默认至少记为一致性主问题；若涉及连续章节，默认 `不可继续顺写`
- 如果发现主角组或连续共演角色删去名字后无法分辨，默认至少记为章节层问题；若跨多章普遍存在，直接上升为机制层人物显影失败
- 如果已经落盘过总施工单，本轮审查不得绕开旧工单另起一份互相冲突的新工单；必须先判断：
  - 这是更新同一施工阶段
  - 还是进入了新的施工阶段
  然后明确选择“覆盖更新”或“新建新单”

## Phase 3.5：施工单落盘（full / lean 强制）

在完成综合裁决后、输出最终报告前，主线程必须把审查结论整理为项目内可执行工单。

### 落盘规则

1. 目标目录：
   - 优先写入审查对象所在项目的 `追踪/`
   - 若不存在则创建
2. 文件选择：
   - 全书 / 大段连续章 / 总审 → 使用 `总施工单_*.md`
   - 小范围章节组 → 允许使用 `施工单_*.md`
   - 但如果当前项目已经明确采用“总施工单统管”，则继续更新总施工单，不要退回散单
3. 覆盖策略：
   - 同一轮次、同一范围、同一施工阶段 → 覆盖更新旧工单
   - 范围扩大为更高层级总审 → 升级为更通用的总施工单并说明替代关系
   - 只有范围完全不同，才新建另一份工单
4. 文件内容必须至少包含：
   - 标题
   - 审查来源
   - 当前总裁决
   - 主问题压缩
   - 施工优先级
   - 分批施工
   - 本轮替换 / 废止的旧工单说明（如有）
   - 若命中 `AI味 = 中度/重度`，还必须写入：
     - `去AI味衔接`
     - `建议调用：story-deslop 检测 / 修文`
5. 如果已经判定“不适合补丁式回炉，需固定验收单”，落盘文件里必须显式写出：
   - 固定验收单口径
   - 本轮先修批次
   - 后续复审入口
6. 最终回复里必须额外给出：
   - `工单文件：{path}`
   - `工单动作：已新建 / 已覆盖更新 / 已升级替换`

4. 如果发现明显机制层问题，也必须指出“solo 只能定位，不能假装单章已修复”
5. 如果存在 `追踪/` 目录，也必须做最小追踪反校验：
   - 新角色是否已入表
   - 显式时间锚是否入时间线
   - 情报是否已入台账
6. 即使是 solo，若发现“假完成追踪 / 双版本并存 / 角色台词混同 / 题面滑轨”，也必须明确写 `不可继续顺写`

**输出硬规则补充**：
- `简评` 不得替代裁决；若命中章节层或机制层问题，简评里也必须再次明确 `不可并入主正文` 或 `不可继续顺写`
