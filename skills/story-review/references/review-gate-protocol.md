# 审查门禁协议

> 本文件把 `story-review` 的最低硬规则压成统一协议。执行任意 `full / lean / solo` 审查时，优先读取本文件；其他 references 负责补充方法，不再承担主门禁入口。

---

## 一、默认硬审范围

无论用户有没有追加检查项，默认都必须审：

1. 正文纯净度
2. 一致性
3. 题面兑现
4. 场面兑现
5. 章尾硬规则
6. 人物显影
7. AI味 / 解释腔 / 作者代判句
8. 追踪反校验
9. 连续章节留存
10. 并入裁决
11. 篇幅预算

规则：
- 用户追加项只能加码，不能替代默认硬审
- 命中门禁层，必须给失败码、并入裁决和施工动作
- 命中需中止推进的问题时，必须同步给 `建议接管态`：`H1-A / H1-B / 无`
- 若任务口径是纯测试 / 基准验证 / 回归验证，建议接管态只能是 `H1-A` 或 `无`
- 若任务口径是生产修文，且失败码不是 `F0`，建议接管态默认是 `H1-B`

补充功能场景反说明书审查：

- 若章节命中 `经营 / 配额 / 分配 / 卖货 / 流程口 / 制度口 / 试炼口 / 审批口 / 封存口 / 看房配套 / 资格口 / 交易口` 任一功能场景，必须额外审：
  - 规则信息是否压在在场动作里
  - 是否真的发生了 `试用 / 交易 / 排队 / 讨价 / 受阻 / 改规矩 / 利益变化`
  - 该段删掉后，场面推进会不会明显变弱

补充“解释腔替代现场结果”审查：

- 若正文已经通过 `器物 / 站位 / 名单 / 纸面落字 / 广播 / 倒计时 / 门位 / 供桌 / 礼台 / 一号桌` 等现实载体，把结果写到现场里了，review 必须继续检查作者是否又补了盖章句
- 以下句式默认优先按 `解释.error` 复核，而不是轻放成普通抒情：
  - `这就是现实起效`
  - `现在不是……`
  - `不是X被解释了，而是Y发生了`
  - `这就不是只……`
  - `这才是……的真值`
- 以下句式默认优先按 `模板写直 / 作者代判句` 复核，而不是轻放成普通过渡：
  - `这是第一口群体改线`
  - `这是第一口外部反应`
  - `第二口外部反应也来了`
  - `这不是帮腔`
  - `这不是只下场`
  - `问，就是要写`
- 若删掉这类句子后，现场意义仍然成立，说明它们只是作者代判句，默认至少 `F1`，跨关键收益位时至少 `F2`
- 若项目写后验收或短执行核里只写了“解释腔已自检 / 待脚本复检 / 已压住”，却没有留下 `原句 -> 替换结果`，review 必须继续追责为“执行层未实填”，不得把它误判成单纯正文偶发失手
- 若正文命中 `解释.error` 的同时，写后验收仍把该项写成 `待脚本复检`，默认优先判为：
  - 不是入口模板缺承载
  - 而是 `解释腔前置自检卡` 执行不够硬
  - 施工单应写成“加硬执行实答”，不得只写“删几句解释”

补充“假进步 / 过闸稿”审查：

- review 不得把 `命中下降` 直接等价成 `正文变好`
- 若回修后主要变化是：
  - 角色更会解释、更统一、更像成熟写手腔
  - 后果被改写成结论句、总结句、关系判断句
  - 冲突被改写成平顺的成熟对话块
  - 人物毛边、迟疑、嘴硬、遮掩一起被磨平
  则必须继续复核是不是 `假进步 / 过闸稿`
- 以下问题不看“句子顺不顺”，只看“现实有没有更硬”：
  - 修改后到底多了哪一拍现实后果
  - 谁的顺位、站位、利益、身体状态因此被改掉
  - 谁的私人别扭、私心、自保因此更清楚
- 若回答不出这些，但只能回答“更流畅 / 更成熟 / 更稳”，默认不得高判
- 若问题命中主要来自“把会触发脚本的句子磨平”，却没有把动作链、后果链、人物链补硬，至少 `F2`
- 若复审样本呈现“错误更少，但读起来更像验收稿”，主线程必须显式写出：
  - `问题减少`
  - 但 `活性回撤`
  二者不得混写成“整体改善”

补充“规则播报 vs 情报流”审查：

- 若章节命中 `广播播报 / 系统播报 / 轮次提示 / 倒计时变化 / 复核口令 / 条文亮起`，review 必须先判它到底属于：
  - `规则执行口`
  - `正式情报文本`
- 只有能跨章追踪、能单列为正式文本本体、且能明确回答 `刷新章 / 预计兑现章 / 现实收益类型` 的，才可判为情报流
- 若其主功能只是改当前口令、顺位、站位、执行权，不得因为脚本命中 `情报.error` 就机械把本章打成情报流失败；必须先看是否属于分流误触发
- 若确定是分流误触发，主线程必须把它记为 `流程级待补规则`，而不是逼正文补假情报块

补充“规则已写入但入口未承载”审查：

- review 不能只看高层协议是否有规则，还必须追查这些规则有没有真正进入本轮实际使用的入口模板：
  - `短执行核`
  - 写前闸门最小模板
  - 写后验收最小模板
- 若高层协议已要求，但上述入口文件没有对应字段、没有实填、或只写空话，必须明确判为 `流程级失守`
- 以下情况默认至少 `F5`：
  - `现实挤压补位卡` 已被协议要求，但 `短执行核` 无承载位
  - `解释腔前置自检卡` 已被协议要求，但写后验收无结果位
  - `末尾反删测试` 已被协议要求，但写后验收无结果位
  - `方法论资料利用率` 已被协议要求，但入口模板没有“资料 -> 风险 -> 落点”三段记录
  - `模板使用率 / 模板效果检查` 已被协议要求，但写前或写后模板没有结果位
- 若问题根因是“规则存在，入口模板没接住”，施工单必须写成“补入口链路”，不得误写成正文自身失败

## 一点五、功能场景反说明书硬闸

以下情况默认至少 `F2`：

- 功能场景名义上有人在场，实际主要在讲制度、流程、福利、玩法、资格或规则介绍
- 连续 `4` 句及以上主要功能是在解释规则，而不是推动在场人物动作
- 该段没有明确的 `试用 / 交易 / 排队 / 讨价 / 受阻 / 改规矩 / 利益变化`
- 删掉该段后，人物关系和场面推进几乎不受影响
- 该段可以直接改写成产品说明 / 制度公告 / 服务介绍，而意思几乎不变

执行口径：

- 同时命中上面 `5` 条中的 `4` 条，默认按“功能场景写成规则说明书”处理，至少 `F2`
- 若只是单句解释偏重，但段落主功能仍是场景推进，可降到 `F1`
- 若连续 `2` 章以上都把功能场景写成说明书，或因此导致题面一直停在“知道规则更多”，默认升级为 `F3`

补充模板落地率审查：

- `模板使用率` 不等于“文档里列了模板名”
- review 必须拆开核查：
  - 入口模板里有没有写清 `第一爆点 / 第二口收益 / 群体反应位 / 章尾前一拍 / 章尾最后一拍`
  - 正文里这些字段有没有真的落成事件链
  - 写后验收里有没有回填“模板使用率结果 / 模板效果结果”
- 若只存在“已用模板A/B/C”的挂名记录，没有“模板最终改了谁的什么”，默认至少 `F2`
- 若正文可见模板效果，但入口模板和写后验收都没记，默认至少 `F5`

补充“模板写直 / 写满 / 写太快”审查：

- 若模板已经明显进入正文，但正文同时出现以下任两项，默认不得判“模板使用有效”：
  - 角色或旁白直接替模板字段盖章，或直接给模板节点贴标签句
  - 写后验收 / 角色状态 / 章组总表把正文仍在过程里的收益提前高判
  - 对手只有第一口嘴上反压，没有第二层现实反压
  - 第三方没有迟疑、自保、观望成本，直接站队
  - 第002章类章节从“主角再顶一句 -> 对手回一口 -> 第三方补一压 -> 直接落袋”快速闭链
- 命中后必须继续区分：
  - `模板写直`：正文在替细纲翻译工位，或正文直接把 `第一口 / 第二口 / 外部反应 / 改线 / 真值 / 起效` 这些模板节点说破
  - `模板写满`：追踪和验收先替正文宣布胜利
  - `模板写太快`：缺少第二层反压、第三方迟疑、收益代价
- 这三项至少命中一项时，默认不得仅判 `F1`；若影响收益档位或连续章节留存，至少 `F2`

补充“解释压住但厚度回弹”审查：

- 若新样本已经不再命中 `解释.error`，却连续命中 `字数.warn / 字数.error`，review 不得简单写成“只剩字数问题”
- 必须继续追问三件事：
  - 首写前是否真的写了 `厚度回补卡`
  - 回补卡里是否写清 `先补哪三口、各落哪一拍、分别改了谁`
  - 正文新增内容是否真的落成 `争口 / 改站位 / 改顺位 / 执行后果`，而不是只删了解释句
- 若解释腔已经压住，但厚度卡没有实填、或实填后正文看不出对应三口，默认优先判为“厚度补足前置化执行不够硬”
- 这种情况不得误判成“解释问题已解决、正文自然会自己变厚”，施工单必须显式写：
  - 不是回头补解释
  - 而是加硬厚度回补卡执行

补充“成熟安全块”排查：

- 若正文看上去更稳，但连续出现以下任两项，默认按 `成熟安全块` 复核：
  - 角色彼此说话都更完整、更体面、更能总结局势
  - 关键冲突后马上进入互相讲明白的对话，而非继续顶住后果
  - 本应落在动作、失态、顺位变化上的东西，被改成了抽象关系判断
  - 金句、硬句、价值句看似减少，但人物也一起失去辨识度
- `成熟安全块` 不属于“语言偏好差异”，而是正文承载方式退化
- 若命中后同时影响人物分化、章尾、高潮后续收益中的任一项，至少 `F2`

排误伤规则：

- 规则多，不等于说明书
- 若规则信息始终压在 `点单 / 试吃 / 结账 / 排队 / 抢位 / 争执 / 改口 / 改规矩 / 受阻 / 站位变化` 里，不得仅因信息密度高判死
- 若删掉规则信息会直接削弱当前交易、争抢、盘面变化，也不得误判为说明书

## 一点七、情节库特性命中硬闸

若当前章组明确依赖 `情节库 / 连续剧情库 / 场景母模板库 / 场景子模板库`，必须额外检查以下 6 项：

- 模板识别
- 骨架命中
- 收益命中
- 角色接口命中
- 语言特征命中
- 本地化完成度

默认执行口径：

- 说不清本轮到底用了哪类模板，默认 `模板识别未命中`
- 只借句式、口号、语录化用，不借骨架和收益，默认 `骨架未命中`
- 借了前半段铺法，没借后半段落袋，默认 `收益未命中`

裁决后果：

- `模板识别未命中` 或 `骨架未命中`：默认至少质量层失败
- `收益未命中`：默认至少 `F2`
- `本地化未完成`：默认不得判“情节库使用有效”

排误伤规则：

- 借了情节库，不等于必须把 6 项全部拉满
- 但至少要命中：`模板识别 + 骨架命中 + 收益命中`
- 若这三项都答不稳，不得再用“完整使用情节库”作为放行理由

## 二、解释句密度硬闸

以下情况默认至少 `F2`：

- 开头 `500` 字内解释句 `>= 2`
- 任意连续 `300` 字内解释句 `>= 2`
- 单章累计解释句 `>= 3`
- 章尾最后 `1-2` 句承担意义总结 / 局势翻译 / 作者代判功能

解释句命中后，必须继续判根因：
- `G1 信息抢跑`
- `G2 现场承载不足`
- `G3 提纲语言漏出`
- `G4 人物失声`

不得只写“解释腔偏多”。

### 解释句黑名单

以下句式默认视为高危解释句，不得按普通解释腔轻放：

- `他只知道一件事`
- `他这才明白`
- `今夜若……他会死`
- `最后留下来的，是……`
- `情报一个字没错`
- `就是现在`
- `就这一瞬，够了`
- `但这份……撑不了多久`
- 任意“先下结论，再补动作”的短句

规则：
- 开头 `500` 字内命中黑名单 `>= 2`，默认至少 `F2`
- 单章其他位置命中黑名单，也不得只做词面替换
- 修法必须回到动作 / 身体反应 / 即时念头
- 若黑名单句式集中在开头或章尾，必须继续判 `G1/G2/G3/G4`
- 若黑名单句式正好压在 `现实收益已落地` 的节点上，必须额外追问：是不是正文没把结果写实，所以作者才补解释句盖章

## 三、人物显影硬闸

以下情况默认至少 `F2`：

- 主角 / 核心反派 / 核心盟友 / 高价值角色首次强出场少于 `2` 项锚点
- 连续登场章节未复现 `1` 项稳定锚点
- 删除角色名后，主角组与核心配角台词仍明显混同
- 角色功能明确，但正文只给功能不给人脸、动作、说话区分
- 高价值角色只在设定里“好看 / 压人 / 值得看”，正文无可脑补细节

若角色卡已标记外形 / 动作 / 说话 / 气场锚点，正文却未连续兑现，默认至少 `F2`。

### 配角工具嘴失效边界

以下情况默认至少 `F1`：

- 某角色只在关键道具 / 关键信息 / 关键反转出现时补一句 `认出 / 确认 / 抬价 / 解释`，且前文无站位铺垫

以下情况默认上升为章节层问题：

- 某角色承担章尾翻面功能，却只靠一声失口或一句“怎么会是这个”顶住

## 四、高潮双段兑现硬闸

所有章节都必须拆成两段审查：

- `第一爆点`
- `高潮后续收益`

如果只有前者成立，而后者只剩：

- 对方变脸
- 黑幕更深
- 事情不简单
- 下一章再处理

默认至少 `F2`。

### 高潮后续收益认定

以下内容不得单独算作高潮后续收益：

- 黑幕更深
- 某物很重要
- 某人认得它
- 局势更复杂
- 事情不简单

合格的高潮后续收益至少命中以下之一：

- `命`
- `人`
- `名`
- `权`
- `物证`
- `站队`
- `规矩变化`
- `垫背者落地`

### 单章兑现 / 章组兑现 / 情节库兑现

默认规则：

- `首章 / 情报刷新章 / 当章主爆点章 / 番茄强爽章` 默认按 `单章兑现` 审
- `围猎章 / 布网章 / 跨章截令章 / 连续翻盘章` 允许按 `章组兑现` 审
- 若当前章组明确声明“本轮直接参考情节库/连续剧情库的跨章母模板”，允许按 `情节库兑现` 审，但不得跳过人工核验

#### 1. 单章兑现

以下情况默认必须在本章内拿到硬收益，不得后拖：

- 首章
- 正式情报首次刷新章
- 当章细纲已标注“主爆点章 / 翻盘章 / 收口章”
- 章尾前已把风险级别抬到“今夜必出事 / 当场必须拦截 / 不拦就直接死人或改规矩”

若只做到：

- 想明白了
- 布置完了
- 知道黑幕更深了
- 下一章去做

默认至少 `F2`。

#### 2. 章组兑现

允许某一章不把最大收益落完，但必须同时满足：

- 已明确属于同一 `2-3` 章兑现组
- 本章存在至少 `1` 口现实中间收益
- 本章章尾咬住的是“下一拍具体动作/具体目标/具体拦截口”，不是总结句
- 后续章仍在同一收益链上，不得中途换主驱动

可认定为“现实中间收益”的至少包括：

- 抓到人
- 截到物
- 锁定命令链
- 抢到站位
- 改了口
- 改了规矩的一半
- 拿到下一拍无法回避的口令 / 物证 / 活口

若本章只有：

- 判断更清楚
- 危险更大
- 线索更多
- 准备下一章行动

不算章组兑现中的合格中间收益，仍按不过线处理。

#### 3. 情节库兑现

若当前章组明确依赖“情节库 / 连续剧情库 / 场景母模板库”的跨章母模板推进，额外满足以下条件时，允许把 `2-3` 章视为一个完整兑现单元：

- 项目内能指出本轮采用的是哪一类母模板或连续剧情模板
- 该模板本身就是“布网 -> 压点 -> 截手 -> 落袋”式跨章结构，而非单章爆点结构
- 当前章没有把模板只用成讲解稿，仍然拿到现实中间收益
- 主线程能明确写出：`本章是兑现组的第几拍`、`本章已经兑现了哪一拍`、`最迟第几章必须落完整收益`

若只是“借了情节库”但：

- 本章没有现实中间收益
- 只剩说明书和判断句
- 或根本说不清套的是哪种跨章母模板

不得以“情节库本来就是章组兑现”为由降级门禁。

### 连续收益链豁免

以下情况，允许把“单一爆点触发的连续失血链”整体认定为合格的高潮后续收益，不得仅因其链条连续，就机械判成“只有一口收益”：

- 第一爆点落地后，连续引发 `控制权转移`
- 连续引发 `敌方失血`
- 连续引发 `我方抬位`
- 连续引发 `站位变化 / 队形变化 / 规矩变化`

适用前提：

- 每一拍都必须带来可见的现实变化
- 不得只是“认出来了 -> 继续解释 -> 继续确认”
- 至少命中上面现实变化中的 `2` 项

命中该豁免时：

- 审查必须先按“连续收益链”记录
- 不得仅因“不是两个明显分段的翻面块”就判 `F2`
- 若仍不过线，必须具体指出是“哪一拍没有造成现实变化”，不能笼统写“链条太完整”

### 后半章信息场失效边界

若单章后 `1/3` 主要在：

- 抬黑幕
- 抬线索价值
- 确认价值
- 变脸失声
- 解释局势

却没有新的现实翻面，默认至少 `F2`。

但若后 `1/3` 虽呈现连续处置链 / 连续翻面链，却同时满足以下任意 `2` 项以上：

- `文书 / 令牌 / 物证 / 兵器 / 车马` 的控制权发生转移
- 敌方被迫 `改动作 / 改站位 / 改判断 / 改口`
- 我方获得 `贴身控制 / 新站位 / 官面口径 / 明确保护`
- 现场其他人发生 `站队变化 / 递缰 / 横枪 / 改边 / 退缩`

则不得仅因“链条完整”或“处置连续”判为信息场；必须继续下钻判断：

- 是不是每一拍都带来现实变化
- 是不是只是同一结果被换说法重复报送
- 是不是读感问题主要来自句噪，而不是章节收益缺失

若此问题落在第 `1` 章章尾，默认从严处理。

### 章尾最后一拍复核

- `章尾前一拍` 合格，不等于 `最后一拍` 合格
- review 必须继续检查最后一句 / 最后一拍到底是不是新增现实结果
- 以下收口方式默认不得判章尾过线：
  - 压话句
  - 止语句
  - 情绪收束句
  - 预告句
  - 意义总结句
- 若最后一句只是 `别急 / 别碰 / 别先记 / 今夜还没完 / 事情不对 / 先这样` 这类压场句，而没有新结果，默认至少 `F2`

#### 情节库章组的排误伤规则

若当前章节命中以下全部条件，不得仅因“本章没把最大收益落完”就机械判 `F2`：

- 已被明确标记为 `章组兑现` 或 `情节库兑现`
- 本章已拿到至少 `1` 口现实中间收益
- 章尾停在下一拍具体拦截动作上
- 下一章仍承接同一收益链，不换主驱动

若缺任一项，仍按普通章节口径审。

### 首章双闸分离

首章必须分开裁决：

- `章尾是否合格`
- `高潮兑现是否过线`

规则：
- 章尾合格只代表“最后一拍停在了现场”，不代表整章已经过线
- 若首章满足以下任一情况，默认至少 `F2`：
  - 章尾合格，但高潮后续收益未成立
  - 章尾合格，但题面另一半未露头
  - 章尾合格，但外部盘面未改写
- 不得用“章尾已经有危险 / 发现 / 反扑”掩盖“整章只做到活下来 + 敌人来了 + 线索拿到了”
- 若首章章尾只是把刀送到门口、把物证送到眼前、把人送到脸前，而主角还没拿到任何现实收益，仍不得判 `F0`

## 五、事实层证据等级硬闸

### 1. 预案层

只允许写未来内容：

- `大纲/`
- `卷纲/`
- `细纲/`
- `追踪/总施工单_*.md`
- `追踪/执行计划_*.md`

### 2. 事实层

只允许写已落地内容：

- `追踪/时间线.md`
- `追踪/角色状态.md`
- `追踪/情报台账.md`
- `追踪/伏笔.md`
- `设定/角色/*.md` 中的“当前已落地事实”区块

硬规则：
- 审查追踪时，必须从正文抽事实，不得默认相信细纲或闸门
- 角色卡不得提前写未发生伤情、关系变化、身份变化
- 若事实层抄入预案层内容，直接按 `F5`

### 事实层措辞降级规则

事实层文件不得把“线索 / 强指向 / 高度疑似 / 死前吐词 / 临死失口”写满成“已确认 / 已主使 / 已执行 / 已坐实 / 第一批物证 / 活口口供”。

默认允许的表述：

- `获得指向 X 的线索`
- `高度疑似`
- `死前吐词 / 临死失口`
- `与 X 有关联迹象`
- `怀疑 X 方向仍有埋杀链`

默认禁止的表述：

- `确认 X 仍有埋杀局`
- `X 已派人灭口`
- `X 是灭口执行者`
- `第一批物证外露`（若正文仍只是线索）
- `活口口供`（若正文并无活口）

若事实层使用了禁止表述，而正文未完成同等级坐实，直接按 `F5`。

### 情报兑现时限硬闸

若核心卖点包含 `情报流 / 系统流 / 预知流 / 任务流`，必须同时满足：

- `追踪/情报台账.md` 写明 `刷新章 / 预计兑现章 / 实际兑现章 / 实际战果`
- 情报在当前章或后 `1` 章内转成现实收益
- 情报首次出现只用 `「」` 包住情报本体，而不是混在普通叙事里
- 情报首次出现时交代来源或触发机制，而不是凭空掉落

若连续出现“情报只负责提醒、解释、抬线索价值”，而没有兑现成现实战果，默认至少 `F3`。

若情报没有被 `「」` 包裹，或首次出现没有来源 / 触发机制，默认至少 `F2`。

补充：
- 不得把“下一章可直接反制 / 已经布好局 / 即将兑现”视为当前章的 `实际战果`
- 若 `追踪/情报台账.md` 缺失，而正文已出现第一条正式情报，默认至少 `F5`

## 六、反校验权限硬闸

`反校验_第XXX章.md` 必须至少检查：

1. 正文 vs 细纲
2. 正文 vs 写前闸门
3. 正文 vs 追踪主表
4. 正文 vs 角色卡

若存在以下任一情况，`反校验` 不得自报 `通过`：

- 细纲仍保留旧证据来源
- 写前闸门仍保留旧战果口径
- 追踪主表存在把线索写满成结论
- 角色卡“当前已落地事实”仍使用正文未坐实的判断

命中任一项时：

- 各分项可写 `待回修`
- `最终反校验结果` 只能写 `未通过`

禁止：

- 先写 `通过`，再等 review 打回来
- 把“尚可理解”当成“已对齐”

若任一组未通过，或旧口径未清而自报通过，默认至少 `F5`。

---

## 七、首章额外高压标准

第 `1` 章在通用协议上再加：

- 前 `300` 字内必须亮出主卖点死局
- 只活命不够，必须显出题面另一半的第一口势
- 后 `1/3` 不得主要靠线索收纳和意义总结撑住
- 开头 `500` 字内黑名单解释句不得超过 `1`
- 章尾前必须至少出现一口“敌方被迫改动作 / 改站位 / 改判断”的外部现实变化

若首章只做到：

- 主角活下来了
- 敌人来了
- 线索拿到了

但没有更硬的外部盘面改写，默认仍至少 `F2`。

---

## 八、章尾单独否决权

以下章尾默认不过线：

- `预告型`
- `总结型`
- `信息型`
- `平收型`
- `污染型`

合格章尾只认现场结果，不认口头承诺。

至少命中以下之一，才可判章尾过线：

- 新危险已经咬上来
- 新反扑已经启动
- 新人物 / 新敌意 / 新物证当场落地
- 某人失口、露底、翻面
- 某条规矩下一秒就要被打破
- 现场盘面已被推进到新的不可逆局势

`不是盖章句` 不等于 `章尾合格`。只要落点仍是预告、总结、平收、解释，照样判不过线。

若章尾最后 `1-2` 句主要功能是：

- 总结主角身份变化
- 总结本章胜负变化
- 总结局势已经变化
- 替读者解释“这一下意味着什么”

即使前面的动作、失态、压制、物证都成立，仍默认至少 `F2`。

只要单章章尾不过线，即使整章主体尚可，该章也默认 `不可并入主正文`。

## 九、失败码落点

- `F0`：过线
- `F1`：句子层失败
- `F2`：章节层失败
- `F3`：机制层失败
- `F4`：污染失败
- `F5`：流程 / 追踪失败

凡命中以下任一项，`F5` 不得留空：

- 事实层把线索口径写满成结论口径
- `反校验_第XXX章.md` 在旧口径未清时自报 `通过`
- 追踪主表 / 角色卡 / 时间线 / 闸门 / 细纲 保留双版本或旧证据来源
- 项目文件出现 `活口口供 / 已主使 / 已执行 / 第一批物证` 等超前口径，而正文未同级坐实
## 脚本结果消费硬规则

- 若项目存在 `scripts/scene_lint.py`，审查前默认优先运行 `scripts/scene_lint.py --json`
- 主线程必须显式区分三层：
  - `脚本 error`：可直接进入门禁层
  - `脚本 warn`：仅作预警，不自动进门禁层
  - `人工复看确认`：主线程结合正文证据后，才可升级为正式问题
- 默认裁决：
  - `summary.errors > 0`：不得忽略
  - 仅 `字数.warn`：不得单独判 `F2`
  - `章尾.warn / 解释.warn / 结构.warn / 情报.warn`：必须回看对应正文，再决定是否入单
- 不得只因脚本有 `warn` 就给章节判不过线
- 也不得只因脚本退出码为 `0`，就跳过章尾、解释腔、情报兑现等人工核查

## 脚本字数预警复看口径

若章节只剩 `字数.warn / 字数.error`，主线程仍必须额外判断“新增内容是否是真正文，不是白句顶字数”。

优先认可的补写类型：

- 新微事件
- 新一轮对话博弈
- 新群体反应 / 站位变化
- 新的现场阻断或代价
- 兑现后的现实余波

默认不认可的补写类型：

- 解释主角已经想明白的判断
- 重复本章已有结论
- 总结局势、翻译意义、作者盖章
- 单独补情绪判断句而不落动作
- 只为跨过字数线补 `1-2` 句空泛抒压句

审查动作：

- 若新增文字删掉后，不影响当前章的冲突、兑现、人物或盘面变化，默认记为“无效补字”
- 若新增文字主要由白话判断句、总结句、抽象狠话构成，即使字数过线，也不得直接放行为“正文厚度已补足”
- 若新增文字通过对话补字，必须检查这轮对话是否真的改掉了 `口径 / 次序 / 站位 / 归属 / 执行权` 中至少一项；若没有，默认仍是水字
