# story-review 回归基线

> 用途：在修改 `story-review` 的门禁规则、报告模板、脚本结果消费口径、固定验收清单或失效复盘单流程后，提供一套最小但必须执行的标准化回归样本，避免“改一处、坏一片”。

---

## 一、什么时候必须跑这份基线

命中以下任一情况，修改后必须至少跑一轮最小回归集：

- 改了 `scene_lint --json` 相关消费规则
- 改了 `F0-F5` 映射口径
- 改了 `固定验收清单` 骨架
- 改了 `失效原因复盘单` 触发条件或骨架
- 改了 `5行施工单` 的强制字段
- 改了 `章尾硬规则`、`并入裁决`、`观察项` 与 `正式问题` 的分流规则

若只是改注释、路径说明、提示文案，且不影响上述行为，可不强制跑全套，但建议至少抽跑 1 个样本自检。

若项目根存在 `scripts/story_review_regression.py`，优先直接运行：

```bash
python3 scripts/story_review_regression.py
```

再根据脚本汇总结果判断是否需要补人工复看与额外落盘核查。

若项目中不存在该脚本，才退回手工按本文件逐组跑样本。

---

## 二、最小回归集（必须覆盖）

### 1. 人工覆核样本（原 F0 干净样本）

- 样本：`测试/skill回归_现代悬疑调查`
- 预期：
  - `scene_lint --json` 为 `errors=0, warnings=0`
  - 真实 `full agents` 不应在人物层、一致性层、纯净度层硬造问题
  - 主线程必须额外人工覆核 `第003章` 章尾
  - 允许得出：`可并入主正文，但不再归类为无争议 F0 干净样本`
  - 不得生成 `失效原因复盘单`

补充说明：
- 该样本旧人工基线曾按“绝对 F0 干净样本”使用
- 真实 `full agents` 联调后，主线程人工覆核确认：`第003章` 章尾更接近“下一步行动预报”
- 因此它现在更适合验证：
  - 主线程会不会在 `0 error / 0 warn` 时乱造人物或一致性问题
  - 主线程会不会对章尾分歧做人工拍板
  - 而不是再把它当作“任何视角都必须直接 F0”的死样本

### 2. F0 + 轻预警样本

- 样本：`测试/skill回归_江湖帮派经营`
- 预期：
  - `scene_lint --json` 仅 1 条 `字数.warn`
  - `story-review full` 不得仅因该项判 `F2`
  - 该项应进入 `脚本预警 / 观察项`
  - 不得生成 `失效原因复盘单`

### 3. F0 + 多预警样本

- 样本：`测试/skill回归_创业商战`
- 预期：
  - `scene_lint --json` 为多条 `字数.warn`
  - `story-review full` 不得因为 `warn` 数量多就自动升格为 `F2`
  - 多条 `warn` 仍应统一落在 `脚本预警 / 观察项`
  - 不得生成 `失效原因复盘单`

### 4. F2 真门禁样本

- 样本：`测试/开局流放，我每天解锁一个情报`
- 预期：
  - `scene_lint --json` 命中 `error`
  - `story-review full` 必须正确进入门禁层
  - 必须生成：
    - `总施工单`
    - `固定验收清单`
    - `失效原因复盘单`
  - 必须区分：
    - 真正拦并入的 `error`
    - 只能算观察项的 `warn`

补充说明：
- 真实 `full agents` 联调后，当前稳定口径不是“脚本 2 条 `error` 全都无争议成立”，而是：
  - `第018章`：人工确认成立，为真实章尾硬闸失败
  - `第027章`：存在人工覆核分歧，不宜再写死为“脚本命中 = 最终失败”
- 因此本样本现在主要用于验证：
  - `scene_lint error` 会不会正确进入门禁候选
  - 主线程会不会把 `warn` 继续压在观察项
  - 主线程会不会在 `error` 命中后保留人工覆核权，而不是直接照脚本抄结论

## 三、扩展回归集（建议覆盖）

以下样本不属于“每次都必须跑”的最小集，但在改题材专项 checks、跨场景规则或多章连续场面判断时，建议加跑：

- `测试/skill回归_古代断案追索`
- `测试/skill回归_异题材审美牵引`
- `测试/skill回归_跨场景过桥`
- `测试/skill回归_母场景避重复`

用途：
- 验证不同题材不会被同一套口径误伤
- 验证边荒流放题材的跨场景、母场景轮换和情报兑现检查没有被新规则误判

---

## 四、验收要点

跑完回归后，至少核这四件事：

1. `warn` 是否被误写进 `失败码`
2. `仅 warn` 的样本是否被错误生成 `失效原因复盘单`
3. `0 error / 0 warn` 的样本是否在人物层/一致性层被强行判出正式问题
4. `有 error` 的样本是否被错误放行为 `F0`
5. `scene_lint error` 是否被误当成最终裁决，而不是“门禁候选 + 人工覆核”
6. 子代理与脚本冲突时，主线程是否明确写出了 `人工覆核裁决`

---

## 五、推荐落盘方式

若本轮是“改 skill 后的回归验证”，建议统一命名为：

- `追踪/总施工单_*_story-review-full回归测试.md`
- `追踪/固定验收清单_*_story-review-full回归测试.md`
- `追踪/失效原因复盘单_*_story-review-full回归测试.md`

这样能与正式审查工单区分，避免覆盖真实施工记录。
