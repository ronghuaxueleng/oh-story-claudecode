# story-review 回归基线

> 用途：在修改 `story-review` 的门禁规则、报告模板、脚本结果消费口径、固定验收清单或失效复盘单流程后，提供一套最小但必须执行的标准化回归样本，避免“改一处、坏一片”。

---

## 一、什么时候必须跑这份基线

命中以下任一情况，修改后必须至少跑一轮最小回归集：

- 改了 `scene_lint --json` 相关消费规则
- 改了 `F0-F5` 映射口径
- 改了 `固定验收清单` 骨架
- 改了 `失效原因复盘单` 触发条件或骨架
- 改了 `5行施工单` 的强制字段
- 改了 `章尾硬规则`、`并入裁决`、`观察项` 与 `正式问题` 的分流规则

若只是改注释、路径说明、提示文案，且不影响上述行为，可不强制跑全套，但建议至少抽跑 1 个样本自检。

若项目根存在 `scripts/story_review_regression.py`，优先直接运行：

```bash
python3 scripts/story_review_regression.py
```

再根据脚本汇总结果判断是否需要补人工复看与额外落盘核查。

若项目中不存在该脚本，才退回手工按本文件逐组跑样本。

## 二、当前真实最小回归集（必须覆盖）

> 当前项目若存在 `scripts/story_review_regression.py`，最小回归集以脚本内真实样本为准，不再依赖旧的 `测试/skill回归_*` 占位目录。

### 1. `A组 / 食堂试勺记`

- 用途：`解释腔命中样本`
- 预期：
  - `scene_lint --json` 稳定为 `errors=1, warnings=1`
  - 至少命中：
    - `解释.error=1`
    - `字数.warn=1`
  - 用来验证：主线程在“模板验证态”里，能否稳定抓到非厚度主问题
  - 不得把这类样本误简化成“只是字数问题”

### 2. `A组 / 裂雾供氧站`

- 用途：`章尾硬闸样本`
- 预期：
  - `scene_lint --json` 稳定为 `errors=1, warnings=1`
  - 至少命中：
    - `章尾.error=1`
    - `字数.warn=1`
  - 用来验证：`章尾硬规则` 是否仍稳定生效
  - 不得因同时有 `字数.warn` 就把章尾问题冲淡成观察项

### 3. `B组 / 黑板竞标会`

- 用途：`情报流误触样本`
- 预期：
  - `scene_lint --json` 稳定为 `errors=5, warnings=1`
  - 至少命中：
    - `情报.error=4`
    - `章尾.error=1`
    - `字数.warn=1`
  - `detect_key_character_promotion.py` 当前允许命中 `missing_cards=1`
  - 用来验证：
    - 信息场载体是否会被误判成正式情报流
    - `情报` 类问题与人物缺卡问题能否同时暴露

### 4. `C组 / 冷库截标会`

- 用途：`三条固定检查条专项正例`
- 预期：
  - `scene_lint --json` 稳定为 `errors=3, warnings=0`
  - 至少命中：
    - `字数.error=2`
    - `情报.error=1`
  - 必须存在专项工单：
    - `追踪/总施工单_第001-002章_专项复审_三条固定检查条.md`
  - 用来验证：
    - `收益档位高判`
    - `非情报流误触`
    - `本组第二章短兑现链`
    已进入真实回归链

## 三、扩展回归集（建议覆盖）

### 1. `B组 / 借婚封心局`

- 用途：`本组第二章短兑现链样本`
- 预期：
  - `scene_lint --json` 稳定命中 `字数.error=2`
  - 重点不是厚度本身，而是保留“第二章被压缩成短兑现链”的真实负例

### 2. `B组 / 潮口封单局`

- 用途：`双章尾风险样本`
- 预期：
  - `scene_lint --json` 稳定命中：
    - `章尾.error=2`
    - `字数.error=2`
  - 用来验证双章尾风险不会在连续样本中被漏扫

### 3. `B组 / 哭庙换命灯`

- 用途：`玄幻异题材纳管样本`
- 预期：
  - `scene_lint --json` 稳定命中 `字数.error=2`
  - 用来验证玄幻脑洞样本是否仍在同一回归链内

### 4. `B组 / 锁门替签录`

- 用途：`解释 + 字数复合样本`
- 预期：
  - `scene_lint --json` 稳定命中：
    - `解释.error=1`
    - `字数.error=2`
  - 用来验证复合问题不会被单一标签吞掉

## 四、验收要点

跑完回归后，至少核这四件事：

1. `warn` 是否被误写进 `失败码`
2. `仅 warn` 的样本是否被错误生成 `失效原因复盘单`
3. `0 error / 0 warn` 的样本是否在人物层/一致性层被强行判出正式问题
4. `有 error` 的样本是否被错误放行为 `F0`
5. `scene_lint error` 是否被误当成最终裁决，而不是“门禁候选 + 人工覆核”
6. 子代理与脚本冲突时，主线程是否明确写出了 `人工覆核裁决`
7. `scene_lint.py` 是否按“逐章文件路径”调用，而不是把 `正文/` 目录直接传给脚本
8. 若样本目录里混有嵌套子样本副本，主线程是否只审主样本根路径，没有把子副本混读进证据链

## 六、当前阶段经验口径

在最近多轮 cold-start / review 联调后，主线程做 `story-review` 时应优先记住：

1. 当前最容易被误判的，不再是“模板有没有挂名”
- 多题材样本已经证明：模板字段、资料调用、第一爆点、第二口收益、两口外部反应，很多时候已经能首写落盘
- 当前更常见的残留，已经收缩到：
  - `收益档位高判`
  - `非情报流误触`
  - `本组第二章短兑现链`
- `厚度不足` 继续记录，但在“模板验证态”里默认降为次级观察项

2. 当前最需要 review 继续分辨的，是“骨架对但肉不够”
- 不能因为 `解释.error` 消失了，就误判成正文已经自然成熟
- 也不能因为 `字数.warn / error` 还在，就简单写成“只是字数问题”
- 必须继续判断：正文是不是只有骨架推进，没有足够多的动作拍、争口拍、外部改线拍

3. 若资料利用率记录完整，review 不应机械判“资料挂名”
- 若 `opening-design / genre-writing-formulas / hooks-chapter / dialogue-mastery / character-design-methods` 的落点、效果、正文对应位都写清了
- 即使正文仍薄，也应优先判为“资料已把骨架拉对，但场面肉量不足”
- 不应误写成“资料未使用”或“方法论挂名”

4. 当前固定检查条应默认前置
- 只要用户任务属于：
  - `模板使用率复审`
  - `情节是否正确使用`
  - `skill 冷启动验证`
  - `story-review full agents 全量检测`
- 主线程就应默认优先输出三条裁决：
  - `收益档位高判`
  - `非情报流误触`
  - `本组第二章短兑现链`
- 不应再优先从厚度出主结论

5. 当前还剩的不是脚本债，而是专项样本债
- 还缺：
  - 接近 `F0` 的干净正向样本
- 纯 `收益档位高判` 样本
- 纯 `非情报流误触` 样本
- 纯 `本组第二章短兑现链` 样本
- 后续补样本时，应优先追求“单一问题更纯”，避免被 `字数`、`章尾`、`情报` 多问题混叠抢掉主结论

8. 专项基线现在要分成两类看
- `脚本可直接稳定命中`：
  - `skill回归_收益高判`
  - `skill回归_双版本并存`
  - `skill回归_非情报流误触`
- `主线程 / agents 人工分流优先`：
  - `skill回归_本组第二章短兑现链`
  - `skill回归_本组第二章最后半拍回弹不足`
- 这两类样本的使用口径不能混：
  - 前一类适合验证“脚本新增规则是否生效”
  - 后一类适合验证“脚本候选出来后，主线程 / agents 会不会判错主结论”

9. `本组第二章短兑现链` 与 `最后半拍回弹不足` 必须拆开验
- `本组第二章短兑现链`
  - 第001章先证明模板已落地
  - 第002章主体只剩四拍主链与少量反馈
  - 即使先命中 `字数.error`，主线程也应优先判“短兑现链”
- `本组第二章最后半拍回弹不足`
  - 第002章中前段必须先落成现实收益
  - 停签 / 落签 / 改顺位 / 外部改站位 / 器物反馈 至少已成立一轮
  - 若最后 `1-3` 句只是偏顺收束，主线程应优先判“最后半拍回弹不足”
  - 当前这类样本即使仍先命中 `字数.error`，也不得直接回收成“短兑现链”或“整章兑现不足”

10. `story-review full agents` 的新增最小联测集
- 除 `A / B / C` 三组制度验收样本外，后续凡改动以下任一口径，至少应补跑这 5 类专项基线的人工联测：
  - `收益档位高判`
  - `双版本并存`
  - `非情报流误触`
  - `本组第二章短兑现链`
  - `本组第二章最后半拍回弹不足`
- 联测目标不是要求 agents 全部“脚本同判”，而是要求：
  - 能识别脚本主候选
  - 能写出人工分流结论
  - 不把 `warn / error` 机械等价成最终失败码
- 真实执行补充硬规则：
  - 跑 `scene_lint.py` 时必须逐章传文件路径，不得把 `正文/` 目录直接传给脚本
  - 若样本目录内部混有嵌套子样本副本，必须先锁定唯一主样本根路径，只按该根路径下文件取证
- 建议同时维护专项联测台账：
  - `测试/story-review_专项基线_agents联测台账.md`
  - 用于固定记录：
    - 每个专项样本的脚本候选
    - 目标主结论
    - 允许噪音
    - 禁止误判方向

6. `非情报流误触` 不能只看 `scene_lint` 标签
- 真实冷启动样本已证明：这类正文很容易先被脚本打成 `情报.error`
- 但主线程不得因此直接下结论“这是正式情报流正文”
- 必须继续做语义复核：
  - 当前承载物是在“播报 / 旧码 / 错单 / 挂签 / 复核口 / 补条广播”里推动现场冲突
  - 还是在做“独立情报发布 / 情报刷新 / 情报兑现”
- 若前者成立，最终应优先记为：
  - `非情报流误触候选`
- 不得因为脚本先命中 `情报.error`，就把专项样本误判成“情报流格式缺失”后直接收口

7. 裁决顺序不能替代最低可判型场面线
- 新一轮冷启动样本 `漏码封航局 / 借牌占头录` 已进一步证明：
  - 新的裁决顺序可以纠偏
  - 但不能替代“正文至少要像一个可判型场面”的最低厚度要求
- 若专项样本太薄，脚本与人工注意力仍会先被以下噪音牵走：
  - `字数.error`
  - `章尾.error`
  - 少量 `解释.error`
- 因此后续做专项负例时，不能只追求“问题做纯”，还要守住最低场面线：
  - `非情报流误触`：至少补足少量第三方动作、站位改线、器物反馈
  - `本组第二章短兑现链`：第001章必须足够证明模板已落地，本组第二章可以短，但不能短到只剩残稿感

---

## 五、推荐落盘方式

若本轮是“改 skill 后的回归验证”，建议统一命名为：

- `追踪/总施工单_*_story-review-full回归测试.md`
- `追踪/固定验收清单_*_story-review-full回归测试.md`
- `追踪/失效原因复盘单_*_story-review-full回归测试.md`

这样能与正式审查工单区分，避免覆盖真实施工记录。
