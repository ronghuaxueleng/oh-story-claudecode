# `workflow-daily.md`：日更续写怎么走

这份汇总“日更续写”这条线的完整走法。SKILL.md 路由到这里后，可按下面推进。

> **日更模式准备步骤**：每章正式开写前，都先走 2 步：先看眼下情况，再看这章想写成什么劲儿。这两步都塞在 Step 2 的逐章循环里。日更模式一般不必回头翻对标拆开的子目录，因为题材路数已经写进 `设定/题材定位.md`，对标部分也已经进入第 2-3 阶段的卷纲和细纲。情绪和技法这块，用 Step 2.2 的意图确认兜住。更完整的准备走法见 `SKILL.md` 第 4 阶段正文写作辅助。

---

## 一般什么时候用这份

- 项目已有 `正文/` 和 `追踪/` 目录
- 用户说"日更""续写""继续写"
- 一轮通常写 2-3 章（4000-9000 字）

## 单文件串行模式补丁

当用户明确要求：

- `每次只写一个文件`
- `冷启动测试`
- `不要停`
- `一直往下推`
- `继续`

而且当前任务还是同一批连续生产时，日更这条线就切到 `单文件串行模式`：

1. 每个执行步只允许新增或改写 `1` 个项目文件
2. 一个文件做完后，若不存在阻塞信息，别问“是否继续”，直接进下一个执行步
3. `单文件` 卡的是单次写进文件的动作，不是单个用户回合；同一用户回合内允许连续执行多个执行步
4. 下一个执行步一般按既定生产顺序顺延：
   - 缺细纲先补细纲
   - 有细纲先写正文
   - 正文写完先做本章必要追踪同步，再进入下一个文件
5. 只有出现以下情况才停：
   - 章节号冲突
   - 细纲缺失且无法安全补建
   - 追踪/正文真实续写点冲突
   - 写到该做阶段审查的时候，先执行 `story-review full`
   - 用户新指令改变既有大纲、卷任务或生产范围
6. 如果当前文件已经做完，模型却准备收束回复，那就算流程还没走完；先看“下一个文件是不是已经能确定”，能确定就继续推，别把“这个文件做完了”误当成“这一轮结束了”

## 人工接管补丁

日更 / 冷启动过程中，`人工接管模式` 是条件一到就直接切进去的自动返修态，不是问完用户再选的可选项。只要撞上 `core-gates.md` 的 `H1` 条件，就直接标记成 `H1-A` 或 `H1-B`，然后按对应说法往下做，别停下来问“是否人工介入”。

### `H1-A`：纯 skill 基准验证

- 这一档是拿来测 skill 自然产出的
- 这里得卡住：不靠人工重写来掩盖结构问题
- 允许接管的范围仅限：
  - 清污染残片
  - 修文件截断
  - 清明显非故事语料
  - 修运行时异常导致的不可读段
- 正文还没过时，只写失败记录、失败码、触发规则和原样样本位置，不修正文

### `H1-B`：生产模式

- 这一档是把书真正写到能用
- 这里就按这个来：主线程直接接手细纲、正文和追踪
- 但接管后得：
  - 明写“当前为人工接管模式”
  - 写明接管原因和文件范围
  - 写后继续走闸门、写后检查、追踪同步

若用户没特别要求“纯 skill 原样测试”，一般就用 `H1-B`；别把生产模式下的 `H1` 又丢回给用户选。

## Step 1：快速上下文加载

### Step 1.0：确认真实续写点

进入日更前，先看清“当前真实续写点”：

1. 读取 `追踪/上下文.md` 的最后完成章节或当前位置
2. 扫描 `正文/` 目录中编号最大的章节
3. 检查 `追踪/时间线.md`、`追踪/角色状态.md` 是否仍停留在更早章节，或错误超前到尚未真正落地的远期预案
4. 如果出现以下任一冲突，先修追踪，再写正文：
   - `追踪/` 仍写在旧章节，但正文已经大修到更新章节
   - `追踪/` 默认续写点超前到未真正落地的后续卷/后续章
   - `时间线/角色状态/上下文` 三者彼此说法不一致
   - 细纲把旧预案当成已发生事实
   - `角色状态/情报记录/伏笔` 把“计划补强、准备兑现、预期会成立”的内容写成已完成
   - `时间线.md` 或其他追踪文件里同一事件存在双版本并存、旧版未废止

**这里就按这个认**：
- 以“已存在正文 + 已同步追踪”的最近一段为真实续写点
- 旧卷纲、旧预案、远期细纲不能直接当成真实已发生内容
- 续写点没看清前，别直接续写正文

**可选：使用 story-explorer 子代理批量加载上下文**。如果项目已部署 `story-explorer`（检查 `.codex/agents/story-explorer.md` 是否存在），可用 `spawn_agent(agent_type="default", task="读取项目目录：{dir}\n查询类型：context_load\n查询参数：准备写第 {N} 章", context=".codex/agents/story-explorer.md 的约束")` 执行 `context_load` 查询，一次获取全部写作上下文。子代理返回后直接使用其 `results`，跳过下方手动加载步骤。如果子代理不可用或返回不完整，回退到下方手动加载。

手动加载就按这个来（默认方式）：

| 序号 | 文件 | 用途 | 如果不存在 |
|------|------|------|-----------|
| 1 | `追踪/上下文.md` | 上次写作进度摘要 + 当前真实续写点 | 从 `追踪/伏笔.md` + `追踪/时间线.md` 重建 |
| 2 | `设定/爽点怎么落.md` | 小说类型 + 这一段主要靠什么爽 + 收益分配 | **得先补建**，不能跳过 |
| 3 | `追踪/伏笔.md` | 待回收伏笔清单 | 跳过 |
| 4 | `追踪/时间线.md` | 当前事件时序 | **得先补建**，不能跳过 |
| 5 | `大纲/细纲_第{N}章.md` | 本章写作计划 | **得先补建**，不能跳过 |

**按需加载角色文件**：从细纲里挑出本章会用到的角色名，按需加载 `设定/角色/{角色名}.md`。如果细纲里没列角色，就跳过。

**按需加载这些写法规则**：只有写作时真要用到这类约束（比如期待感、爽点、信息差），再去加载 `references/genre-writing-formulas.md`。一般别默认就加载，免得白白吞下 1500+ 行文件浪费 token。

### 已写内容太多时怎么压着记

当项目超过 30 章时，`追踪/上下文.md` 中的已写内容摘要就按三段来写，压缩早期章节、保留近期细节：

| 摘要层 | 粒度 | 格式 | 维护时机 |
|------|------|------|----------|
| **近5章详记** | 近 5 章逐章摘要 | 保留现有格式（每章一段：事件+状态变化+伏笔） | 每章写完后更新 |
| **十章概要** | 每 10 章一段概要 | `章节范围 | 已写最主要的事 | 角色状态变化` | 每写完 10 章生成一段 |
| **卷级总览** | 每卷一句话总结 | `卷名 | 已写主线进展 | 关键转折` | 每卷结束时生成 |

**示例（50 章项目）**：
```markdown
## 近5章详记（第 46-50 章）
第 46 章：主角进入秘境，发现上古遗迹...
第 47 章：遭遇守护兽，险胜后获得传承...
...

## 十章概要
第 1-10 章 | 主角觉醒金手指，进入修炼学院 | 从废柴变为内门弟子
第 11-20 章 | 学院大比，击败天才对手 | 获得长老关注，实力突破
第 21-30 章 | 离开学院，探索秘境 | 发现身世线索，结交盟友
第 31-40 章 | 卷入势力争斗，暴露身份 | 被追杀，实力再突破
第 41-50 章 | 反杀追兵，进入新地图 | 确认血脉传承，进入上界

## 卷级总览
第一卷·觉醒 | 废柴逆袭成天才 | 金手指觉醒，身世初现端倪
```

> **首次日更 fallback**：如果 `追踪/` 目录下的文件全是空的，或者压根没有（刚完成第 3 阶段但还没日更过），就额外加载 `大纲/卷纲_当前卷.md` 和最新一章正文，把上下文重新搭起来。

> **确定下一章编号 N**：从 `追踪/上下文.md` 的"最后完成章节"这一项读取。如果文件不存在，扫描 `正文/` 目录中编号最大的章节 +1。

> **续写点冲突先处理**：如果 `追踪/上下文.md` 记录的最后完成章节，与 `正文/` 最大章节号、`时间线.md` 当前时间切点、`角色状态.md` 默认续写点冲突，不要猜测，不要硬续。先把追踪同步到真实正文，再确定下一章编号。

确定本轮写作范围后直接进 Step 2，不做“是否继续”式确认。K 默认 2-3 章；若进入 `单文件串行模式`，就把 `K` 理解成“这一批连续要推进到哪”，不是“单步写多个文件”。这时得改成“每步 1 文件、步间不停、同回合连续推进”。只有章节号冲突、细纲缺失、请求范围超过已有细纲、用户要求会改大纲/追踪，或存在其他会导致写错的阻塞信息时才暂停确认。

## Step 2：串行批量写作

一次加载多章细纲，但**得在主会话内串行逐章写作**：别把多章同时交给多个子代理并发写。长篇章节依赖上一章正文和追踪文件，并发会导致上下文断裂、追踪覆盖和标题去重失效。

**批量续写就按这一轮一直往下推**：进了本 Step 以后，"继续"/"续写"/"日更"都只表示继续当前这轮日更批量流程。别把这些词理解成跳过 Step 2.2 / Step 2.3，直接去续正文；也别每章之间都重复问要不要继续，除非用户明确要求逐章确认，或者中间出了阻塞。

1. **读取细纲**：加载本次要写的 2-3 章细纲
2. **逐章往下写**（下面这些步都放在每章循环里）：
  - 读细纲 → 按需加载角色设定
  - **2.0 先把当前情况看清**：进本章前，先把当前项目情况看清：
     - `S0-S7` 现在走到哪一步
     - 上一章最终失败码是否为 `F0`
     - 若上一章不是 `F0`，立即中断本章，禁止继续顺写
     - `追踪/时间线.md`、`追踪/角色状态.md` 得存在；任一缺失，立即停在本步，先补文件
     - 若作品是 `情报流 / 系统流 / 预知流 / 任务流`，先检查 `追踪/情报记录.md` 是否存在；不存在直接停在本步，先补文件
     - 若本章编号 `>= 3`，先执行最近 `2-3` 章的 `章节组闸门`；未执行或没过，禁止写正文
   - **2.1 标题预检**：读取 `references/chapter-title-rules.md`，扫描既有章节标题；如本章标题同名、明显重复、过于概括、过于剧透、只有物件说明没有冲突感，或出现“不是……而是……”这种旁白先替读者把轻重说出来的句子，先按本章第一爆点 / 最猛的收获 / 章尾钩子重命名，并同步细纲标题与正文文件名
  - **2.2 先把这章要用的东西过一遍**：每章开始前，先看这些来源在本轮里是不是已经读过，或者刚更新过：本章细纲、上一章正文（或上一章刚写入的正文）、`追踪/上下文.md`、`追踪/伏笔.md`、`追踪/时间线.md`、`追踪/角色状态.md`；如果是情报流，还得看 `追踪/情报记录.md` 的来源；如果涉及角色，还得看对应 `设定/角色/{角色名}.md` 的来源。这里说的“已加载”，只算本轮里真的读过或更新过的文件，别拿没标明来源的聊天记忆顶替。角色最新状态先从 `追踪/角色状态.md` 里挑，待回收或待推进的伏笔先从 `追踪/伏笔.md` 里挑；缺任何一份追踪文件，都先补文件，别直接写正文
     - 若本章涉及连续登场的重要角色，除角色卡外，还得回看其最近一次正文出场，确认外形记忆点、动作记忆点、说话记忆点没有漂移
  - **2.3 先用一句话说清这章要写成什么感觉**：从细纲里的「这章想让读者吃到什么情绪」这一项出发，结合刚才过过的内容，用一句话说清本章写作意图（情绪 + 节奏）
  - **2.3.1 先对一遍这章该怎么爽**：用 `设定/爽点怎么落.md` 对一遍本章，明确：
     - 本章这次主要在兑现哪一种爽点
     - 本章不该滑成哪类写法
     - 本章收益应该落在命 / 人 / 名 / 权 / 物证 / 人心变化 / 规矩变化中的哪几类
     - 若书名主打涉及配额 / 经营 / 分配 / 执牌 / 掌规则，本章得明确“结果化兑现口”是什么；若只能写成“更接近”，就停下，先别写正文
  - **2.3.1.1 情报条目先看一眼**：如果作品是情报流：
     - 读取 `追踪/情报记录.md`
     - 确认本章已预填 `刷新章 / 预计兑现章 / 预计收益类型`
     - 确认本章关键翻盘不是靠“临场观察”替代情报
     - 任一项不成立，先补情报记录或回细纲，正文先别写
  - **2.3.1.2 章节组这道坎先答清**：如果本章编号 `>= 3`，正式写正文前先把最近 `2-3` 章的情况答清：
     - 最近三章主收益走法是否重复
     - 最近三章翻盘走法是否重复
     - 上一章章尾是否提前做空本章主任务
     - 书名主打卖点是否真的往前推了一步
     只要有一项答不稳，按 `F3` 回退 `章组控制 / 细纲`
   - **2.3.2 先把重场面单拎出来**：正式写正文前，先从本章细纲里单独拎出 `本章关键场面卡`，最少答清：
     - 本章最重的 `1-2` 个场面到底是哪一场
     - 每场的先手方、截堵点/爆点、动作拍序、险点、翻盘结果是否已写清
     - 若本章是追逐 / 截堵 / 反绕 / 抢签 / 战斗 / 谈判翻桌 / 势力撞局 / 灾异逃生这类高动作章节，正文得按 `场面卡 -> 正文` 顺序落，别直接拿章节摘要硬扩写
     - 若 `场面卡` 仍只有“对手来抢 / 主角反打 / 最后翻车”这类收口话，正文先停，先回补动作拍序
  - **2.3.2.1 事件拍序先过一遍**：正式写正文前，单独把 `本章事件拍序` 过一遍：
     - 本章最少有 `3-5` 拍推进，别只有“目标 -> 执行 -> 结果”
     - 里面得有 `1` 拍是受阻或被压回来
     - 里面得有 `1` 拍是变招、舍取或代价落地
     - 里面得有 `1` 拍让现场真的改了什么，而不是只抬信息
     - 若写不出以上四项，正文先停，先回细纲
     - 若本章还是冷启动首章组或边界题材回归样本，且事件拍序只能答出单拍推进，额外读取 `references/chain-reaction-scene-thickening.md`，先补出 `1-2` 条 `起手动作 -> 第一被动改线 -> 第二被动改线` 连锁，再进入正文
  - **2.3.2.2 情绪回摆先看清**：正式写正文前，单独把 `本章情绪回摆` 看清：
     - 本章最少有 `2` 次情绪摆动
     - 情绪摆动得绑在事件上，别只写心理句
     - 每次摆动最少要落到 `动作停顿 / 把话吞回去 / 舍取 / 失态 / 代价 / 谁往前谁往后变了` 之一
     - 若整章情绪只有一条直线，如一直查、一直稳、一直强、一直冷，正文先停，先回细纲
  - **2.3.2.2.1 非预期障碍先落实**：正式写正文前，单独把 `本章非预期障碍` 落实：
     - 最少得有 `1` 口障碍，不是主角已知旧阻力的平推复现
     - 障碍来源得写清：第三方突然插手 / 世界环境改压 / 对手换招 / 协作方失手 / 时间窗口卡死
     - 若障碍只能写成“更难了 / 对手更狠了 / 局势更乱了”，就当还没落到具体上，正文先别写
  - **2.3.2.2.2 受阻过桥先答出来**：正式写正文前，单独把 `本章灾后反应 -> 两难 -> 决断过桥` 答出来：
     - 最少得有 `1` 次受阻后，不是直接跳结果，而是出现短反应、舍取或改决断
     - 这次过桥得答出来：主角或关键角色到底舍了哪一样、改了哪一步
     - 若答不出，说明情绪只是概念，没有钉在事件里，正文先别写
  - **2.3.2.3 先把对标里的对白拆开**：正式写正文前，先把本章 `对白对标拆解结果` 做完：
     - 若项目已经启用对标流程，先检查 `对标/对标采样执行单.md` 是否存在；不存在先创建，后面先别继续
     - 本轮采样记录得先补写到 `对标/对标采样执行单.md`，再写入本章细纲或写前闸门
     - 先读取 `references/benchmark-sampling.md`，看清这次对标现在只做到哪一步：
       - 学开头
       - 学正文
       - 章级局部借法
       - 准备升级为 skill 级规则
     - 要写清本轮实际读了哪些区段、多少章、多少场景、约多少字、这次判断现在稳到什么程度
     - 如果当前只够到“本章局部借法”这个采样量，就别把话抬成“通用正文写法 / 通用对白走法”

- 连续两章只靠查账、验货、摸口、放饵推进：暂停批量，先回炉细纲

- 连续两章只有信息升级，没有情绪升级：暂停批量，先补兑现
- 连续两章主要战果都是“拿到下一条线/更接近幕后/更接近证据”：暂停批量，先补当章翻面

- 连续两章前 1/3 都没有现实翻面：暂停批量，先重切章节走法
- 连续两章都主要靠“统一判断、统一说法、分工布置”完成章节作用：暂停批量，先补当章压倒对象和现实翻面
- 连续两章读感像“高质量提纲扩写”：暂停批量，先把正文分量补回来
- 连续两章书名主打收益都虚，只剩副线收益：暂停批量，先回钉主要推进
- 连续两章同场角色说话越来越像同一个人：暂停批量，先做角色出场分量和台词边界回炉
- 连续两章关键配角都只会认出 / 害怕 / 解释 / 附和：暂停批量，先回细纲补 `进场三问 / 独立动作 / 独有记忆点`
- 连续两章人物谁前谁后主要靠旁白解释，不靠动作/回避/矛盾漏出来：暂停批量，先回炉人物写法，别继续往下推
- 连续两章补字主要靠直译人物算盘或补旁白解释：暂停批量，先改字数扩写写法，别继续顺推
- 连续两章关键场景都只有一道冲突、没有第三方插手：暂停批量，先回细纲补 `冲突拆开 / 插手点 / 升级动作`
- 连续两章都在正文阶段临时补桥、补动作、补险点：暂停批量，先回细纲给关键场面补 `场面卡 / 动作拍序`
- 连续两章都像“一章一个平任务”，没有中段被压回来和情绪回摆：暂停批量，先回细纲补 `事件拍序 / 情绪回摆`
- 发现追踪文件提前写死完成值或时间线双版本并存：立刻停写，先清追踪里的旧说法
- 发现系统性解释腔：不要靠继续写“冲过去”，应切到 revision workflow 或先做局部回炉
- 发现系统性口号腔/抽象词复读：别靠换几个近义词糊弄，得回到具体动作、具体后果、具体疼感层面重写

**全量审查节点（平时就按这个走）**：

- 日更续写前，先读 `追踪/阶段审查计划.md`
- 如果该文件已经写明“这一阶段写完了”或“已经写到该做阶段审查的时候”，就先停下顺写，先跑一次 `story-review full`
- 如果该文件缺失，就先补阶段审查计划，再继续写

另外下面这些时候也得停一下：

- 连续 `3` 章以上回炉或大修刚完成
- 章节组主要推进发生改写
- 已经出现一次 `F5`，且修的是已落地事实 / 追踪 / 交叉复查
- 当前卷结束，且阶段计划尚未显式覆盖该节点

这里别跳步：

- 写到该做阶段审查的时候，别直接跳过去顺写
- `scene_lint.py` 过了，也不等于能跳过阶段审查
- 如果用户明确要求“先继续写”，也先提示现在已经写到阶段审查节点

**资料研究（按需）**：如果写作中遇到需要查证的外部事实（历史年代、地理方位、职业细节等），就先停下写作，拉 `story-researcher` 子代理去查，并输出到 `参考资料/` 目录。研究完再继续写。

## Step 3：最后总查一遍

批量写作结束后，把这次新写的所有章节再过一遍第 5 阶段检查，最少看这些：

1. **禁用词扫描**：对照 `references/banned-words.md`，一级词一旦出现就换掉
2. **标题去重检查**：汇总本轮新写章节与既有标题；发现同名或明显重复时，回到对应细纲和正文文件统一重命名
   - 同时看一眼有没有“开始 / 线索 / 局势 / 真相 / 博弈 / 残角 / 刀鞘”这类单薄概括型标题；一旦出现就按标题规则回炉
3. **钩子检查**：每章章尾是否有钩子
4. **伏笔盘点（只看本轮新增）**：只确认这一批新增 / 推进 / 回收的伏笔已经写进 `追踪/伏笔.md` 并更新状态；别在日更这条线里通读所有 session 或扫描全部正文做全量伏笔审计。全量伏笔审计只在 `/story-review` 或用户明确要求"全面检查伏笔"时做

> 更完整的第 5 阶段检查清单，看 SKILL.md 对应的第 5 阶段部分。

## 写作进度

- 下一章细纲状态：{已有/需补建}

## 细纲缺了就按这条补

如果发现细纲不存在，别跳过，就按下面这几步补：

1. 加载 `大纲/卷纲_当前卷.md`（本章对应的事件规划）
2. 加载本章涉及的 `设定/角色/{角色名}.md`（角色状态）
3. 读取最新一章正文（情节衔接）
4. 按 SKILL.md 第 3 阶段那套细纲写法补建本章细纲
5. 补建完后继续 Step 2 写作

## 常见情况

| 问题 | 处理 |
|------|------|
| 细纲不存在 | 按上面的“细纲缺了就按这条补”去做 |
| 追踪文件为空 | 正常继续，写作中逐步填充 |
| 用户要求改大纲 | 提醒"改大纲会影响后续细纲"，确认后再改，并标记受影响的细纲 |
| 写到卷末 | 提示用户"当前卷已完成，是否开新卷？" |
| 用户中断批量写作 | 保存当前章节，已更新追踪文件，下次从断点继续 |
