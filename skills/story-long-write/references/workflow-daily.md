# `workflow-daily.md`：日更续写工作流

本文件为"日更续写"场景的完整指引。SKILL.md 路由到本文件后，按以下流程执行。

> **日更模式准备步骤**：每章写作前执行 2 步（状态筛选 + 意图确认），嵌入 Step 2 逐章执行循环内。日更模式省略对标结构化子目录检索（SKILL.md 步骤 3.2 的第二部分），因为题材框架已在 `设定/题材定位.md` 中固化，对标模块已在 Phase 2-3 中内化到卷纲和细纲。情绪-技法匹配（步骤 3.2 的第一部分）通过 Step 2.2 的意图确认覆盖。完整准备层逻辑见 `SKILL.md` 的 Phase 4 正文写作辅助部分。

---

## 适用条件

- 项目已有 `正文/` 和 `追踪/` 目录
- 用户说"日更""续写""继续写"
- 目标：每次会话写 2-3 章（4000-9000 字）

## 单文件串行模式补丁

当用户明确要求：

- `每次只写一个文件`
- `冷启动测试`
- `不要停`
- `一直往下推`
- `继续`

且当前任务属于同一批次连续生产时，日更工作流必须切换到 `单文件串行模式`：

1. 每个执行步只允许新增或改写 `1` 个项目文件
2. 一个文件完成后，若不存在阻塞信息，不得询问“是否继续”，必须直接进入下一个执行步
3. `单文件` 约束的是单次落盘动作，不是单个用户回合；同一用户回合内允许连续执行多个执行步
4. 下一个执行步默认按既定生产顺序顺延：
   - 缺细纲先补细纲
   - 有细纲先写正文
   - 正文写完先做本章必要追踪同步，再进入下一个文件
5. 只有命中以下情况才允许暂停：
   - 章节号冲突
   - 细纲缺失且无法安全补建
   - 追踪/正文真实锚点冲突
   - 命中阶段审查节点，必须先执行 `story-review full`
   - 用户新指令改变既有大纲、卷任务或生产范围
6. 若当前文件已完成而模型仍准备收束回复，视为流程未执行完；应先检查是否存在“下一文件已可确定”，若可确定，则继续推进，不得把“本文件已完成”误判成“本轮工作已结束”

## 人工接管补丁

日更 / 冷启动过程中，`人工接管模式` 是命中条件后的自动返修态，不是询问用户后的可选项。只要命中 `core-gates.md` 的 `H1` 条件，必须直接标记为 `H1-A` 或 `H1-B` 并执行对应口径，不得停下来问“是否人工介入”。

### `H1-A`：纯 skill 基准验证

- 目标：测试 skill 自然产出
- 原则：不靠人工重写来掩盖结构问题
- 允许接管的范围仅限：
  - 清污染残片
  - 修文件截断
  - 清明显非故事语料
  - 修运行时异常导致的不可读段
- 正文不过线时，只写失败记录、失败码、触发规则和原样样本路径，不修正文

### `H1-B`：生产模式

- 目标：把书真正写到可用
- 原则：主线程自动接管细纲、正文、追踪
- 但接管后必须：
  - 明写“当前为人工接管模式”
  - 写明接管原因和文件范围
  - 写后继续走闸门、验收、追踪同步

若用户未特别要求“纯 skill 原样测试”，默认使用 `H1-B`；不得把生产模式下的 `H1` 再交还给用户选择。

## Step 1：快速上下文加载

### Step 1.0：先确认真实锚点

进入日更前，先确认“当前真实续写点”：

1. 读取 `追踪/上下文.md` 的最后完成章节或当前位置
2. 扫描 `正文/` 目录中编号最大的章节
3. 检查 `追踪/时间线.md`、`追踪/角色状态.md` 是否仍停留在更早章节，或错误超前到尚未真正落地的远期预案
4. 如果出现以下任一冲突，先修追踪，再写正文：
   - `追踪/` 仍写在旧章节，但正文已经大修到更新章节
   - `追踪/` 默认锚点超前到未真正落地的后续卷/后续章
   - `时间线/角色状态/上下文` 三者彼此口径不一致
   - 细纲把旧预案当成已发生事实
   - `角色状态/情报台账/伏笔` 把“计划补强、准备兑现、预期会成立”的内容写成已完成
   - `时间线.md` 或其他追踪文件里同一事件存在双版本并存、旧版未废止

**判定原则**：
- 以“已存在正文 + 已同步追踪”的最近一段为真实锚点
- 旧卷纲、旧预案、远期细纲不能直接当成真实已发生内容
- 锚点没确认清楚前，不得直接续写正文

**可选：使用 story-explorer 子代理批量加载上下文**。如果项目已部署 `story-explorer`（检查 `.codex/agents/story-explorer.md` 是否存在），可用 `spawn_agent(agent_type="default", task="读取项目目录：{dir}\n查询类型：context_load\n查询参数：准备写第 {N} 章", context=".codex/agents/story-explorer.md 的约束")` 执行 `context_load` 查询，一次获取全部写作上下文。子代理返回后直接使用其 `results`，跳过下方手动加载步骤。如果子代理不可用或返回不完整，回退到下方手动加载。

手动加载（默认方式）：

| 序号 | 文件 | 用途 | 如果不存在 |
|------|------|------|-----------|
| 1 | `追踪/上下文.md` | 上次写作进度摘要 + 当前真实锚点 | 从 `追踪/伏笔.md` + `追踪/时间线.md` 重建 |
| 2 | `设定/爽点模型说明书.md` | 当前小说类型 + 当前阶段爽法 + 收益分配 | **必须先补建**，不允许跳过 |
| 3 | `追踪/伏笔.md` | 待回收伏笔清单 | 跳过 |
| 4 | `追踪/时间线.md` | 当前事件时序 | **必须先补建**，不允许跳过 |
| 5 | `大纲/细纲_第{N}章.md` | 本章写作计划 | **必须先补建**，不允许跳过 |

**按需加载角色文件**：从细纲中提取本章涉及的角色名，按需加载 `设定/角色/{角色名}.md`。如果细纲中未列出角色，跳过。

**按需加载创作公式**：当写作中需要引用创作公式约束时（如期待感公式、爽点公式、信息差公式），加载 `references/genre-writing-formulas.md`。默认不加载，避免无条件加载 1500+ 行文件浪费 token。

### 已写内容分层摘要

当项目超过 30 章时，`追踪/上下文.md` 中的已写内容摘要应按三层结构组织，压缩早期章节、保留近期细节：

| 层级 | 粒度 | 格式 | 维护时机 |
|------|------|------|----------|
| **近5章详记** | 近 5 章逐章摘要 | 保留现有格式（每章一段：事件+状态变化+伏笔） | 每章写完后更新 |
| **十章概要** | 每 10 章一段概要 | `章节范围 | 已写核心事件 | 角色状态变化` | 每写完 10 章生成一段 |
| **卷级总览** | 每卷一句话总结 | `卷名 | 已写主线进展 | 关键转折` | 每卷结束时生成 |

**示例（50 章项目）**：
```markdown
## 近5章详记（第 46-50 章）
第 46 章：主角进入秘境，发现上古遗迹...
第 47 章：遭遇守护兽，险胜后获得传承...
...

## 十章概要
第 1-10 章 | 主角觉醒金手指，进入修炼学院 | 从废柴变为内门弟子
第 11-20 章 | 学院大比，击败天才对手 | 获得长老关注，实力突破
第 21-30 章 | 离开学院，探索秘境 | 发现身世线索，结交盟友
第 31-40 章 | 卷入势力争斗，暴露身份 | 被追杀，实力再突破
第 41-50 章 | 反杀追兵，进入新地图 | 确认血脉传承，进入上界

## 卷级总览
第一卷·觉醒 | 废柴逆袭成天才 | 金手指觉醒，身世初现端倪
```

> **首次日更 fallback**：如果 `追踪/` 目录下的文件全部为空或不存在（刚完成 Phase 3 但未日更过），额外加载 `大纲/卷纲_当前卷.md` 和最新一章正文来重建上下文。

> **确定下一章编号 N**：从 `追踪/上下文.md` 的"最后完成章节"字段读取。如果文件不存在，扫描 `正文/` 目录中编号最大的章节 +1。

> **锚点冲突优先处理**：如果 `追踪/上下文.md` 记录的最后完成章节，与 `正文/` 最大章节号、`时间线.md` 当前时间切点、`角色状态.md` 默认锚点冲突，不要猜测，不要硬续。先把追踪同步到真实正文，再确定下一章编号。

确定本轮写作范围后直接进入 Step 2，不做"是否继续"式确认。K 默认 2-3 章；若命中 `单文件串行模式`，则把 `K` 理解为“同批次连续执行的目标范围”，不是“单步写多个文件”。此时必须改为“每步 1 文件、步间不停、同回合连续推进”。仅在章节号冲突、细纲缺失、请求范围超过已有细纲、用户要求会改变大纲/追踪，或存在其他会导致写错的阻塞信息时暂停确认。

## Step 2：串行批量写作

一次加载多章细纲，但**必须在主会话内串行逐章写作**：不得把多章同时交给多个子代理并发写。长篇章节依赖上一章正文和追踪文件，并发会导致上下文断裂、追踪覆盖和标题去重失效。

**批量 continuation 规则**：进入本 Step 后，"继续"/"续写"/"日更"只表示继续当前日更批量流程。不得把这些词解释为跳过 Step 2.2/Step 2.3 的直接正文续写；也不得在每章之间重复询问是否继续，除非用户明确要求逐章确认或出现阻塞。

1. **读取细纲**：加载本次要写的 2-3 章细纲
2. **逐章执行**（以下每步在每章循环内执行）：
  - 读细纲 → 按需加载角色设定
  - **2.0 状态确认**：进入本章前，必须先明确当前项目状态：
     - `S0-S7` 中处于哪一档
     - 上一章最终失败码是否为 `F0`
     - 若上一章不是 `F0`，立即中断本章，禁止继续顺写
     - `追踪/时间线.md`、`追踪/角色状态.md` 必须存在；任一缺失，立即停在本步，先补文件
     - 若作品命中 `情报流 / 系统流 / 预知流 / 任务流`，先检查 `追踪/情报台账.md` 是否存在；不存在直接停在本步，先补文件
     - 若本章编号 `>= 3`，先执行最近 `2-3` 章的 `章节组闸门`；未执行或不过线，禁止写正文
   - **2.1 标题预检**：读取 `references/chapter-title-rules.md`，扫描既有章节标题；如本章标题同名、明显重复、过于概括、过于剧透、只有物件说明没有冲突感，或命中“最值钱的不是……是……”这类作者判价句，先按本章第一爆点 / 核心战果 / 章尾钩子重命名，并同步细纲标题与正文文件名
   - **2.2 状态筛选**：每章开始前必须确认以下来源已经在本轮 workflow 中读取或刚更新：本章细纲、上一章正文（或上一章刚写入的正文）、`追踪/上下文.md`、`追踪/伏笔.md`、`追踪/时间线.md`、`追踪/角色状态.md`；涉及情报流时，还必须确认 `追踪/情报台账.md` 的来源；涉及角色时，还必须确认对应 `设定/角色/{角色名}.md` 的来源。"已加载"只指本轮 workflow 内实际读取/更新过的文件，不得用未标明来源的聊天记忆替代。角色最新状态优先从 `追踪/角色状态.md` 筛选，待回收/推进伏笔从 `追踪/伏笔.md` 筛选；缺任一追踪主表时，先补主表，不允许直接写正文
     - 若本章涉及连续登场的重要角色，除角色卡外，还必须回看其最近一次正文出场，确认外形锚点、动作锚点、说话锚点没有漂移
   - **2.3 意图确认**（一句话）：从细纲「目标情绪」字段确认本章情绪目标，综合状态筛选结果，用一句话确认本章写作意图（情绪+节奏）
   - **2.3.1 爽法校准**：用 `设定/爽点模型说明书.md` 校准本章，明确：
     - 本章属于哪一类爽点兑现
     - 本章不该滑成哪类写法
     - 本章收益应该落在命 / 人 / 名 / 权 / 物证 / 站队 / 规矩变化中的哪几类
     - 若题面涉及配额 / 经营 / 分配 / 执牌 / 掌规则，本章必须明确“结果化兑现口”是什么；若只能写成“更接近”，停止写正文
   - **2.3.1.1 情报条目预检**：若作品命中情报流：
     - 读取 `追踪/情报台账.md`
     - 确认本章已预填 `刷新章 / 预计兑现章 / 预计收益类型`
     - 确认本章关键翻盘不是靠“临场观察”替代情报
     - 任一项不成立，先补台账或回细纲，不得写正文
   - **2.3.1.2 章节组闸门**：若本章编号 `>= 3`，正式写正文前必须写出最近 `2-3` 章结论：
     - 最近三章主收益骨架是否重复
     - 最近三章翻盘模板是否重复
     - 上一章章尾是否提前做空本章主任务
     - 题面卖点是否真的往前推了一步
     只要有一项答不稳，按 `F3` 回退 `章组控制 / 细纲`
   - **2.3.2 场面卡抽取**：正式写正文前，必须从本章细纲中单独抽出 `本章关键硬场面卡`，至少确认：
     - 本章最硬的 `1-2` 个场面到底是哪一场
     - 每场的先手方、截堵点/爆点、动作拍序、险点、翻盘结果是否已写清
     - 若本章属于追逐 / 截堵 / 反绕 / 抢签 / 战斗 / 谈判翻桌 / 势力撞局 / 灾异逃生等高动作章节，正文必须按 `场面卡 -> 正文` 顺序落，不得直接拿章节摘要硬扩写
     - 若 `场面卡` 仍只有“对手来抢 / 主角反打 / 最后翻车”这类结论句，停止写正文，先回补动作拍序
   - **2.3.2.1 事件拍序预检**：正式写正文前，必须单独确认 `本章事件拍序`：
     - 本章至少有 `3-5` 拍推进，不得只有“目标 -> 执行 -> 结果”
     - 至少有 `1` 拍是受阻或反压
     - 至少有 `1` 拍是变招、舍取或代价落地
     - 至少有 `1` 拍造成现实改盘，而不是只抬信息
     - 若写不出以上四项，停止写正文，先回细纲
   - **2.3.2.2 情绪回摆预检**：正式写正文前，必须单独确认 `本章情绪回摆`：
     - 本章至少有 `2` 次情绪摆动
     - 情绪摆动必须绑在事件上，不得只写心理句
     - 每次摆动至少落到 `动作停顿 / 改口 / 舍取 / 失态 / 代价 / 站位变化` 之一
     - 若整章情绪只有一条直线，如一直查、一直稳、一直强、一直冷，停止写正文，先回细纲
   - **2.3.2.2.1 非预期障碍预检**：正式写正文前，必须单独确认 `本章非预期障碍`：
     - 至少 `1` 口障碍不是主角已知旧阻力的平推复现
     - 障碍必须写清来源：第三方插刀 / 世界环境改压 / 对手换招 / 协作方失手 / 时间窗口收口
     - 若障碍只能写成“更难了 / 对手更狠了 / 局势更乱了”，视为未具象化，不得写正文
   - **2.3.2.2.2 受阻过桥预检**：正式写正文前，必须单独确认 `本章灾后反应 -> 两难 -> 决断过桥`：
     - 至少有 `1` 次受阻后，不是直接跳结果，而是出现短反应、舍取或改决断
     - 这次过桥必须回答：主角或关键角色到底舍了哪一口、改了哪一步
     - 若答不出，说明情绪只是概念，没有钉在事件里，不得写正文
   - **2.3.2.3 对标拆对白预检**：正式写正文前，必须先完成本章 `对白对标拆解结果`：
     - 若项目已经启用对标流程，先检查 `对标/对标采样执行单.md` 是否存在；不存在先创建，不得继续
     - 本轮采样记录必须先回填到 `对标/对标采样执行单.md`，再写入本章细纲或写前闸门
     - 先读取 `references/benchmark-sampling.md`，确认这次对标属于：
       - 学开头
       - 学正文
       - 章级局部借法
       - 准备升级为 skill 级规则
     - 必须写清本轮实际读了哪些区段、多少章、多少场景、约多少字、当前结论稳定度
     - 若当前只达到“本章局部借法”采样量，不得把结论写成“通用正文规则 / 通用对白规律”
     - 优先读取项目内 `对标/` 目录现成样本；没有项目内样本时，才允许退回通用方法层
     - 样本数量控制在 `1-3` 本，避免混腔
     - 每个样本只允许提炼机制，不允许照抄成品句子
     - 至少写清以下五项：
       - 主角这类场景通常用对白干什么
       - 对手在这种场景先护什么、会怎样说
       - 摇摆位通常如何露底
       - 围观位 / 第三方通常只会报什么自身利益
       - 章尾最后 `1-3` 轮对话通常停在命令 / 动作 / 物证 / 名单中的哪一类
     - 若提炼结果仍是“对白自然 / 很有气势 / 很像人话”这类形容词，视为未拆开，不得写正文
     - 若提炼结果主要是漂亮金句、爽句、口号句，而不是角色功能与动作承载，视为拆偏，不得写正文
     - 若对标记录里没有写清“实际读量”和“结论稳定度”，视为采样口径不合格，不得写正文
     - 若 `对标/对标采样执行单.md` 没有同步更新本轮实际读量，视为对标流程未完成，不得写正文
   - **2.3.2.4 对标拆旁白预检**：正式写正文前，必须先完成本章 `叙述对标拆解结果`：
     - 同样必须沿用上面的采样口径，不得对白写一套、叙述写另一套模糊口径
     - 优先读取项目内 `对标/` 目录现成样本
     - 不得只记一句“某书叙述很强”，必须拆成以下四项：
       - 开场如何用现场载体压进局里
       - 中段如何让冲突自己翻出来
       - 人物站位如何通过动作 / 器物 / 目光 / 改口显出来
       - 章尾如何停在新结果，不停在作者盖义
     - 若提炼结果仍是“文风强 / 叙述有味 / 气氛好”这类形容词，视为未拆开，不得写正文
     - 若提炼结果主要是成品开场句、成品收尾句、成品旁白金句，视为拆偏，不得写正文
     - 若当前采样只够支持“本章叙述借法”，不得上升成“本书通用叙述规则”
   - **2.3.2.5 功能场景四步预检**：若本章命中 `顺位 / 配额 / 放行 / 审批 / 焚票 / 屏幕 / 夹板 / 名额 / 服务口 / 制度口 / 规则口` 任一功能场景，正式写正文前必须先写清：
     - `谁想拿什么`
     - `谁先动手`
     - `谁被迫改动作 / 改口 / 改站位`
     - `规则因此如何显形`
     - 若答题顺序自然变成“先讲规则再讲动作”，停止写正文，先回细纲
   - **2.3.2.6 动作骨架卡**：正式写正文前，必须单独写出本章 `动作骨架卡`：
     - 至少 `4-6` 拍
     - 每拍只能写 `谁做了什么 -> 谁被迫怎么改`
     - 不得把“规则说明 / 人物总结 / 作者判断 / 世界全景”塞进动作骨架卡
     - 若删掉规则解释后，这张卡自己跑不动，停止写正文，先回细纲
   - **2.3.2.7 人物冲突卡**：正式写正文前，必须单独写出本章 `人物冲突卡`：
     - 主角：嘴上报什么，动作先护什么
     - 对手：嘴上压什么，动作先藏什么
     - 摇摆位：嘴上推什么，动作先避什么
     - 若答不出任何一人的“先护哪口 / 先避哪口”，停止写正文，先回角色卡或场面卡
   - **2.3.2.8 章尾现场句预案**：正式写正文前，必须先钉一条 `章尾现场句预案`：
     - 只许属于 `新动作 / 新物证 / 新名单 / 新命令 / 新翻面`
     - 只许写现场，不许写意义
     - 若自然写成“更大的还在后头 / 真正值命的是 / 现在才只是开始”，停止写正文，先回章尾设计
   - **2.3.2.9 人物说话权限表**：正式写正文前，必须给本章关键角色写一句 `只许怎么说，不许怎么说`：
     - 主角：
       - 只许报后果、报死线、报口子、报下一拍会坏在哪
       - 不许完整播报制度全貌
     - 对手：
       - 只许护自己那一口，拿规则压人
       - 不许替读者讲清整套规则怎么运转
     - 摇摆位 / 协作方：
       - 只许报自己利益、留半句、递半真半假的口
       - 不许一张口就把真相和规则全说完
     - 围观位：
       - 只许报自己要不要活、要不要进、要不要抢
       - 不许替主角总结局势
     - 若任一角色的台词功能主要变成“制度解说员”，停止写正文，先回角色卡和场面卡
   - **2.3.3 写章施工单**：进入正文前，用 `3-6` 行写出本章施工顺序，至少包含：
     - 章首第一爆点落在哪
     - 中段最硬场面按哪张场面卡展开
     - `动作骨架卡` 的第几拍最值钱
     - 后 `1/3` 的现实翻面落在哪
     - 章尾停在哪一拍
     - 章尾最后一句属于哪一种 `末句功能型`：
       - `新动作`
       - `新物证`
       - `新口径`
       - `新名单`
       - `新翻面`
       - `下一步命令`
     该施工单只服务本章落地，不并入正文；若写不出，视为本章尚未可施工
   - **2.4 写前硬闸**：正式写正文前，必须逐条确认：
     - 本章最硬的现场是什么
     - 本章核心战果是什么
     - 本章题面结果化兑现是什么，不得回答“更接近 / 快摸到 / 差一点”
     - 本章谁压谁、谁翻盘
     - 本章是否只是“知道更多”，却没有“拿到更多/逼出更多”
     - 本章最该被读者记住的三个“具体”是什么：
       - 具体动作
       - 具体后果
       - 具体疼感/失态/代价
     - 本章是否在依赖抽象效果词、判断词、作者定性词撑气氛；如果删掉“狠、硬、爽、炸、压、危险、关键、值钱”这些词，正文是否还成立
     - 本章关键翻盘若去掉“群众炸锅 / 围观起哄 / 物证见光”，是否还成立；如果不成立，说明又在复用旧模板，必须回细纲
     - 本章前 1/3 是否就会发生一次现实翻面；如果不会，为什么还能撑住留存
     - 本章是否至少有 `1` 次“快拿到 -> 被打断”和 `1` 次“被打断 -> 改决断 / 变打法”
     - 本章是否至少有 `1` 次非预期障碍打乱原计划，而不是只把原计划写得更长
     - 如果上一章也是查信息推进，本章是否承担兑现职责
     - 本章谁会先失态、先丢脸、先被逼站队
     - 本章除了主推进外，至少还有哪两层“肉”：阻力层、动作层、围观层、对手失态层、人物气场层、代价层
     - 本章如果写完只剩“拿到下一条线”，是否应直接判薄并回补细纲
     - 本章是否符合 `设定/爽点模型说明书.md` 规定的当前阶段爽法；如果不符合，是不是已经写偏
     - 本章连续共演的关键角色，删去名字后能否仅凭口气、句式、动作习惯分出来；如果分不出来，先回补角色卡和细纲
     - 本章重要角色是否至少有一个稳定外形/动作/气场锚点会在现场真正落地，而不是只在设定里存在
     - 本章关键角色是否都答得出 `进场三问`：
       - 现在想要什么
       - 现在怕失去什么
       - 进场前 `5` 分钟在做什么
     - 本章关键配角是否至少有一次按自己利益作出的动作，而不只是认出 / 解释 / 害怕 / 失口
     - 本章关键场景是否至少有 `双层冲突`：
       - 明面在抢什么
       - 暗地里各自真正护什么
     - 本章关键场景是否已有 `第三方插刀点`；若只有主角和对手互顶，先回补
     - 本章冲突升级是否靠动作完成，而不是只靠对骂和狠话
     - 本章关键场面是否按主角感知顺序落地，而不是按作者说明顺序落地；若只能说出“先发生了什么后发生了什么”，说不出“主角先注意到什么、先误判什么、先改了什么”，停止写正文
     - 本章最值钱的 `1-2` 秒是否已经写清近景承载；若还准备靠“很快 / 一下 / 顿时 / 局势变了”带过去，停止写正文，先回场面卡
     - 本章有限视角是否已经锁定；若一段里准备同时写清多方心思和全场总览，停止写正文，先收回 POV
     - 本章对白对标拆解是否已经落到“主角 / 对手 / 摇摆位 / 围观位 / 章尾收口”五项，而不是只写“参考某书风格”
     - 本章是否正在偷学对标书的成品金句、公告腔、收尾腔；若是，停止写正文，先回对标拆解结果改成机制借法
     - 本章叙述对标拆解是否已经落到“开场压现场 / 中段翻冲突 / 动作显站位 / 章尾停结果”四项，而不是只写“学某书文风”
     - 本章是否正在偷学对标书的成品开场句、成品章尾句、成品旁白金句；若是，停止写正文，先回叙述对标拆解结果改成承载借法
     - 本章对标记录里是否已经写清“本轮抽读 / 本轮连续读到 / 本轮有效场景单元 / 当前结论稳定度”；若没有，视为对标流程未完成
     - 本章人物算盘是否主要靠旁白直译；若删掉“他怕的不是A，是B / 他最想要的是 / 他终于算明白了”这类句子，人物是否还成立
     - 本章关键人物是否至少有 `1` 处“嘴上说A，动作却护B”或“被问A，却绕去答B”的矛盾 / 回避点
     - 本章关键对白能否答出“这句是在压人 / 套话 / 甩锅 / 自保 / 拖时间 / 逼站位”中的哪一项；若答不出，只剩讲解功能，停止写正文
     - 本章高压场景对白是否写成了“完整论证 / 制度播报 / 工整金句”；若是，停止写正文，先回场面卡改对白打法
     - 本章关键角色是否越权说话：
       - 主角是否在完整播报制度答案
       - 对手是否在替读者讲清规则全貌
       - 摇摆位是否一张口就把真相全抖完
       - 围观位是否在替主角总结局势
       任一为“是”，停止写正文，先回人物说话权限表
     - 本章旁白是否准备替人物、替场面、替章尾发力；若已经出现“先乱的不是……是…… / 最值钱的不是……是…… / 这不是在……是在……”这类叙述收力句，停止写正文，先回动作层
     - 本章旁白是否正在替读者完成判断；若已经出现以下任一句法，停止写正文，先改成动作版：
       - `这一下比……更……`
       - `从这一刻开始`
       - `真正……的是……`
       - `不是怕A，是怕B`
       - `不是为了A，是因为B`
     任一条答不上来，停止写正文，先回补细纲
   - 写正文
     - 正文落地第一原则：`先动作，后规则`
     - 规则信息只能跟在：
       - 抢
       - 卡
       - 放
       - 挡
       - 改口
       - 改站位
       - 停门
       - 锁闸
       - 广播改令
       这些现实变化后面出现
     - 禁止一整段先讲“这地方平时怎么运转 / 常规规则如何执行 / 真正值钱的口子在哪”，再让人物去跑场
     - 正文落地顺序固定为：`章首爆点 -> 场面卡1 -> 第二口收益/外部反应 -> 场面卡2（如有） -> 章尾翻面`
     - 不得把“场面卡里没写清的桥、动作、险点、抢口、撞局结果”留到正文里临场发明
     - 若写到中段才发现必须新增一个未备案的人、地点、物证、暗号或新规则才能跑通，立即停写，按 `F3/F5` 回退到细纲层
   - **字数验证（优先 Python 字符统计，`wc -m` 仅作 Unix 备选，< 目标90%则强制扩充）** → 检查钩子/爽点 → 禁用词扫描
     - 强制说明：字数不足时，扩充只能补：
       - 动作
       - 阻力
       - 对手反应
       - 围观反应
       - 代价
       - 角色显影
       - 当章翻面
       - 人物互顶
       - 身体先泄露的反应
       - 回避 / 岔答 / 装硬 / 甩锅
   - **对白人话复检**：正文首稿完成后、写后验收前，必须单独抽查所有关键对白：
     - 是否存在“说给读者听”的解释型对白
     - 是否存在一句话包办 `事实 + 判断 + 结论 + 方案`
     - 是否存在“不是A，是B / 谁现在……谁就…… / 不是你……说了算”这类模板发力句集中复用
     - 是否存在制度播报腔、公文腔、公告腔
     - 是否存在角色在极乱场面里还说得过分完整、过分漂亮、过分会说
     - 是否还能看出在照某一本对标书的成品声腔，而不是只借了角色功能机制
     - 若命中以上任一项，不得进入 `F0`；至少按 `F1` 回修
     - 若命中项集中在章尾关键对顶或高位敌我对撞处，默认升 `F2`
   - **旁白发力句复检**：正文首稿完成后、写后验收前，必须单独抽查关键叙述句：
     - 是否存在“先乱的不是……是……”这类开场硬发力句
     - 是否存在“最值钱的不是……是……”这类作者判价句
     - 是否存在“这不是在……，是在……”这类冲突翻译句
     - 是否存在“真怕……的，不是……”这类替读者总结人物站位的句子
     - 是否存在删掉后段落更顺、更像现场的叙述金句
     - 是否存在“作者已经替读者把这一拍解释完”的代判句
     - 若命中以上任一项，不得按“文风句”放行；至少按 `F1`
     - 若命中位置在章首、章尾、第一爆点或敌我对顶处，默认优先升 `F2`
       - 角色正在护的具体东西
       - 附着在上述内容上的具体细节
     - 若项目脚本下限为 `2900`，正文首稿低于 `3100` 时不得只做自由补写，必须按配额回补后再统计：
       - 前 `1/3` 至少补 `1` 口新的受阻 / 截堵 / 反压
       - 中段至少补 `1` 口第三方插刀 / 协作方改口 / 围观站位变化
       - 后 `1/3` 至少补 `1` 口现实代价 / 关系变坏 / 盘面改线 / 外部可感后果
       - 关键角色至少补 `1` 处“身体先泄露，台词后补位”或“嘴上说A，动作护B”
       - 未补满上述配额，不得进入下一轮字数统计，更不得收章
       - 第二层冲突
       - 第三方插刀
       - 冲突升级动作
     - 细节使用口径：
       - 细节只能跟着 `动作 / 阻力 / 代价 / 外部反应 / 人物显影` 走
       - 细节不能单独承担“这一段终于有内容了”的主功能
       - 若一段只有环境、神态、声响、物件描写，却没有改动作、改站位、改利益，视为无效补写
     - 不得用解释、总结、回顾、局势翻译、作者盖章来凑字数
     - 若为人物显影不足或字数不足而补写，优先补：
       - 谁先伸手去抢 / 去挡 / 去藏
       - 谁先护住门卡 / 名册 / 钱 / 面子 / 盘口 / 证据
       - 谁被点破后先改口、先沉默、先看向别处
       - 哪个第三方借机改站位 / 改口 / 递假东西 / 先撇责任
       - 谁把明面冲突翻成里层冲突
      - 哪个器物先露破绽 / 哪道旧伤先发作 / 哪条规矩先压到具体动作上
      - 不得优先补“他其实怕的是 / 他真正想要的是 / 他终于明白了”
   - **反流水账复检**：正文首稿完成后、写后验收前，必须单独抽查：
     - 是否连续出现“谁做了什么 -> 又做了什么 -> 然后又怎样”的事件记录句链
     - 是否能明确指出主角在每个关键场面里：
       - 先注意到什么
       - 先误判或起疑什么
       - 先改了什么招
     - 是否把关键翻面写成了“很快 / 一下 / 顿时 / 局势变了”这种总述句
     - 是否存在俯拍总览太多、主角肩后感太少的监控视角
     - 若命中以上任一项，不得按“节奏快”放行；至少按 `F1`
     - 若命中位置在第一爆点、高潮翻面、章尾前最后一拍，默认优先升 `F2`
   - **2.5 写后硬检**：每章写完后，必须确认：
     - 能不能回答“谁赢了、谁亏了”
     - 爽点是否已经通过场面落地，而不是只靠解释
     - 章尾是不是具体危险/发现/反扑，而不是总结
     - 章尾最后一句是否真的在 `新增现场`，而不是在解释前一拍为什么重要
     - 有没有出现“新线索很多，但情绪没抬升”
     - 有没有出现“删掉最后 1-3 句解释后，章节反而更利索”的情况；若有，必须删改
     - 必须执行 `末尾反删测试`：
       - 删掉最后一句后是否更强
       - 删掉最后两句后，本章是否仍完整成立
       - 若任一答案为“是”，默认章尾功能错误，不得进入下一章
     - 有没有出现“删掉几个抽象效果词后，整段就没劲了”的情况；若有，说明这段缺动作、缺后果、缺疼感，必须重写
     - 有没有出现连续 300-500 字只有判断、并盘、解释、分工，而没有现实动作变化
     - 有没有把配角写成作者嘴替
     - 这章读起来像“正式正文”，还是像“提纲扩写”
     - 是否真的有人被打疼、打丢脸、打失态，而且这些是否通过现场而不是作者盖章呈现
     - 是否改掉了一口现实盘面，而不只是更接近下一步
     - 是否落下了至少两层以上的“肉”，而不是只剩骨架推进
     - 这章的爽点类型是不是仍然符合说明书，而不是滑成“慢查黑幕”“只盘局势”“纯悬疑挂钩”
     - 情报流章节有没有真的出现“情报 -> 动作 -> 外部现实变化”，还是又被临场观察替代
     - 本章是否真实掌到题面承诺的一口权；若没有，是否至少逼敌方公开改动作 / 改规则 / 改站位
     - 本章翻盘模板是否仍与最近 `2-3` 章高度同型；若是，禁止进入下一章
     - 本章翻盘发动机是否与上一章实质不同；若只是换了地点、物证或对手名字，但仍是同一套 `抬证据 -> 对手护私账 -> 第三方怕背锅让位 -> 群体改口`，默认不得进入下一章
     - 若当前章与上一章共用同一发动机，本章是否至少换掉其中 `2` 个核心环节：
       - 主角主要抓手
       - 对手主要护点
       - 第三方插刀来源
       - 现实改盘口
       任一答不稳，先回细纲，不得顺推
     - 本章是否只有单一主事件结果，没有中段受阻、变招、舍取、代价
     - 本章关键场景是否真实形成了双层冲突，而不是只有单层抢口
     - 本章是否至少有 `1` 个第三方插刀点真正改了局，而不是只站着看
     - 本章情绪是否至少出现 `2` 次回摆，而不是一路平推到章尾
     - 本章重要角色这次出场后，读者能否再次记住其脸、动作、口气或压场方式；若仍只剩功能位，必须回炉
     - 本章多人对话删去人名后，是否仍能区分至少主角组与核心配角；若不能，必须重写对话边界
     - 本章关键配角是否真的做过一次“按自己利益出手”的动作；若只剩认出 / 翻译 / 失口 / 附和，默认人物显影不过线
     - 本章高价值角色是否至少留下 `1` 个可复现锚点；若只剩“设定上很强/很好看/很危险”，默认不过线
     - 本章关键人物的欲望、怕失去的东西、站位变化，是否主要靠旁白讲明；若是，默认人物显影不过线
     - 同一关键角色是否累计出现 `2` 句及以上“旁白替他讲算盘”的直译句；若是，默认人物显影不过线
     - 本章是否至少出现 `1` 处“身体先泄露，台词后补位”或“台词在回避，动作在说真话”；若没有，优先检查人物是否仍偏功能位
     - 命中解释腔 / 盖章句 / 作者替读者算账句时，是否已判明根因属于：
       - `G1 信息抢跑`
       - `G2 现场承载不足`
       - `G3 提纲语言漏出`
       - `G4 人物失声`
     - 若属于 `G2 现场承载不足`，是否已经回到前一拍补了：
       - 动作
       - 后果
       - 代价
       - 他人失态
       - 外部现实变化
       只删解释句不算修完
     任一项明显失败，本章不得进入下一章，先回炉本章
   - 每章写完后**立即更新**以下文件，不允许拖到整批结束再统一补：
     - `追踪/伏笔.md`（新增/回收伏笔）
     - `追踪/时间线.md`（记录事件时序）
     - `追踪/角色状态.md`（如本章引起角色状态变化——身份、能力、关系、公众形象——则更新对应角色条目并追加变更记录）
     - `追踪/上下文.md`（只更新进度元信息：当前位置+已写字数+本次变更，不写详细角色状态/伏笔内容）
     - `追踪/情报台账.md`（如果作品存在“系统/情报/任务/规则”类逐章兑现机制，本章新增、兑现、改口径时必须同步）
     - `设定/角色/{角色名}.md`（凡本章新增关键角色，或旧角色新增稳定外形/动作/说话/气场锚点，必须立刻同步）
   - **2.6 追踪反校验**：每章写完并更新追踪后，再反查一次：
     - `上下文.md` 的当前位置是否已经前推
     - `时间线.md` 是否写入了本章事件
     - `角色状态.md` 是否反映了本章最关键的人物变化
     - `伏笔.md` / `情报台账.md` 是否仍保留旧口径
     - 追踪文件是否把“计划中、待补、下一章准备写”的内容误记成已完成
     - `时间线.md`、`角色状态.md`、`情报台账.md` 是否存在同一事项双版本并存、旧版未废止
     - 本章新增的人物锚点是否已经回写角色卡，而不是只留在正文里
     - 若作品命中情报流，`情报台账.md` 是否已回填 `实际兑现章 / 实际战果`，且没有把“下一章预期”冒充“本章已兑现”
     - 追踪主表是否都能实答：
       - 主角这一章手里多了什么
       - 敌方这一章少了什么 / 被迫改了什么
       - 哪个配角这一章改了站位
       - 哪条旧线 / 旧物证 / 旧名单这一章往前推进了一口
       任一答不上来，说明追踪还停在空摘要，直接判 `F5`
     - 若本章编号 `>= 3`，是否已把本章 `章节组闸门` 结论同步进相关追踪，不得只在脑内通过
     若正文与追踪描述不一致，不得进入下一章
   - **质检提示**（可选）：本章写作完成。如需额外一致性检查，可使用 `story-review lean`。批量写作模式只允许跳过这一步“可选 lean 审查”，不允许跳过本章即时纠错、写后验收、追踪同步、反校验，也不允许把已识别的明确错误拖到“全部写完后再统一处理”。
3. **不中断但不并发**：一章写完不问用户，直接写下一章（除非用户要求逐章确认）；若命中 `单文件串行模式`，则改为“一文件写完不问用户，直接进入下一个文件”。下一文件开始前，必须读取上一文件刚写入的正文或追踪更新后再开始。

补充硬规则：
- 若当前任务口径为 `H1-C 协同生产态`，则“不中断推进”不等于“带错推进”。
- 只要当前章已经识别出明确错误，必须先在当前章原地修正并复检，再进入下一章。
- 不得用“批量写作”“后面统一审查”“先把几章写完再说”覆盖当前章的即时修正义务。

**批量报警规则**：

- 连续两章只靠查账、验货、摸口、放饵推进：暂停批量，先回炉细纲
- 连续三章无外显战果：暂停批量，先做节奏诊断
- 连续两章只有信息升级，没有情绪升级：暂停批量，先补兑现
- 连续两章主要战果都是“拿到下一条线/更接近幕后/更接近证据”：暂停批量，先补当章翻面
- 连续两章章尾都像“下一章再办”：暂停批量，先补即时反扑或即时结果
- 连续两章前 1/3 都没有现实翻面：暂停批量，先重切章节结构
- 连续两章都主要靠“统一判断、统一口径、分工布置”完成章节功能：暂停批量，先补当章压倒对象和现实翻面
- 连续两章读感像“高质量提纲扩写”：暂停批量，先补章肉厚度
- 连续两章题面收益都虚，只剩副驱动收益：暂停批量，先回钉主驱动
- 连续两章同场角色说话越来越像同一个人：暂停批量，先做角色显影和台词边界回炉
- 连续两章关键配角都只会认出 / 害怕 / 解释 / 附和：暂停批量，先回细纲补 `进场三问 / 独立动作 / 独有锚点`
- 连续两章人物站位主要靠旁白解释，不靠动作/回避/矛盾泄露：暂停批量，先回炉人物承载，不得继续往下推
- 连续两章补字主要靠直译人物算盘或补旁白解释：暂停批量，先改字数扩写口径，不得继续顺推
- 连续两章关键场景都只有单层冲突、没有第三方插刀：暂停批量，先回细纲补 `冲突分层 / 插刀点 / 升级动作`
- 连续两章都在正文阶段临时补桥、补动作、补险点：暂停批量，先回细纲给关键场面补 `场面卡 / 动作拍序`
- 连续两章都像“一章一个平任务”，没有中段反压和情绪回摆：暂停批量，先回细纲补 `事件拍序 / 情绪回摆`
- 发现追踪文件提前写满完成值或时间线双版本并存：立即停写，先清追踪口径
- 发现系统性解释腔：不要靠继续写“冲过去”，应切到 revision workflow 或先做局部回炉
- 发现系统性口号腔/抽象词复读：不要靠换几个近义词糊弄，必须回到具体动作、具体后果、具体疼感层面重写

**全量审查节点（默认生效）**：

- 日更续写前，必须读取 `追踪/阶段审查计划.md`
- 若该文件判定“当前阶段已完成”或“已命中阶段审查触发条件”，默认暂停顺写，先执行一次 `story-review full`
- 若该文件缺失，先补阶段审查计划，再继续写

额外强制触发条件：

- 连续 `3` 章以上回炉或大修刚完成
- 章节组主驱动发生改写
- 已经出现一次 `F5`，且修的是事实层 / 追踪层 / 反校验层
- 当前卷结束，且阶段计划尚未显式覆盖该节点

执行规则：

- 命中阶段审查触发条件时，不得直接跳过继续顺写
- `scene_lint.py` 通过不等于可以跳过阶段审查
- 若用户明确要求“先继续写”，也应先提示当前已命中阶段审查节点

**资料研究（按需）**：如果写作中遇到需要查证的外部事实（历史年代、地理方位、职业细节等），暂停写作，spawn `story-researcher` 子代理搜索并输出到 `参考资料/` 目录。研究完成后再继续写作。

## Step 3：质量检查

批量写作结束后，对本次所有新写章节执行 Phase 5 质量检查（至少包含）：

1. **禁用词扫描**：对照 `references/banned-words.md`，一级词命中即替换
2. **标题去重检查**：汇总本轮新写章节与既有标题；发现同名或明显重复时，回到对应细纲和正文文件统一重命名
   - 同时检查是否存在“开始 / 线索 / 局势 / 真相 / 博弈 / 残角 / 刀鞘”这类单薄概括型标题；命中则按标题规则回炉
3. **钩子检查**：每章章尾是否有钩子
4. **伏笔盘点（仅本轮增量）**：只确认本批新增/推进/回收的伏笔已写入 `追踪/伏笔.md` 并更新状态；不得在日更流程中通读所有 session 或扫描全部正文做全量伏笔审计。全量伏笔审计只在 `/story-review` 或用户明确要求"全面检查伏笔"时执行

> 完整 Phase 5 检查清单见 SKILL.md Phase 5。

## 细纲缺失补建流程

当检测到细纲不存在时，不能跳过。按以下步骤补建：

1. 加载 `大纲/卷纲_当前卷.md`（本章对应的事件规划）
2. 加载本章涉及的 `设定/角色/{角色名}.md`（角色状态）
3. 读取最新一章正文（情节衔接）
4. 按 SKILL.md Phase 3 的细纲模板补建本章细纲
5. 补建完成后继续 Step 2 写作

## 常见问题

| 问题 | 处理 |
|------|------|
| 细纲不存在 | 执行上方"细纲缺失补建流程" |
| 追踪文件为空 | 正常继续，写作中逐步填充 |
| 用户要求改大纲 | 提醒"改大纲会影响后续细纲"，确认后修改，标记受影响的细纲 |
| 写到卷末 | 提示用户"当前卷已完成，是否开新卷？" |
| 用户中断批量写作 | 保存当前章节，已更新追踪文件，下次从断点继续 |
