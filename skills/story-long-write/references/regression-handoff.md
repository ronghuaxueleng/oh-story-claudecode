# story-long-write 回归怎么交接

> 这一轮如果不是单纯写正文，而是在改 `story-long-write` / `story-review` 的流程、关键拦截、脚本读取规则或阶段检查规则，就先跑项目里的自动回归脚本，别只靠手工挑样本看。

---

## 一、什么时候要触发

碰上下面任一情况，这轮收尾前都要补一轮回归验证：

- 改了 `story-long-write` 的写前闸门 / 写后检查 / `F0-F5` 判断说法
- 改了 `scene_lint.py` 消费方式
- 改了项目脚本版本同步规则 / 脚本拷贝规则
- 改了收文判断必须统一 / 审查判断冲突规则
- 改了“中段会议化 / 作者直接替读者收尾的句子 / 过程写窄”拦截规则
- 改了“何时触发 `story-review full`”的阶段判断规则
- 改了连续回炉、固定检查单、复盘单、阶段审查计划相关规则
- 改了与 `story-review` 联动的默认流程
- 改了 `情报流 / 情报记录 / 章节组关键拦截 / 书名主打兑现阻断 / 失败码状态机` 任一执行结构

## 二、一般先这么走

如果项目根有：

```bash
scripts/story_review_regression.py
```

则优先运行：

```bash
python3 scripts/story_review_regression.py
```

然后读：

- `测试/story-review_回归脚本汇总.md`
- `测试/story-review_回归脚本汇总.json`

先看自动汇总，再决定要不要补人工复看，以及要不要把额外核查写进文件。

跑的时候盯这几件事：
- 手工跑 `scene_lint.py` 时，要逐章传“正文文件”，别把 `正文/` 目录整块塞进去
- 手工跑 `template_exhaustion_lint.py`、`scene_narrowness_lint.py` 时，也先按逐章传“正文文件”来；只有确认支持项目根模式，才传书目录
- 如果样本目录里混着“嵌套旧样本 / 子副本 / 备份样本”，回归报告先写清唯一主样本根目录，后面都按这个根目录取证

如果项目里没有这个脚本，再退回手工按 `story-review/references/review-regression-baseline.md` 一组一组跑样本。

另外这几条也别漏：
- 如果这轮改动落在下面任一项上，手工回归里至少再加 `1` 组“全新题材冷启动样本”，别只在旧书上来回测：
  - 情报流硬检查
  - 书名主打结果化兑现闸门
  - 章节组反重复闸门
  - 失败码阻断顺写
- 冷启动样本至少要盖到：
  - 开书设定
  - 前 `3` 章细纲
  - 前 `3` 章正文
  - 至少 `1` 次 `story-review full`
- 如果只跑旧样本、没跑新冷启动样本，就别说这轮流程改动“已经稳了”

## 三、最后最少说清什么

如果这轮是在改 skill 或改流程，不是单纯写正文，最后至少把这些话说清：

- 是否已跑自动回归脚本
- 已覆盖哪些基线组：`A组 / B组 / C组 / 扩展组`
- 当前用的是不是真实样本基线，还是还停在旧 `skill回归_*` 占位目录说法
- 是否额外覆盖“新题材冷启动组”
- 有没有误伤：
  - `warn` 被误写进失败码
  - `仅 warn` 样本误触发复盘单
  - `0 error / 0 warn` 样本在人物层 / 一致性层被硬判问题
  - `有 error` 样本被误判成可过
  - `scene_lint error` 被直接当成最终判断，未做人工覆核
  - 子代理与脚本冲突时，主线程没有写出 `人工覆核判断`
  - 项目内脚本版本落后，导致新拦截规则未真实执行
  - 写后检查与总安排表 / 固定检查清单的收文结论互相打架，却未被流程拦下
  - 模板主线已落地，但中段仍写成 `翻册/拍板/像在照着说明往下念`

## 四、注意事项

- 自动回归脚本只负责汇总，不能直接顶掉人工判断
- 如果脚本结果和人工预期撞上了，先查：
  - `script_version_check.py` 是否先跑过
  - `scene_lint.py` 版本是否一致
  - `scene_lint.py` 是否被错误喂成正文目录，而不是文件
  - 样本回归写入文件是否缺失
  - 是否误把嵌套子样本副本混读进主样本证据链
  - `story-review` 是否把 `warn` 与 `error` 混判
- 现在这批回归组再记一下：
  - `A组 / 食堂试勺记`
    - `解释腔样本`
    - 一直稳定报出：`解释.error=1`、`字数.warn=1`
  - `A组 / 裂雾供氧站`
    - `章尾硬拦截样本`
    - 一直稳定报出：`章尾.error=1`、`字数.warn=1`
  - `B组 / 黑板竞标会`
    - `情报流误触样本`
    - 一直稳定报出：`情报.error=4`、`章尾.error=1`、`字数.warn=1`
  - `C组 / 冷库截标会`
    - `三条固定要查项的单项正例`
    - 一直稳定报出：`字数.error=2`、`情报.error=1`
  - `扩展组 / 借婚封心局 / 潮口封单局 / 哭庙换命灯 / 锁门替签录`
    - 这几组分别拿来测：`本组第二章偏短链`、`双章尾风险`、`玄幻异题材纳管`、`解释+字数复合问题`
- 这一阶段顺手记一句：
  - 模板验证态下，先盯：
    - `收益写得太满了`
    - `非情报流误触`
    - `本组第二章偏短链`
  - `厚度 / 字数` 继续记，但除非用户明确说要测厚度，不然别抢主结论
- 这一轮新冒出来的根因也记上：
  - 如果出现“明明 skill 已修过，但新书还复发”，先查 `项目 scripts 是否与 skill 同步`
  - 如果出现“review 判不能收进正文，但写后检查自报能收进正文”，先查 `verdict_conflict_lint.py` 是否已接入
  - 如果出现“模板已用，但读起来像会议纪要”，先查 `scene_narrowness_lint.py` 和写后检查那三问有没有真跑
- 现在主要欠的不是脚本这边，而是单项样本这边：
  - 接近 `F0` 的干净正向样本
  - 纯 `非情报流误触` 样本
  - 纯 `本组第二章偏短链` 样本
  - 纯 `重要角色靠旁白先把话说破` 样本
  - 纯 `配角只剩讲解作用` 样本
  - 纯 `软肋角色掉线` 样本

### 6. 人物单项样本再怎么拆

- 后续新增人物单项回归样本时，建议至少拆成三类：
  - `skill回归_重要角色旁白先把话说破`
    - 目标：正文里重要角色看着像是写出来了，但关键价值主要靠旁白先替读者下判断的句子支撑
    - 期望：`character_agency_lint.py` 至少报出 `重要角色 ... 靠旁白先把话说破/把身价直接说满`
  - `skill回归_配角只剩讲解作用`
    - 目标：配角主要承担“把局势翻译给读者”
    - 期望：`character_agency_lint.py` 至少报出 `只剩讲解作用/收尾说明句功能`
  - `skill回归_软肋角色掉线`
    - 目标：第一章建立软肋，后续两章不再碰这条私人状态
    - 期望：`character_agency_lint.py` 至少报出 `软肋角色 ... 缺少私人状态再碰一下`

- 这三类样本在设计时不要混入：
  - `双版本并存`
  - `收益写得太满了`
  - `情报流误触`
  - 明显模板术语回流

- 人物单项样本的自动断言默认应为：
  - `character_agency_lint.py` 至少报出 `1` 条人物告警
  - 且必须报到该单项要抓的文案片段，如 `重要角色` / `只剩讲解作用` / `软肋角色`
  - 不要求 `agency warnings == 1`
  - 原因：同一坏样本往往会一起冒出“只像出来晃一下 + 只剩讲解作用 + 私人状态掉线”这类复合告警，若硬绑“只能 1 条”，反而会把有效检出误判成失败

- 否则人物脚本回归结果会被流程噪音抢走主结论。
  - 这组是这么搭的：
    - 复用“本组第二章偏短链”样本的题材与第001章完整开场
    - 第002章中前段先把最主要收益落成：
      - 停签状态已挂
      - 灰章已压
      - 外部先后已经变了
      - 蓝夹子已空置
    - 只把最后 `1-3` 句故意收成偏顺停位
  - 现在脚本报的是：
    - `scene_lint.py`：`字数.warn = 1`（第001章），`字数.error = 1`（第002章）
    - `validate_tracking_state.py`：`0 error / 0 warn`
  - 这一阶段先这样看：
    - 这组样本已经能证明“流程链纯净、最主要收益已在章尾前落成”
    - 但就脚本表层看，仍会先被 `字数.error` 抢走
    - 所以它现在更适合当“主线程人工分开判断基线”，不是给“脚本单看就能自动纯判”用的
  - 主线程这边要这样看：
    - 先确认第002章的停签、灰章、外部先后变化是否已在章尾前落成
    - 若已落成，再先把末句收尾问题记成 `本组第二章最后半拍回弹不足`
    - 不得直接因为 `字数.error` 把它退回成 `偏短链` 或 `整章兑现不足`
