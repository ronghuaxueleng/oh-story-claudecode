# story-long-write 回归移交流程

> 用途：当本轮工作不只是写正文，而是修改 `story-long-write` / `story-review` 的流程、门禁、脚本消费口径或阶段验收规则时，规定优先使用项目内自动回归脚本，而不是只靠手工挑样本验证。

---

## 一、什么时候要触发

命中以下任一情况，本轮结束前必须补一轮回归验证：

- 改了 `story-long-write` 的写前闸门 / 写后验收 / `F0-F5` 口径
- 改了 `scene_lint.py` 消费方式
- 改了“何时触发 `story-review full`”的阶段闸门规则
- 改了连续回炉、固定验收单、复盘单、阶段审查计划相关规则
- 改了与 `story-review` 联动的默认流程
- 改了 `情报流 / 台账 / 章节组闸门 / 题面兑现阻断 / 失败码状态机` 任一执行结构

---

## 二、默认优先路径

若项目根存在：

```bash
scripts/story_review_regression.py
```

则优先运行：

```bash
python3 scripts/story_review_regression.py
```

并读取：

- `测试/story-review_回归脚本汇总.md`
- `测试/story-review_回归脚本汇总.json`

先看自动汇总，再决定是否补人工复看与额外落盘核查。

若项目中不存在该脚本，才退回手工按 `story-review/references/review-regression-baseline.md` 逐组跑样本。

补充硬规则：
- 若本轮改动命中以下任一项，手工回归中必须至少追加 `1` 组“完全新题材冷启动样本”，不得只在旧书上复测：
  - 情报流门禁
  - 题面结果化兑现闸门
  - 章节组反重复闸门
  - 失败码阻断顺写
- 冷启动样本必须覆盖：
  - 开书设定
  - 前 `3` 章细纲
  - 前 `3` 章正文
  - 至少 `1` 次 `story-review full`
- 若只跑旧样本、不跑新冷启动样本，不得宣称这轮流程改动“已稳定”

---

## 三、写作 skill 的最小交付要求

若本轮属于“改 skill / 改流程”的工作，而不是单纯写正文，最终回复至少要交代：

- 是否已跑自动回归脚本
- 已覆盖哪些基线组：`A组 / B组 / C组 / 扩展组`
- 是否额外覆盖“新题材冷启动组”
- 是否发现误伤：
  - `warn` 被误写进失败码
  - `仅 warn` 样本误触发复盘单
  - `0 error / 0 warn` 样本在人物层 / 一致性层被硬判问题
  - `有 error` 样本被误放行
  - `scene_lint error` 被直接当成最终裁决，未做人工覆核
  - 子代理与脚本冲突时，主线程没有写出 `人工覆核裁决`

---

## 四、注意事项

- 自动回归脚本只负责汇总，不替代人工判型
- 若脚本结果与人工预期冲突，优先先查：
  - `scene_lint.py` 版本是否一致
  - 样本回归落盘文件是否缺失
  - `story-review` 是否把 `warn` 与 `error` 混判
- 当前回归组口径补充：
  - `A组 / 测试/skill回归_现代悬疑调查` 不再视为“绝对 F0 干净样本”，而是“人工覆核样本”
  - `A组` 的重点不是“任何视角都必须直接 F0”，而是验证主线程会不会在 `0 error / 0 warn` 时乱造人物/一致性问题，以及会不会对章尾分歧人工拍板
  - `C组 / 测试/开局流放，我每天解锁一个情报` 仍是 `F2` 真门禁样本
  - 但 `C组` 当前稳定口径应写成：`第018章人工确认成立；第027章保留人工覆核分歧`
  - 不得再把 `C组` 简化成“脚本 2 条 error 都无争议成立”
