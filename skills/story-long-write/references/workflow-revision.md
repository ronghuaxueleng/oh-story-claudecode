# `workflow-revision.md`：大修怎么走

这份汇总“大修 / 回炉”这条线的走法。SKILL.md 路由到这里后，可按下面推进。

---

## 什么时候走这条

- 用户说"修改第X章""回炉第X章""重写第X章"
- 目标：修改已经写好的章节内容

> 用户得给章节号或章节名，不然这边没法稳妥判断到底该改哪一章。

## 这轮到底该怎么改

进到这条线后，先判断这轮到底该怎么修：

### 1. 小修句子

只有同时满足下面这些条件，才按这一种走：
- 问题只落在单个句子或单个短段
- 失败码主要是 `F1`
- 不牵动收益落点、段落主线、人物前后关系、追踪说法

### 2. 一次把整段翻掉重写

只要碰上以下任一项，就直接切到这一路：
- 同一章连续 2 轮复审后仍保留相同主失败码
- 同一问题同时跨“句子层 + 章节层”两层
- 问题同时涉及“确认链 / 段落主线 / 收益落点 / 人物怎么压住别人”中的 2 项及以上
- 已经出现“局部越改越碎、复审意见越说越细但还没过”

### 3. 这一轮别改着改着又改路数

- 本轮一旦判成“一次性整段重写”，后续同一批次内用户再说“继续”，也都按整段重写处理
- 别一边说“整段重写”，一边又退回到句子补丁式小修
- 若复审后仍没过，先继续整段重写，别退回到零碎小修

## Step 1：定位章节

1. 根据"第X章"找到文件：`正文/第{X}章_*.md`
2. 如果用户说的是章节名而非编号，用 `find 正文/ -name "*{关键词}*"` 搜索
3. 找不到时，让用户确认具体章节
4. 如果用户指定段落范围（"第3-5段"/"那场打斗"/"对话部分"），记录为局部修改目标

### Step 1.5：先确认这次回炉卡在故事哪儿

回炉前，先确认这章所在的故事进度，到底是不是当前这一步：

1. 读取 `追踪/上下文.md`，确认当前默认续写点停在哪一章
2. 读取 `追踪/时间线.md`、`追踪/角色状态.md`，确认它们对第 `{X}` 章前后的描述是否与现有正文一致
3. 如果正文已被大修过，而追踪文件仍保留旧稿人物、旧事件顺序、旧卷收尾说法，先修追踪，再回炉正文
4. 如果后续细纲把旧预案、旧章节走向当成已发生事实，先标记为续写点污染，别直接沿该细纲重写

**这里按这几条认**：
- 以当前已经存在的真实正文为准
- 旧大纲、旧细纲、旧追踪，如果和真实正文冲突，就先按真实正文来
- 世界线没对齐前，回炉很容易越修越乱

### Step 1.5.5：先定这轮怎么改，再动正文

正式加载正文前，先把这轮准备怎么修先写明白：

- 本轮回修方式：`小修句子 / 一次性整段重写`
- 触发依据：
  - 单一问题
  - 连续复发
  - 跨层问题
  - 复合问题

先记这几句：
- 这轮准备怎么修还没写出来，就先别进 Step 2
- 若已判成“一次性整段重写”，别再把修改范围写成“局部润色/顺手修一下”

### Step 1.6：连续回炉先立一张这轮到底要改到哪的单子

如果本次回炉满足任一条件：
- 涉及连续 `3` 章及以上
- 用户已连续多轮反馈“怎么还没修干净 / 为什么改了这么多遍”
- 这次看下来，问题不止句子层，而是章节层或整层问题

那就在进正文修改前，先生成一份单子，存进 `追踪/检查单_*.md`，最少答清：

1. 本批次处理范围
2. 改到什么样才算这一轮收住
3. 章级硬拦截
4. 连续章节硬拦截
5. 这一轮总共怎么判
6. 章节分级：
   - 局部改
   - 中等重修并补分量
   - 重写级扩写
7. 固定修改顺序
8. 最后只回答的检查问题

这张单子没立出来前：
- 先别直接改正文
- 别边看边补
- 别说“先修一点再看”

若本轮已判定为“一次性整段重写”：
- 单子里得写明 `回修模式：一次性整段重写`
- 同时答清：
  - 重写段范围
  - 本轮要一次解决的复合问题项
  - 禁止回退到补丁式小修

### Step 1.7：开工前先把顺序排出来

如果这次回炉要照 `总安排表 / 修改表 / 检查单` 往下走，那进正文前还得再补一份执行计划，存进 `追踪/执行计划_*.md`，把修改顺序写明白，免得改着改着乱套。

执行计划最少答清：

1. 所属修改表 / 检查单
2. 本批次处理范围
3. 固定修改顺序
4. 每项状态：
   - `待处理`
   - `进行中`
   - `已完成`
   - `阻塞`
5. 当前正在做的唯一事项
6. 这一批改到什么样才算收住

先按这几句来：
- 没生成执行计划前，先别直接改正文
- 执行计划一旦生成，就按这张表的顺序往下推
- 若发现修改顺序要变，先更新修改表/检查单，再更新执行计划
- 别只在对话里说“继续下一项”，文件里却一点状态变化都没有
- 若当前模式为“一次性整段重写”，执行计划要把“重写问题段主线”单列成第一项，别把句子润色排在前面

可以直接按这个格式记：

```markdown
# 执行计划｜{批次名称}

来源单子：
- `{修改单文件名}`

处理范围：
- `{章节/文件范围}`

完成定义：
- `{什么情况下本批次才算完成}`

## 状态表

1. [ ] {事项1}（状态：进行中）
2. [ ] {事项2}（状态：待处理）
3. [ ] {事项3}（状态：待处理）

## 执行记录

- {日期}：创建计划，启动 `{事项1}`
```

## Step 2：把上下文补齐

这里得比平时写作多读一点上下文，因为修改时要看前后怎么接：

| 序号 | 文件 | 用途 | 如果不存在 |
|------|------|------|-----------|
| 1 | `正文/第{X}章_*.md` | 待修改章节 | 得存在 |
| 2 | `大纲/细纲_第{X}章.md` | 原始写作计划 | 跳过 |
| 3 | `正文/第{X-1}章_*.md` | 前一章（衔接） | 第1章时加载 `设定/世界观/背景设定.md` 替代 |
| 4 | `正文/第{X+1}章_*.md` | 后一章（衔接） | 末章时检查 `大纲/细纲_第{X+1}章.md`（如有）确保连续性 |
| 5 | `追踪/上下文.md` | 当前默认续写点与进度 | 跳过 |
| 6 | `追踪/时间线.md` | 本章前后事件顺序 | 跳过 |
| 7 | `追踪/角色状态.md` | 当前人物状态说法 | 跳过 |
| 8 | `追踪/伏笔.md` | 涉及的伏笔 | 跳过 |
| 9 | `追踪/情报记录.md` | 系统/情报/任务类规则记录 | 跳过 |
| 10 | `设定/角色/{相关角色}.md` | 本章涉及角色 | 跳过 |

**相关角色判定**：从 `大纲/细纲_第{X}章.md` 中提取角色名。如果细纲不存在，从待修改章节正文中搜索 `设定/角色/` 目录下的角色名关键词。将角色名映射为文件名：`设定/角色/{角色名}.md`。

## Step 3：修改

1. **阅读原文**：读完整章内容，记下原始字数
2. **备份原文**：将原文复制为 `正文/第{X}章_章名_原稿_{YYYYMMDD}.md`，把回退口留好
3. **先定这次到底改多大**：先根据用户本轮指令、修改表、检查单、执行计划自动判断，别把本来该流程自己就该判断的事又丢回给用户
   - 用户明确说"全文重写" → 全文重写
   - 用户明确点名段落/场面/问题类型 → 局部修改
   - 碰上章节层或整层问题 → 一般别做“局部润色式修补”，按章节重修或连续回炉处理
   - 已有修改表/检查单/执行计划时，修改范围就按文件来，别临时换说法
4. **先看问题到底卡在哪一层**：修改前先判断问题属于哪一层：
   - 句子层：AI 腔、重复句式、解释句、下结论句
   - 章节层：场面不足、节奏拖、战果弱、章尾弱
   - 整层问题：细纲像讲解稿、连续慢节奏、信息推进代替情绪推进
   如果看下来是整层问题，就别只做局部润色；直接写明“本章单修大概率不够，得回到连续章节回炉或细纲回炉”
   - 如果这次回炉涉及连续 3 章及以上，额外先写出：
     - 每章战果是什么
     - 哪一章是整组最猛的一下
     - 是否有连续两章都只查不当章兑现
     任一项答不清，就先回细纲层，别直接写正文
   - 如果这次回炉已经生成检查单，还得顺手把每章归进：
     - 局部改
     - 中等重修并补分量
     - 重写级扩写
     这一步没分完级，就先别开写
   - 额外强制诊断：
     - 是否存在“书名主打的主要推进已经失频，副线反客为主”
     - 是否存在“人物出场分量只补首轮，连续登场又退回只剩办事作用”
     - 是否存在“主角组/关键配角删去名字后台词分不出是谁”
     - 是否存在“追踪文件已写完成，正文实际未稳定落地”的假完成
     - 是否存在“时间线/角色状态/情报记录”双版本并存
     任一项碰上且跨 2 章及以上，就别当成单章小修收尾
   - 如果当前已经判成“一次性整段重写”，还得额外写清：
     - 旧主线的问题段范围
     - 本轮要一次解决的复合问题项
     - 本轮保留的旧事实与重写后的新事实
5. **正式下手改**：改写文件
6. **字数对比**：修改后与原文字数差异 > 30% 或 > 800 字时提醒用户（取较大值）

如果这次已经有执行计划：
- 开始做某一项前，先把该项标为 `进行中`
- 完成该项后，立刻在执行计划中：
   - 把该项改为 `已完成`
   - 若还有下一项，就把下一项改为 `进行中`
  - 在“执行记录”里补一条完成说明
- 别做完多项后再一次性补写状态

**资料研究（按需）**：如果这次修改碰到需要查清的外部事实（历史年代、地理方位、职业细节等），就拉 `story-researcher` 子代理去查。

### Step 3.5：整段重写时就按这条走

如果这轮已经判成“一次性整段重写”，那正文修改就按下面这个顺序走：

1. 先看清问题段原来是怎么跑歪的：
   - 前后是怎么串起来的
   - 收益落点落在哪
   - 人物压住别人的那一下靠谁撑
   - 哪些拍子其实只是把事情往前推
2. 再重组新的写法，只保留必要记忆点：
   - 最猛的那一下
   - 二段收益
   - 人物失态 / 往前往后挪 / 改规矩
   - 章尾钩子承接
3. 先重写整段，再做少量句子修边
4. 别按下面这个顺序改：
   - 先抠句子
   - 先替换台词
   - 先删 AI 味词
   - 先做局部顺句
5. 若重写后仍没过，就继续整段重排，别回退成“逐句小补丁”

## Step 4：看有没有带出后手问题

改完以后，再一项项看有没有连带影响到后面的章节：

1. **伏笔检查**：对比修改前后的伏笔列表（读取旧记录 vs 新内容），把变动项挑出来：
   - 新增伏笔 → 添加到 `追踪/伏笔.md`
   - 删除伏笔 → 从 `追踪/伏笔.md` 移除，并检查后续章节是否引用该伏笔
   - 修改伏笔 → 更新描述，检查后续引用是否一致
2. **时间线检查**：对比修改前后的事件顺序，更新 `追踪/时间线.md`
3. **后续影响**：如果修改动到了角色状态、关系，或者世界观设定，就去扫后续章节正文，把受影响的地方标出来：

```
⚠️ 修改第{X}章后，下面这些章节也得跟着回头看：
- 第{X+1}章：{原因}（状态：待回看 / 进行中 / 已完成）
- 第{X+3}章：{原因}（状态：待回看 / 进行中 / 已完成）
```

4. **禁用词扫描**：对照 `references/banned-words.md` 检查修改后的内容
5. **追踪有没有跟上**：本章回炉后，把这些地方顺手一起看掉，该改就改：
   - `追踪/上下文.md`
   - `追踪/时间线.md`
   - `追踪/角色状态.md`
   - `追踪/伏笔.md`
   - `追踪/情报记录.md`（如存在）
   - `设定/角色/{角色名}.md`（凡本章新增或修正了稳定外形/动作/说话/气场记忆点，都得同步）
   正文已经改了，追踪却还留着旧说法，那就说明这次回炉还没收住
   - 追踪如果把“待补、准备补、理论成立”写成“已完成 / 已补强 / 已落地”，也说明回炉还没收住
   - `时间线 / 角色状态 / 情报记录` 里同一件事如果并着两版，也说明回炉还没收住
6. **把那张单子同步回去**（只在连续回炉时）：
   - 记录已完成章节
   - 记录剩余章节
   - 若修改中发现原分级失真，只能在那张单子里整体重判，别口头临时漂移
7. **把执行计划同步回去**（如果有）：
   - 同步当前项状态
   - 记录剩余项
   - 若出现阻塞，就把阻塞原因和解除条件写明白

---

## Step 5：最后再过一遍

对改完后的章节，再走一遍第 5 阶段检查，最少看下面这些：

1. **禁用词扫描**：如 Step 4 未覆盖全章，再次扫描
2. **人物一致性**：修改后的角色行为是否与角色设定一致
3. **节奏检查**：修改是否破坏了章节节奏
4. **这次回炉到底补到没有**：
   - 这章现在是更好看了，还是只是更顺了？
   - 有没有真正补出爽点，而不只是删掉毛病？
   - 这类问题后面还会不会继续长出来？
   如果第三条答案还是“会”，那就直接回报这是整层问题，别装成本章已经解决
   - 重要角色这次是否真正补出了稳定厚度，而不是只把事情写顺
   - 主角组/关键配角删去名字后，台词与动作是否仍可分辨
   这里面只要有一项没过，本章就先别算过
5. **残留解释腔扫描**：
   - 额外检查 `这不是 / 这一下 / 他知道 / 原来 / 从这一刻起`
   - 若在修改后仍形成连续复用，判定本章还没过，继续回修
6. **连续章节这几章到底有没有真落战果**（只在连续 3 章及以上回炉时）：
   - 每章是否都有外显战果
   - 最猛的那一下是否真的立住
   - 是否还存在“两章连续只查不当章兑现”
   只要还在，就要直接回报这是整层残留，别说回炉已经做完
7. **世界线再对一遍**：
   - `上下文/时间线/角色状态/伏笔/情报记录` 是否都和回炉后正文同说法
   - 是否还残留旧稿人物名、旧事件顺序、旧任务线、旧卷收尾描述
   - 后续细纲是否仍在沿用被本次回炉推翻的旧事实
   这些里面只要有一项还在，就继续修追踪，或者把受影响章节标出来，别说“已经清干净”
   - 若存在“假完成状态”或“双版本并存”，就按 `不可继续顺写` 看
8. **别又拿补丁式说法收尾**：
   - 若本次属于连续回炉，完成后得回答：
     - 当前剩的是句子噪音、分量问题，还是仍有整层毛病
     - 是否仍需要重写级处理
   - 别用“已经更好了”“接近能过了”这类模糊话代替判断
   - 若本轮模式是“一次性整段重写”，别在结案时把同一问题降格成“后面顺手再抠两句”
9. **整组最后只判一次**：
   - 同一份检查单批次内，别每修一章就重新发起整组结论
   - 要等该批次既定顺序都跑完，再给整组“能不能收进主正文”的判断
10. **计划到底有没有真的走完**：
   - 执行计划里只要还挂着 `待处理 / 进行中 / 阻塞`，就别说这一批已经做完
   - 嘴上说“按修改单修完了”，执行计划却没同步，这条线就还没收住
11. **最后怎么收口**：
   - 只要还残留章节层、整层、假完成追踪、双版本并存、人物出场分量失败、台词边界失败里的任一项，就别说“清干净”
   - 只有当剩余问题明确只剩句子噪音时，才允许判“可收进主正文”
   - 若本批次处理范围来自修改表/检查单，那就得同时满足“正文算过 + 追踪同步 + 计划勾销”这三项，才能写“本批次完成”


> 更完整的第 5 阶段检查清单，看 SKILL.md 对应的第 5 阶段部分。

## 常见问题

| 问题 | 处理 |
|------|------|
| 用户没说改哪里 | 问"你想改哪一章？哪方面？情节/节奏/对话/描写？" |
| 修改后字数暴增/暴减 | 提醒用户，由用户决定是否调整 |
| 连续改多章 | 先生成检查单，再按定好的顺序一章章改；别边改边换判法 |
| 改完发现后续不一致 | 列出受影响章节，由用户决定是否现在修改 |
