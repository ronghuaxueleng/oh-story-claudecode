# `workflow-revision.md`：大修工作流

本文件为"大修/回炉"场景的完整指引。SKILL.md 路由到本文件后，按以下流程执行。

---

## 适用条件

- 用户说"修改第X章""回炉第X章""重写第X章"
- 目标：修改已写好的章节内容

> 用户必须指定章节号或章节名。无法自动推断需要修改哪一章。

---

## Step 1：定位章节

1. 根据"第X章"找到文件：`正文/第{X}章_*.md`
2. 如果用户说的是章节名而非编号，用 `find 正文/ -name "*{关键词}*"` 搜索
3. 找不到时让用户确认具体章节
4. 如果用户指定段落范围（"第3-5段"/"那场打斗"/"对话部分"），记录为局部修改目标

### Step 1.5：先确认本次回炉的真实锚点

回炉前，先确认这章所处世界线是不是当前真实世界线：

1. 读取 `追踪/上下文.md`，确认当前默认锚点停在哪一章
2. 读取 `追踪/时间线.md`、`追踪/角色状态.md`，确认它们对第 `{X}` 章前后的描述是否与现有正文一致
3. 如果正文已被大修过，而追踪文件仍保留旧稿人物、旧事件顺序、旧卷收口口径，先修追踪，再回炉正文
4. 如果后续细纲把旧预案、旧章节走向当成已发生事实，先标记为锚点污染，不得直接沿该细纲重写

**原则**：
- 以当前已存在的真实正文为准
- 旧大纲、旧细纲、旧追踪，如果和真实正文冲突，必须先让它们退位
- 世界线没校准前，回炉很容易“越修越乱”

### Step 1.6：连续回炉先生成验收单

如果本次回炉满足任一条件：
- 涉及连续 `3` 章及以上
- 用户已连续多轮反馈“怎么还没修干净 / 为什么改了这么多遍”
- 当前判断问题不止句子层，而是章节层或机制层

则在进入正文修改前，必须先生成一份验收单，存入 `追踪/验收单_*.md`，至少包含：

1. 本批次处理范围
2. 过线标准
3. 章级硬闸
4. 连续章节硬闸
5. 当前总裁决
6. 章节分级：
   - 局改
   - 中修补厚
   - 重写级重扩
7. 固定施工顺序
8. 最终只回答的验收问题

未生成验收单前：
- 不得直接改正文
- 不得边看边补
- 不得宣称“先修一点再看”

### Step 1.7：开工前先生成执行计划

如果本次回炉将按 `总施工单 / 施工单 / 验收单` 执行，则在进入正文修改前，必须再生成一份执行计划，存入 `追踪/执行计划_*.md`，用于把施工顺序落成可勾选状态表。

执行计划至少包含：

1. 所属施工单 / 验收单
2. 本批次处理范围
3. 固定施工顺序
4. 每项状态：
   - `待处理`
   - `进行中`
   - `已完成`
   - `阻塞`
5. 当前进行中的唯一事项
6. 本批次完成定义

硬规则：
- 未生成执行计划前，不得直接改正文
- 执行计划一旦生成，必须按计划顺序推进
- 若发现施工顺序要变，先更新施工单/验收单，再更新执行计划
- 不允许只在对话里说“继续下一项”，文件里没有状态变化

推荐模板：

```markdown
# 执行计划｜{批次名称}

来源工单：
- `{工单文件名}`

处理范围：
- `{章节/文件范围}`

完成定义：
- `{什么情况下本批次才算完成}`

## 状态表

1. [ ] {事项1}（状态：进行中）
2. [ ] {事项2}（状态：待处理）
3. [ ] {事项3}（状态：待处理）

## 执行记录

- {日期}：创建计划，启动 `{事项1}`
```

## Step 2：加载上下文

加载比日常写作更多的上下文（因为修改需要理解前后衔接）：

| 序号 | 文件 | 用途 | 如果不存在 |
|------|------|------|-----------|
| 1 | `正文/第{X}章_*.md` | 待修改章节 | 必须存在 |
| 2 | `大纲/细纲_第{X}章.md` | 原始写作计划 | 跳过 |
| 3 | `正文/第{X-1}章_*.md` | 前一章（衔接） | 第1章时加载 `设定/世界观/背景设定.md` 替代 |
| 4 | `正文/第{X+1}章_*.md` | 后一章（衔接） | 末章时检查 `大纲/细纲_第{X+1}章.md`（如有）确保连续性 |
| 5 | `追踪/上下文.md` | 当前默认锚点与进度 | 跳过 |
| 6 | `追踪/时间线.md` | 本章前后事件顺序 | 跳过 |
| 7 | `追踪/角色状态.md` | 当前人物状态口径 | 跳过 |
| 8 | `追踪/伏笔.md` | 涉及的伏笔 | 跳过 |
| 9 | `追踪/情报台账.md` | 系统/情报/任务类规则台账 | 跳过 |
| 10 | `设定/角色/{相关角色}.md` | 本章涉及角色 | 跳过 |

**相关角色判定**：从 `大纲/细纲_第{X}章.md` 中提取角色名。如果细纲不存在，从待修改章节正文中搜索 `设定/角色/` 目录下的角色名关键词。将角色名映射为文件名：`设定/角色/{角色名}.md`。

## Step 3：修改

1. **阅读原文**：读完整章内容，记录原始字数
2. **备份原文**：将原文复制为 `正文/第{X}章_章名_原稿_{YYYYMMDD}.md`，确保可回退
3. **确认修改范围**：优先根据用户本轮指令、施工单、验收单、执行计划自动判定，不得把本应由流程判断的事重新丢回给用户
   - 用户明确说"全文重写" → 全文重写
   - 用户明确点名段落/场面/问题类型 → 局部修改
   - 命中章节层或机制层问题 → 默认不做“局部润色式修补”，按章节重修或连续回炉处理
   - 已有施工单/验收单/执行计划时，修改范围必须服从文件，不得临时改口
4. **先做分层诊断**：修改前必须先判断问题属于哪一层：
   - 句子层：AI 腔、重复句式、解释句、盖章句
   - 章节层：场面不足、节奏拖、战果弱、章尾弱
   - 机制层：细纲像说明书、连续慢节奏、信息推进代替情绪推进
   如果判断为机制层问题，不要只做局部润色，必须明确提示“本章单修可能无效，需要连续章节回炉或细纲回炉”
   - 如果这次回炉涉及连续 3 章及以上，额外先写出：
     - 每章战果是什么
     - 哪一章是主爆点
     - 是否有连续两章都只查不当章兑现
     任一项答不清，先回细纲层，不直接写正文
   - 如果这次回炉已生成验收单，必须额外把每章归入：
     - 局改
     - 中修补厚
     - 重写级重扩
     未完成分级，不得开写
   - 额外强制诊断：
     - 是否存在“题面主驱动已经失频，副驱动反客为主”
     - 是否存在“人物显影只补首轮，连续登场又退回功能位”
     - 是否存在“主角组/核心配角删去名字后台词分不出是谁”
     - 是否存在“追踪文件已写完成，正文实际未稳定落地”的假完成
     - 是否存在“时间线/角色状态/情报台账”双版本并存
     任一项命中且跨 2 章及以上，不得按单章局改结案
5. **执行修改**：改写文件
6. **字数对比**：修改后与原文字数差异 > 30% 或 > 800 字时提醒用户（取较大值）

如果本次存在执行计划：
- 开始做某一项前，先把该项标为 `进行中`
- 完成该项后，立刻在执行计划中：
  - 将该项改为 `已完成`
  - 若还有下一项，将下一项改为 `进行中`
  - 在“执行记录”追加一条完成说明
- 禁止做完多项后再一次性回填状态

**资料研究（按需）**：如果修改涉及需要验证的外部事实（历史年代、地理方位、职业细节等），spawn `story-researcher` 子代理搜索验证。

---

## Step 4：级联检查

修改完成后，逐一检查是否影响后续章节：

1. **伏笔检查**：对比修改前后的伏笔列表（读取旧快照 vs 新内容），标记变化项：
   - 新增伏笔 → 添加到 `追踪/伏笔.md`
   - 删除伏笔 → 从 `追踪/伏笔.md` 移除，并检查后续章节是否引用该伏笔
   - 修改伏笔 → 更新描述，检查后续引用是否一致
2. **时间线检查**：对比修改前后的事件顺序，更新 `追踪/时间线.md`
3. **后续影响**：如果修改改变了角色状态/关系/世界观设定，扫描后续章节正文标记受影响项：

```
⚠️ 修改第{X}章后，以下章节必须同步复核：
- 第{X+1}章：{原因}（状态：待复核 / 进行中 / 已完成）
- 第{X+3}章：{原因}（状态：待复核 / 进行中 / 已完成）
```

4. **禁用词扫描**：对照 `references/banned-words.md` 检查修改后的内容
5. **追踪同步检查**：本章回炉后，必须同步检查并按需更新：
   - `追踪/上下文.md`
   - `追踪/时间线.md`
   - `追踪/角色状态.md`
   - `追踪/伏笔.md`
   - `追踪/情报台账.md`（如存在）
   - `设定/角色/{角色名}.md`（凡本章新增或修正了稳定外形/动作/说话/气场锚点，必须同步）
   如果正文已经改了，但追踪还保留旧口径，视为回炉未完成
   - 如果追踪把“待补、准备补、理论成立”写成“已完成/已补强/已落地”，视为回炉未完成
   - 如果 `时间线/角色状态/情报台账` 同一事项存在双版本并存，视为回炉未完成
6. **验收单回写**（仅连续回炉时）：
   - 记录已完成章节
   - 记录剩余章节
   - 若修改中发现原分级失真，只能在验收单中整体改判，不得口头临时漂移
7. **执行计划回写**（若存在）：
   - 同步当前项状态
   - 记录剩余项
   - 若出现阻塞，明确写出阻塞原因和解除条件

---

## Step 5：质量检查

对修改后的章节执行 Phase 5 质量检查（至少包含）：

1. **禁用词扫描**：如 Step 4 未覆盖全章，再次扫描
2. **人物一致性**：修改后的角色行为是否与角色设定一致
3. **节奏检查**：修改是否破坏了章节节奏
4. **返工结果检查**：
   - 这章现在是更好看了，还是只是更顺了？
   - 有没有真正补出爽点，而不只是删掉毛病？
   - 这类问题后面还会不会继续长出来？
   如果第三条答案是“会”，必须回报这是机制问题，不应伪装成“本章已解决”
   - 重要角色这次是否真正补出了稳定显影，而不是只把功能写顺
   - 主角组/核心配角删去名字后，台词与动作是否仍可分辨
   若任一项失败，本章不得判通过
5. **残留解释腔扫描**：
   - 额外检查 `这不是 / 这一下 / 他知道 / 原来 / 从这一刻起`
   - 若在修改后仍形成连续复用，判定本章未过线，继续回修
6. **连续章节兑现密度复核**（仅连续 3 章及以上回炉时）：
   - 每章是否都有外显战果
   - 主爆点是否足够硬
   - 是否还存在“两章连续只查不当章兑现”
   若存在，必须回报为机制层残留，不得宣称回炉完成
7. **世界线一致性复核**：
   - `上下文/时间线/角色状态/伏笔/情报台账` 是否都和回炉后正文同口径
   - 是否还残留旧稿人物名、旧事件顺序、旧任务线、旧卷收口描述
   - 后续细纲是否仍在沿用被本次回炉推翻的旧事实
   若任一项存在，必须继续修追踪或标记受影响章节，不得宣称“已经清干净”
   - 若存在“假完成状态”或“双版本并存”，默认 `不可继续顺写`
8. **禁止补丁式结案**：
   - 若本次属于连续回炉，完成后必须回答：
     - 当前剩的是句子噪音、厚度问题，还是仍有机制病
     - 是否仍需要重写级处理
   - 不允许用“已经更好了”“接近过线了”这类模糊表述代替裁决
9. **最终只做一次连续裁决**：
   - 同一份验收单批次内，不允许每修一章就重新发起整组结论
   - 必须在该批次既定顺序完成后，再给整组 `可并入 / 不可并入` 裁决
10. **计划完成校验**：
   - 若执行计划中仍有 `待处理 / 进行中 / 阻塞` 项，不得宣称本批次施工完成
   - 若口头称“按工单修完”，但执行计划未同步，视为流程未完成
11. **最终裁决硬规则**：
   - 只要仍残留章节层、机制层、假完成追踪、双版本并存、人物显影失败、台词边界失败中的任一项，就不得宣称“清干净”
   - 只有当剩余问题明确收束为句子噪音时，才允许判 `可并入主正文`
   - 若本批次处理范围来自施工单/验收单，则必须同时满足“正文过线 + 追踪同步 + 计划勾销”三项，才允许判“本批次完成”

## 常见问题

| 问题 | 处理 |
|------|------|
| 用户没说改哪里 | 问"你想改哪一章？哪方面？情节/节奏/对话/描写？" |
| 修改后字数暴增/暴减 | 提醒用户，由用户决定是否调整 |
| 连续改多章 | 先生成验收单，再按固定顺序逐章施工；不得边修边改判 |
| 改完发现后续不一致 | 列出受影响章节，由用户决定是否现在修改 |
