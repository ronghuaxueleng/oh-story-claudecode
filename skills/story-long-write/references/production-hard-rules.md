# 长篇生产硬规则

用于给 `story-long-write` 增加“执行闸门”，防止长篇在大纲、细纲、正文生产过程中自然滑向慢节奏、解释腔、低兑现。

---

## 零、执行性质

这不是参考建议。

这是生产规则。

命中硬规则失败项时：

1. 不得进入下一步
- 细纲不过线，不得写正文
- 正文不过线，不得并入主正文
- 追踪/角色卡未同步，视为本次工作未完成

2. 不得靠“先写下去再说”绕过
- 命中连续慢节奏、解释腔回潮、人物功能化、连续两章只查不当章兑现，必须先回炉
- 不允许靠多写几章把问题冲过去

3. 不得用局部修句伪装机制修复
- 句子层问题才允许局改
- 章节层、机制层问题，必须回炉整章、连续章节或细纲

4. 不得把“基本能看”当成过线
- 过线标准不是“逻辑顺”
- 是“能发、能留、能让读者继续点下一章”

5. 写作默认带验收
- 每次写章、改章、回炉，都必须执行写前问答、写后问答、连续章节复核
- 任一硬闸失败，本次工作直接判未完成
6. 连续回炉默认先钉验收单
- 若一次回炉涉及连续 3 章及以上，或用户已明确抱怨“反复改很多轮还没清干净”，必须先生成固定验收单
- 验收单未生成前，禁止直接改正文
- 验收单一旦生成，本批次必须按单施工，不得中途跳章、换级、改优先级

7. 正常生产顺序必须固定
- 正常写作默认顺序是：核心设定 -> 卷纲 -> 细纲 -> 细纲过线 -> 正文 -> 验收 -> 追踪同步
- 不允许把正文提到细纲之前
- 也不允许把“正文写完后再补细纲/追踪/角色卡”当成常规工作法

---

## 一、总裁决口径

本文件不解释“为什么这样写更好”，只定义“什么情况直接不过线”。

1. 留存优先于解释完整
- 只要正文主要在补背景、翻译局势、说明风险，而不是让现实继续发生，默认不过线

2. 信息推进不能冒充情绪推进
- 若一章答不出“谁赢了、谁亏了、谁被踩了、情绪是否抬升”，默认不过线
- “知道更多”不等于“爽点成立”

3. 场面不足按章节失败处理
- 若正文长期只有说局势、说风险、说后果、说下一步，而缺少动作、阻力、失态、代价、盘面变化，默认至少 `F2`
- 若读感像提纲扩写、功能链顺推、线索中转站，默认至少 `F2`

4. 角色功能化按质量失败处理
- 重要角色若只负责递信息、翻译局势、关键节点顶剧情，而没有显影锚点，默认不过线
- 高价值角色若不能被脑补，默认不过线
- 连续登场后只剩功能、不剩人味，默认回炉补显影

5. 章节太薄按回炉处理
- 若一章主交付只是“推进到下一步”，却没有改掉任何一口现实盘面，默认章肉不足
- 若动作、外部反应、对手失态、代价、站位变化、规则变化里几乎都缺，默认至少 `F2`

6. 连续“更接近了”按机制病处理
- 连续两章主收益都是“更接近真相 / 更接近幕后 / 更接近下一步”，默认至少 `F3`

7. 题面滑轨按机制病处理
- 若主驱动持续让位给副驱动，默认至少 `F3`
- 若书名 / 文案 / 金手指 / 复仇 / 身份 / 流放处境只能靠总结句证明还在，默认至少 `F3`

8. 情报卖点必须限时兑现
- 情报若只负责解释更深、指向下一站、证明这段也在推进，默认不过线
- 情报落下后，`1-2` 章内必须转成现实收益；否则默认至少 `F3`

## 二、跨阶段裁决

以下规则跨越细纲、正文、回炉三个阶段，优先级高于单文件局部说明。

1. 细纲准入细则统一读取：
- `references/gates/outline-gates.md`

2. 正文执行清单统一读取：
- `references/gates/draft-gates.md`

3. 正文句子与段落红线统一读取：
- `references/writing-red-lines.md`

4. 追踪同步与未来预案禁区统一读取：
- `references/gates/tracking-gates.md`

5. 只要以下任一项未完成，就不得并入主正文：
- 章级验收未通过
- 连续性验收未通过
- 追踪同步未完成
- 角色卡同步未完成
- 若当前属于 review / 回炉批次，则 review 结论未放行

6. review 裁决口径：
- `APPROVE` 或“仅句子层噪音，可局改保留”才算可并入
- `CONCERNS` 命中章节层 / 机制层问题时，不可并入
- `REJECT` 直接退回回炉

## 三、连续章节与机制层报警

1. 连续慢节奏报警
- 连续两章只靠查信息推进：报警
- 连续三章无外显战果：报警
- 连续两章只有信息升级，没有情绪升级：报警
- 连续两章主要战果都是“拿到下一个抓手”：报警
- 连续两章章尾都更像“准备去下一章做事”：报警
- 连续两章读感像“高质量提纲扩写”：报警

2. 报警后禁止顺写
- 不能靠“继续多写几章”把问题冲过去
- 必须回炉：
  - 细纲
  - 连续章节
  - 或当前章的兑现结构
- 回炉时必须优先补的不是解释，不是背景，不是总结，而是：
  - 动作
  - 阻力
  - 对手反应
  - 外部反应
  - 代价
  - 站队变化
  - 关系 / 规则 / 资源变化

3. 题面滑轨报警
- 命中以下任一情况，直接禁止顺写：
  - 连续 `3` 章主收益都不在题面上
  - 连续 `2-3` 章情报没有兑现成硬战果
  - 题面身份 / 核心处境只剩名义，不再持续带来限制、羞辱、风险、资源门槛或反压收益
  - 复仇线只剩“以后再查”，当章不再交付撕口收益
  - 书名卖点只能靠追踪文件或作者总结来证明还在
- 这类问题一律判机制病，必须回到细纲层重做章节组

4. 连续章节回炉额外复核
- 如果一次回炉涉及连续 3 章及以上，必须额外回答：
  - 每章最值钱的战果是什么
  - 哪一章是整组主爆点
  - 是否仍有连续两章只查不当章兑现
- 任一条答不清，默认这次回炉还没过线

5. 连续章节题面复核
- 如果一次回炉涉及连续 3 章及以上，必须再额外回答：
  - 这组的主驱动到底是哪一项题面卖点
  - 这组结束后，主角在题面层到底赢到了什么
  - 这组有没有让副驱动反客为主
- 任一条答不清，默认这组仍在滑轨
- 如果答案只能落在“更接近真相/更接近下一站/更接近幕后”，默认未过线

6. 角色显影连续复核
- 如果一次回炉涉及核心人物连续出场段，必须额外回答：
  - 这几章里读者能记住他/她什么
  - 说话方式和动作习惯是否稳定
  - 是否仍然只剩功能，没有魅力、压迫感或可想象空间
- 任一条答不清，默认人物还没立住

7. 章节厚度复核
- 回炉或新写完成后，必须额外回答：
  - 这章像“正式正文”，还是像“提纲扩写”
  - 这章除了主推进，还有哪两层以上的肉
  - 本章哪一方被当场改写了处境
  - 本章改掉了哪一口现实
- 如果回答只能落在“发现更多”“更接近了”“下一步更清楚了”，默认未过线

8. 连续两章兑现不足即整组回炉
- 若连续两章主战果都只是：
  - 拿到下一条线
  - 更接近真相
  - 逼出下一步地点
  - 发现更高层影子
- 则不允许只修后一章
- 必须把这两章作为一个回炉单元重做

9. 禁止补丁式回炉
- 如果问题已经被判断为章节层或机制层：
  - 不得采用“先局改一点再看”的循环
  - 不得每修完一章就重新定义整组问题
  - 必须先做全量验收，再分成：
    - 局改
    - 中修补厚
    - 重写级重扩
- 未分级完成，不得开写

10. 连续回炉必须给出刚性裁决
- 每个回炉批次结束时，必须明确回答：
  - 当前剩下的是句子噪音、厚度问题，还是机制病
  - 是否可并入主正文
- 禁止使用以下模糊结论替代裁决：
  - “已经好多了”
  - “更接近过线了”
  - “基本可以了”
  - “再看一轮”

## 四、返工分层机制

发现问题后，先判断是哪一层：

1. 句子层
- AI 腔
- 重复句式
- 作者解释句
- 章尾盖章句

处理：局部改写

2. 章节层
- 场面不足
- 节奏拖
- 战果弱
- 章尾弱
- 提纲扩写感
- 章肉偏薄
- 连续几章都只是在“更接近了”

处理：回炉整章

3. 机制层
- 细纲像说明书
- 连续慢节奏
- 信息推进代替情绪推进
- 爽点分布失衡
- 角色长期只有功能，没有显影度

处理：回炉连续章节或回 Phase 3 修细纲

原则：
- 机制层问题，禁止伪装成“改几句就好了”

## 五、连续回炉问答

连续回炉或整组验收时，至少回答：

1. 这一章最硬的现场是什么？
2. 这一章最值钱的战果是什么？
3. 这一章谁压谁，谁翻盘？
4. 这一章如果只有信息没有兑现，会不会拖？
5. 章尾靠什么把人拽进下一章？
6. 本章最重要的人，读者能记住他/她什么脸、什么动作、什么口气？
7. 本章谁会先失态、先丢脸、先被逼站队？
8. 本章如果写完只剩“拿到下一条线”，是不是立刻判薄？

任一条答不上来，默认这组未过线。

---

## 六、强制验收输出格式

每次新写/回炉完成后，主线程必须产出一份简短验收，不得跳过。格式固定：

### 章级验收
- 本章最硬现场：
- 本章最值钱战果：
- 本章谁被打疼：
- 本章改掉的现实盘面：
- 本章关键角色显影：
- 是否命中硬规则失败项：否/是（若是，停止并回炉）

### 连续章节验收（涉及多章时必填）
- 本组主爆点章节：
- 本组是否存在重复推进：
- 本组是否存在连续两章只查不当章兑现：
- 若存在，停止并回炉哪几章：

### 并入结论
- 可并入主正文 / 不可并入主正文

没有这份验收，默认本次工作未完成。
