# Story Consistency Rules

修改设定、大纲、追踪文件时必须遵守的一致性补充规则。

## 优先级

- 项目根 `CLAUDE.md`、`写作执行铁律.md`、`story-long-write` 的追踪主表边界、脚本门禁、失败码规则高于本文件。
- 本文件只补“改文件时应顺手核对什么”，不得替代 `validate_tracking_state.py`、`verdict_conflict_lint.py`、`script_version_check.py`。

## Rules

1. **设定变更必须检查追踪/**：修改任何角色设定或世界规则时，必须检查追踪/目录下的伏笔.md 和时间线文件，确保变更不会导致已有伏笔矛盾。
2. **新增角色必须创建独立文件**：每个主要角色应有独立的设定文件，并更新关系.md（如果存在）。
3. **时间线变更必须同步更新**：如果修改涉及时间顺序，必须同步更新时间线.md。
4. **设定变更必须记录到上下文.md**：任何重要的设定决策必须记录到追踪/上下文.md 的最近决策部分。
5. **一致性检查必须用 `rg`**：不靠记忆，用搜索验证。常用检查命令：
   - 角色属性：`rg -n "角色名" 设定/ 正文/`
   - 时间线：`rg -n "第.*天|.*日后|过了.*时" 正文/`
   - 力量体系：`rg -n "体系关键词" 设定/ 正文/`
   - 伏笔状态：`rg -n "伏笔#" 追踪/`
6. **水字数检测信号**：全章对话无新信息、同一情绪连续 3 段+、场景描写超 500 字不推进、角色回忆旧事无新视角、连续 2 章无冲突。
7. **追踪主表只记已发生事实**：未来章节预案、远期收益、未落正文的人物变化，不得先写进 `时间线.md`、`角色状态.md`、`情报台账.md`、`伏笔.md`。
8. **双版本并存必须清理**：同一章组若同时存在两种命名的写后验收或总表文件，必须视为流程风险，不能默认共存。

## Examples

### Correct
```
修改沈栀的灵力等级为“第六层”后：
1. 检查追踪/伏笔.md 中是否有依赖灵力等级的伏笔
2. 更新追踪/上下文.md: [2024-05-05] 修改沈栀灵力等级为第六层
3. 如有时间线文件，检查第45章对应的时间点是否合理
```

### Wrong
```
直接修改设定/角色/沈栀.md 中的灵力等级，不检查任何关联文件。
结果：第20章中“沈栀灵力只有第三层”的伏笔变得矛盾。
```
